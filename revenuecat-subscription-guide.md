このドキュメントは2部構成です。

# 第1部: サブスクリプション（定期課金）について

## 1-1. サブスクリプションとは

ユーザーが一定期間ごと（月額・年額など）に料金を支払い、その間サービスや機能を継続利用できる課金モデル。買い切り（売り切り）型と対照的に、**継続的・予測可能な収益（リカーリングレベニュー）** を生み出すのが最大の特徴。

主な提供形態:
- **モバイルアプリ内課金（IAP）** … App Store / Google Play 経由
- **Web 課金** … Stripe など決済サービス経由
- **B2B SaaS** … 法人向け月額/年額契約

## 1-2. メリット

| 観点 | 内容 |
|------|------|
| 収益の安定性 | 毎月・毎年の継続収益で売上が予測しやすい |
| LTV 向上 | 1人あたりの生涯価値（LTV）が買い切りより高くなりやすい |
| 継続的な関係 | アップデート提供を前提にユーザーと長期的につながれる |
| 段階的導入 | 無料トライアル・フリーミアムで新規獲得しやすい |
| 価格最適化 | 月額/年額/プラン分けで多様な価格戦略が組める |

## 1-3. デメリット・注意点

| 観点 | 内容 |
|------|------|
| 解約（チャーン）対策 | 継続率（リテンション）の維持が常に課題 |
| 価値の継続提供義務 | 払い続けてもらうため機能追加・改善が前提になる |
| ストア手数料 | App Store / Google Play は売上の **15〜30%** を徴収 |
| 課金処理の複雑さ | 更新・返金・トライアル・猶予期間・課金復旧など状態管理が煩雑 |
| プラットフォーム依存 | ストア審査・ポリシー・仕様変更の影響を受ける |
| 心理的ハードル | ユーザーは「買い切り」より契約に慎重 |

## 1-4. ストア手数料（重要）

- **Apple App Store / Google Play**: 標準 **30%**
- 小規模事業者プログラム / 2年目以降の継続課金: **15%** に減免される場合あり
- → サブスクの原価計算では「ストア手数料」が最大のコスト要因。後述の RevenueCat 手数料も、この手数料を**引く前の総額（グロス）** に対して計算される点に注意。

## 1-5. 保守・メンテナンスの観点

自前でサブスク基盤を作る場合に発生する継続的な保守作業:

- レシート検証サーバーの構築・運用（Apple / Google 各々の仕様対応）
- ストア仕様変更・SDK アップデートへの追従
- 課金状態の同期（更新・失効・返金・課金復旧 / Restore）
- サブスク状態 Webhook（サーバー通知）の受信・処理
- 複数プラットフォーム（iOS / Android / Web）間の整合性維持
- 不正・返金・チャージバック対策

→ これらを **自前で全部やると重い**。ここを肩代わりするのが RevenueCat のようなサービス。

# 第2部: RevenueCat について

## 2-1. RevenueCat とは

**アプリ内サブスクリプション/課金の管理基盤（インフラ）を提供する SaaS**。
App Store・Google Play・Stripe などの課金を **統一的な SDK と API** で扱えるようにし、レシート検証・課金状態管理・分析・A/Bテストまでをワンストップで提供する。

世界中の多数のアプリで採用されており、サブスク管理のデファクトに近い存在。

## 2-2. 主な機能

| 機能 | 内容 |
|------|------|
| クロスプラットフォーム SDK | iOS / Android / Flutter / React Native / Unity / Capacitor / Cordova / Web |
| サーバーサイドレシート検証 | Apple / Google のレシート検証をRevenueCat側で実施。**自前の検証サーバー不要** |
| Entitlements（権利）管理 | 「ユーザーが特定機能の利用権を持つか」をシンプルに判定 |
| Offerings / Packages | 価格プラン・表示順・トライアル設定を**ダッシュボードから変更**（コード変更・再リリース不要） |
| ビルトイン A/Bテスト | ペイウォール・価格・オファーを比較し、有意差付きで効果測定 |
| 分析ダッシュボード | MRR・チャーン・LTV・トライアル転換率などを可視化 |
| Webhook / 連携 | 課金イベントを各種ツール（分析・CRM等）へ連携 |

## 2-3. メリット

- **開発工数の大幅削減** … 自前のレシート検証サーバーやサブスク状態管理を作らなくてよい
- **クロスプラットフォーム対応が容易** … 1つのモデルで iOS/Android/Web 等を抽象化、保守負担が減る
- **ストア仕様変更への追従を肩代わり** … Apple/Google の仕様変更にRevenueCat側が対応してくれる
- **ノーコードで価格・ペイウォール変更** … マーケ施策を再リリースなしで回せる
- **分析・実験ツールが標準装備** … 別途分析基盤を作らなくてもKPIが見える
- **小規模なら無料で始められる** … 後述の無料枠

## 2-4. デメリット・注意点

- **コストが売上連動** … 売上が伸びるほど手数料額も増える（グロス課金）
- **ベンダーロックイン** … RevenueCat 前提で作り込むと移行が大変
- **外部サービス依存** … RevenueCat 側の障害・ダウンタイムの影響を受けうる（過去に稀に報告あり）
- **ペイウォールのカスタマイズに限界** … デザイン自由度に物足りなさを感じる声（特に Flutter 等）
- **ドキュメントが膨大** … 多機能ゆえ学習コスト・情報の探しにくさがある
- **サポート品質にばらつき** … 迅速な対応の評価がある一方、返信が遅いという声も

## 2-5. 費用 / ランニング費用（2026年時点）

RevenueCat の料金は **MTR（Monthly Tracked Revenue: 月間トラッキング売上）** ベース。

| プラン | 料金 | 備考 |
|--------|------|------|
| **Free（無料）** | $0 | **月間 MTR $2,500 まで**無料。多くの個人開発・初期スタートアップはここで十分 |
| **Pro** | **グロス MTR の 1%** | $2,500 を超えた分に課金。導入費・最低月額・サブスク数上限なし |
| **Enterprise** | 個別見積もり | 大規模・カスタム要件向け |

### 重要: 「実質手数料率」は1%より高くなる

RevenueCat の 1% は **ストア手数料（30%）を引く前のグロス売上** に対して計算される。

- 例: グロス MTR **$100,000** の場合
  - RevenueCat 手数料 = **$1,000**（1%）
  - ストア手数料30%を引いた手取り（ネット）は $70,000
  - → **手取りに対する実質料率 ≒ 1.43%**

> ランニング費用を見積もる際は「グロス売上 × 1%」で計算し、手取りベースでは約1.4%前後になると理解しておくとよい。

## 2-6. 開発のし易さ

- **非常に高い**。SDK を入れて「Entitlement を持っているか」を確認するだけ、という最小実装で動かせる
- レシート検証サーバーの構築・運用が不要 → サーバーサイド開発をほぼ省略できる
- 公式SDK・サンプル・ドキュメントが充実（多言語・多フレームワーク）
- 課金復旧（Restore）・トライアル・更新などの面倒な状態管理を肩代わり
- 注意点: 多機能ゆえ最初は概念（Offerings/Packages/Entitlements）の理解にやや時間がかかる

## 2-7. 開発費（初期コスト）の目安

| 項目 | 自前実装 | RevenueCat 利用 |
|------|----------|-----------------|
| レシート検証サーバー構築 | 必要（重い） | **不要** |
| 課金状態管理ロジック | 必要 | SDK が肩代わり |
| iOS/Android 個別対応 | 各々必要 | 共通SDKで吸収 |
| 分析・A/B基盤 | 別途構築 | 標準装備 |
| 初期開発工数の目安 | 数週間〜数ヶ月 | **数日〜1〜2週間程度** |

→ 初期開発費・期間を大きく圧縮できるのが RevenueCat の最大の価値。

## 2-8. まとめ（RevenueCat 採用判断）

| こんな場合は採用が向く | こんな場合は再検討 |
|------------------------|--------------------|
| クロスプラットフォーム展開したい | 単一プラットフォーム & 内製余力が潤沢 |
| サブスク基盤を自前で作りたくない | 売上規模が非常に大きく1%でも手数料額が無視できない |
| 分析・A/Bを早く回したい | ペイウォールを完全自由にデザインしたい |
| 小〜中規模で素早く立ち上げたい | ベンダーロックインを徹底的に避けたい |

**総評**: 個人開発〜中規模アプリでは「無料枠＋開発工数削減」のメリットが非常に大きく、第一候補になりやすい。売上が大きくなったら手数料額とロックインを天秤にかけ、Enterprise 交渉や代替（Adapty 等）も検討する、というのが現実的な戦略。

# 第3部: 開発ステップ数・サーバー/保守の削減効果

「RevenueCat を使うと開発がどれだけ楽になるか」を、実装ステップ・サーバー・保守の3観点で定量比較する。

## 3-1. 開発ステップ数の比較

### 自前実装（ネイティブIAP）の典型フロー — 約15ステップ

**クライアント側**
1. ストアに商品（サブスク）を登録（App Store Connect / Google Play Console）
2. iOS に StoreKit を実装
3. Android に Google Play Billing Library を**別途**実装
4. 商品情報の取得・表示
5. 購入処理の実行・トランザクション管理
6. レシート / 購入トークンの取得
7. 購入復元（Restore）処理の実装

**サーバー側**
8. レシートをサーバーへ送信
9. Apple のレシート検証（App Store Server API）
10. Google の購入トークン検証（Play Developer API）
11. サブスク状態を DB に保存・管理
12. App Store Server Notifications（V2）の Webhook 受信
13. Google Real-time Developer Notifications（Pub/Sub）の受信
14. 更新・失効・猶予期間・課金リトライ・返金・プラン変更（proration）の状態遷移処理
15. iOS/Android 間のエンタイトルメント整合

### RevenueCat の場合 — 約6〜7ステップ

1. ストアに商品を登録（※これは共通で必要）
2. RevenueCat ダッシュボードで商品 / Entitlement / Offering を設定
3. SDK をインストール
4. SDK を APIキーで初期化（実質1行）
5. Offering（ペイウォール）を表示
6. `purchase()` を呼ぶ
7. `CustomerInfo` で「Entitlement を持っているか」を確認するだけ

> **→ ステップ数で約 15 → 7、おおよそ半減以下。** 特にサーバー側の7ステップ（8〜14）がまるごと消えるのが効く。

## 3-2. サーバーの削減効果

| サーバー側で本来必要な実装 | RevenueCat |
|---|---|
| レシート検証サーバー（Apple/Google両対応） | **不要**（RevenueCat側で実施） |
| サブスク状態を保持する DB | **不要**（RevenueCat が真実の源） |
| App Store Server Notifications 受信 | **不要** |
| Google RTDN（Pub/Sub）受信 | **不要** |
| 更新・失効・返金の状態遷移ロジック | **不要** |

**→ サブスク用バックエンドはほぼ100%カット可能。** 自社サーバーが要るのは「課金状態を自前DBにも持ちたい」「独自の不正対策を強化したい」等の追加要件がある場合のみで、その場合も RevenueCat の Webhook を受けるだけでよく、検証ロジックの自作は不要。

## 3-3. 保守・メンテナンスの削減効果

| 保守タスク | 削減度 |
|---|---|
| Apple / Google のレシート検証 API 仕様変更への追従 | **ほぼ全削減** |
| StoreKit / Play Billing のメジャー更新対応 | **大幅削減**（SDK更新で吸収） |
| サーバー運用（監視・スケール・障害対応） | **大幅削減**（サーバー自体が無い） |
| 価格・トライアル・ペイウォール変更 | **大幅削減**（ダッシュボードでノーコード変更、再リリース不要） |
| 分析・KPI集計基盤の保守 | **削減**（標準ダッシュボード利用） |

**残る保守**: RevenueCat SDK のバージョンアップ追従、ダッシュボード設定管理、RevenueCat 障害時の対応程度。

## 3-4. まとめ（削減効果のざっくり感）

| 観点 | 自前実装 | RevenueCat |
|------|----------|------------|
| 実装ステップ数 | 約15 | **約7（半減以下）** |
| サブスク用サーバー開発 | 必須 | **ほぼ全カット** |
| 初期開発期間 | 数週間〜数ヶ月 | **数日〜1〜2週間** |
| 継続保守の負荷 | 大（仕様追従＋サーバー運用） | **体感7〜8割減** |

トレードオフは「グロス売上の1%手数料」と「ベンダーロックイン」。小〜中規模では削減メリットが手数料を大きく上回るケースが多い。
