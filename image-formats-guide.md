## 大分類:ラスター画像とベクター画像

| 種類 | 仕組み | 拡大時の劣化 | 代表フォーマット |
| --- | --- | --- | --- |
| **ラスター(ビットマップ)** | ピクセル(点)の集合で表現 | 拡大すると粗くなる | PNG, JPEG, GIF, WebP, AVIF |
| **ベクター** | 点・線・面を数式で表現 | 拡大しても劣化しない | SVG |

写真のような複雑な階調はラスター、ロゴやアイコンのような図形はベクターが得意、という大原則を押さえておくと選択しやすくなります。

---

## SVG(ベクター)

**Scalable Vector Graphics**。XMLベースのテキストで図形を記述するベクター形式。

### 特徴・メリット
- 拡大・縮小しても劣化しない(あらゆる解像度・Retina対応に強い)
- ファイルサイズが小さい(単純な図形の場合)
- CSS / JavaScript で色やアニメーションを動的に制御できる
- テキストなので Git で差分管理しやすく、検索・編集も可能
- アクセシビリティ対応(`<title>` `<desc>` など)

### デメリット
- 写真など複雑な画像には不向き(逆に重くなる)
- 複雑なSVGはレンダリング負荷が高くなることがある
- 外部SVGの読み込みやインライン展開時にXSSなどセキュリティ配慮が必要

### 主な用途
- **Web**:ロゴ、アイコン、UIパーツ、インフォグラフィック、チャート、地図、アニメーション
- **アプリ**:解像度の異なる端末向けのアイコン・イラスト、データ可視化

---

## PNG(ラスター)

**Portable Network Graphics**。可逆圧縮(劣化なし)でアルファチャンネル(透過)に対応。

### 特徴・メリット
- 透過(半透明含む)に対応
- 可逆圧縮で画質が劣化しない
- 文字・線・ベタ塗りなどシャープなエッジに強い

### デメリット
- 写真ではJPEGよりファイルサイズが大きくなりがち
- アニメーション非対応(APNGという拡張はあるが対応環境が限定的)

### 主な用途
- **Web**:透過が必要なロゴ・アイコン、スクリーンショット、図解、UI素材
- **アプリ**:アプリアイコン素材、透過が必要な画像、UI部品

---

## JPEG / JPG(ラスター)

**Joint Photographic Experts Group**。非可逆圧縮で写真に最適。

### 特徴・メリット
- 写真・グラデーションなど色数の多い画像を高効率に圧縮
- 圧縮率を調整してファイルサイズと画質のバランスを取れる
- ほぼすべての環境で表示可能

### デメリット
- 非可逆圧縮のため保存のたびに劣化する
- 透過に非対応
- 文字や境界がはっきりした画像ではノイズ(モスキートノイズ)が出やすい

### 主な用途
- **Web**:写真、サムネイル、ヒーロー画像、背景写真
- **アプリ**:カメラ撮影画像、プロフィール写真、商品写真

---

## GIF(ラスター)

**Graphics Interchange Format**。256色までの簡易アニメーション対応形式。

### 特徴・メリット
- 簡単なアニメーションを1ファイルで表現できる
- 透過(1bit、オン/オフのみ)対応
- 互換性が非常に高い

### デメリット
- 最大256色しか扱えず写真には不向き
- ファイルサイズが大きくなりやすい
- 半透明表現ができない

### 主な用途
- **Web**:短いアニメーション、ローディング表示、ミーム
- **アプリ**:簡易アニメーション(ただし現在は動画やWebP/Lottieへの置き換えが進む)

> 補足:アニメーション用途では **WebP / APNG / 動画(MP4)/ Lottie(JSON)** の方が高品質・低容量なケースが多い。

---

## WebP(ラスター)

Googleが開発した、可逆・非可逆の両方に対応する次世代フォーマット。

### 特徴・メリット
- JPEG/PNGより高圧縮(同等画質でファイルサイズが小さい)
- 透過とアニメーションの両方に対応
- 主要ブラウザで広くサポート済み

### デメリット
- 非常に古い環境では非対応(フォールバックが必要な場合あり)
- 一部の編集ソフトでの扱いがJPEG/PNGほど普及していない

### 主な用途
- **Web**:写真・イラスト・透過画像・アニメーションを軽量化したいほぼ全般
- **アプリ**:通信量を抑えたい画像配信、サムネイル

---

## AVIF(ラスター)

AV1動画コーデックをベースにした最新世代の画像フォーマット。

### 特徴・メリット
- WebPよりさらに高圧縮で高画質(特に低ビットレートに強い)
- 透過・HDR・広色域に対応

### デメリット
- エンコードに時間・負荷がかかる
- 対応環境はWebPより新しく、フォールバック設計が望ましい

### 主な用途
- **Web**:大きな写真やヒーロー画像の最適化、パフォーマンス重視サイト
- **アプリ**:高画質を保ちつつ容量を最小化したい画像配信

---

## ICO / favicon

ブラウザのタブやブックマークに表示されるアイコン用形式。

### 特徴
- 複数サイズ(16×16、32×32 など)を1ファイルに格納できる
- 現在は `favicon.ico` に加え、PNG / SVG を `<link>` で指定する手法も一般的

### 主な用途
- **Web**:ファビコン、ブラウザタブのアイコン、PWAアイコン(PNG/SVG併用)

## 用途別おすすめ早見表

| やりたいこと | おすすめフォーマット |
| --- | --- |
| ロゴ・アイコン・UIパーツ | **SVG**(代替:PNG) |
| 写真・ヒーロー画像 | **AVIF / WebP**(フォールバック:JPEG) |
| 透過が必要な画像 | **SVG / PNG / WebP** |
| スクリーンショット・図解 | **PNG**(または WebP 可逆) |
| 簡単なアニメーション | **WebP / Lottie / 動画(MP4)**(代替:GIF) |
| データ可視化・グラフ | **SVG** |
| ファビコン | **SVG + PNG**(`favicon.ico` 併用) |
| 通信量を最小化したい配信全般 | **AVIF / WebP** |

## 実装のポイント

### `<picture>` でフォールバックする(Web)
新しいフォーマットを優先しつつ、非対応環境では従来フォーマットを表示します。

```html
<picture>
  <source srcset="hero.avif" type="image/avif">
  <source srcset="hero.webp" type="image/webp">
  <img src="hero.jpg" alt="ヒーロー画像" width="1200" height="600">
</picture>
```

### レスポンシブ対応(`srcset` / `sizes`)
端末の解像度・画面幅に応じて適切なサイズを配信します。

```html
<img
  src="photo-800.jpg"
  srcset="photo-400.jpg 400w, photo-800.jpg 800w, photo-1600.jpg 1600w"
  sizes="(max-width: 600px) 400px, 800px"
  alt="商品写真">
```

### その他のベストプラクティス
- `width` / `height` または `aspect-ratio` を指定してレイアウトシフト(CLS)を防ぐ
- ファーストビュー以外は `loading="lazy"` で遅延読み込み
- アイコンは可能な限りSVG化して解像度非依存にする
- アプリでは端末ごとの解像度別アセット(@1x/@2x/@3x)や、ベクター(SVG/Vector Drawable)を活用する

---

### まとめ
- **図形・ロゴ・アイコン → SVG**
- **写真 → AVIF / WebP(フォールバックにJPEG)**
- **透過・図解 → PNG / WebP**
- **アニメーション → WebP / 動画 / Lottie(GIFは限定的に)**

「ベクターかラスターか」「透過の要否」「写真か図形か」の3点で考えると、最適なフォーマットを選びやすくなります。
