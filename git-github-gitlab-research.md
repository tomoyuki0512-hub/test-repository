# Git / GitHub / GitLab 調査まとめ

## 1. Git・GitHub・GitLab の関係

- **Git**: バージョン管理システムそのもの（ツール）。ローカルで動くソフトウェア。Linus Torvalds が2005年に開発
- **GitHub / GitLab**: Git リポジトリをクラウド上でホストするサービス。Git の上に構築されたプラットフォーム

```
Git（コア技術）
  └── GitHub（Git をホストするサービスA）
  └── GitLab（Git をホストするサービスB）
  └── Bitbucket（同様のサービスC）
  └── 自前サーバー（Gitea など）
```

**例え**: Git が SMTP（メールプロトコル）、GitHub/GitLab が Gmail/Outlook（そのプロトコルを使ったサービス）

---

## 2. Git の一番の目的

**ソースコードや資材をバージョン管理しながら保管すること**

- 変更履歴が残る（誰が・いつ・何を変えたか追跡）
- 過去に戻れる（任意の時点に復元可能）
- 複数人で作業できる（マージ機能）
- バックアップになる

※ 画像・動画などバイナリファイルは差分が取れないため苦手

---

## 3. GitHub vs GitLab の主な違い

| 観点 | GitHub | GitLab |
|---|---|---|
| セルフホスト | GitHub Enterprise（有料） | GitLab CE（無料あり） |
| CI/CD | GitHub Actions | GitLab CI/CD（内蔵・歴史が長い） |
| 機能統合 | エコシステム連携重視 | オールインワン設計 |
| コミュニティ | OSS の中心地・最大規模 | 企業内・オンプレ利用に強い |
| 所有 | Microsoft 傘下 | 独立企業（上場） |

**一言**: GitHub はコミュニティと連携の広さ、GitLab はオールインワンとセルフホストの柔軟性が強み

## 4. 自前ホストの方法

| ツール | 難易度 | 特徴 |
|---|---|---|
| Gitea | 易 | 軽量・シンプル・Docker で即起動 |
| GitLab CE | 中 | 高機能・CI/CD内蔵・リソース消費大 |
| Forgejo | 易 | Gitea のフォーク・コミュニティ主導 |
| 素の Git | 易 | UI なし・最小構成 |

**Gitea の起動例（Docker）**
```bash
docker run -d \
  -p 3000:3000 \
  -v /srv/gitea:/data \
  gitea/gitea:latest
```

## 5. 自前ホストを選ぶべき条件

| 観点 | 評価 |
|---|---|
| **セキュリティ** | データを外部に出せない法規制・契約がある場合に有効。ただしパッチ適用は自己責任 |
| **コスト** | 大人数でクラウド課金が高額になる場合に有利。小規模はクラウドの方が安い |
| **チェックアウト速度** | 社内 LAN 上にあれば遅延ほぼゼロ。大容量リポジトリで差が出る |
| **CI/CD** | 社内リソース（GPU等）を使いたい場合・ビルド時間課金を避けたい場合に有利 |
| **可用性** | 自前は障害対応が自己責任。クラウドは SLA あり |

**自前を選ぶ明確な理由がない限りクラウドで十分**

## 6. ハイブリッド運用（GitHub + 自前）

### 可能なパターン

| パターン | 内容 | 複雑さ |
|---|---|---|
| リポジトリ単位で分ける | GitHub→公開、自前→機密 | 低（推奨） |
| 依存ライブラリだけ自前 | アプリ本体は GitHub、社内ライブラリは自前レジストリ | 低 |
| Git Submodule で組み合わせる | 1プロジェクト内で混在 | 高（非推奨） |

### LFS（Large File Storage）だけ外出し

動画・画像・ログなど大容量ファイルのみ自前 LFS サーバーに置く構成。

**弊害**:
- クローン後に LFS サーバーへの認証設定が別途必要
- CI/CD にも LFS サーバーへのアクセス設定が必要
- 自前 LFS サーバーが落ちるとファイル取得不可

**有効なケース**: GitHub の LFS 容量課金を避けたい、大量の動画・ログがある場合

---

## 7. 結論：管理方針の優先順位

```
1. GitHub / GitLab.com（クラウド）で1つに統一  ← 基本はこれ
      ↓ 以下の理由がある場合のみ
2. 自前ホストで1つに統一
      ↓ どうしても分けざるを得ない場合のみ
3. ハイブリッド
```

**1つで管理するメリット**:
- `git clone` が1回で完結
- CI/CD の設定が1箇所
- 開発者の認証管理が1つ
- 「どこに何があるか」で迷わない

**複雑な構成は「必要になってから」検討で十分。**
