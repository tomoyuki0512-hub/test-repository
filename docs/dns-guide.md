# DNS・ドメイン完全ガイド

むずかしい用語をたとえ話でわかりやすく解説しつつ、技術的な詳細も網羅したガイド。

---

## 1. DNSとは何か — 「インターネットの電話帳」

スマートフォンに「お母さん」と登録しておけば、番号を覚えなくても電話できる。DNSはそれと同じしくみ。

- **ドメイン名**（例：`example.com`）＝「お母さん」という名前
- **IPアドレス**（例：`93.184.216.34`）＝実際の電話番号

ブラウザに `google.com` と入力すると、裏側でDNSが「その名前の番号は何番？」と調べ、正しいサーバーにつないでくれる。

### 名前解決の流れ（技術的な詳細）

```
あなたのPC（ブラウザ）
    │
    ▼
フルサービスリゾルバー（ISPやGoogle 8.8.8.8、Cloudflare 1.1.1.1）
    │
    ├─→ ルートネームサーバー（「.comのことは.comサーバーへ」）
    │
    ├─→ TLDネームサーバー（「example.comはns1.example.comへ」）
    │
    └─→ 権威ネームサーバー（「example.com = 93.184.216.34」）
```

各レコードには **TTL（Time To Live）** が設定されており、その秒数だけ結果がキャッシュされる。DNSの変更が世界中に反映されるまで時間がかかるのはこのため。

## 2. IPアドレスとは何か — 「インターネット上の住所」

荷物を届けるために「住所」が必要なように、インターネットでもデータを届けるための「住所」が必要で、それが **IPアドレス**。

| 種類 | 形式 | 例 |
|---|---|---|
| **IPv4** | 4つの数字をドットでつなぐ | `192.168.1.1` |
| **IPv6** | 英数字の長い形式 | `2001:db8::1` |

IPv4は全世界で約43億個しかなく、すでに枯渇済み。IPv6への移行が進んでいる。

### IPアドレスはどこから来るの？

```
IANA（グローバル管理）
    └─ APNIC（アジア太平洋地域の管理機関）
        └─ ISP（NTT、ソフトバンクなど）
            ├─ 固定IP：毎回同じIPを割り当て
            └─ 動的IP：接続のたびに変わる（一般家庭はこちら）
```

### グローバルIPとローカルIP

| 種類 | 説明 | 例 |
|---|---|---|
| **グローバルIP** | インターネット上で使える住所（世界に1つ） | `203.0.113.1` |
| **ローカルIP** | 自宅・会社内だけで使う住所 | `192.168.1.x` |

家庭のルーターが「グローバルIP ↔ ローカルIP」を変換（NAT）しているため、1つのグローバルIPでスマホ・PC・タブレット全てがインターネットに接続できる。

## 3. DNSレコードの種類

電話帳に「電話番号」だけでなく「FAX番号」や「メールアドレス」も書けるように、DNSにも用途別のレコードがある。

### Aレコード — 「このドメインのIPアドレスはこれ」

最もよく使うレコード。ドメイン名とIPv4アドレスを結びつける。

```
example.com.    300    IN    A    93.184.216.34
```

GitHub PagesのカスタムドメインでAレコードを使う場合はGitHubが指定するIPアドレスを設定する。

---

### AAAAレコード — 「AレコードのIPv6版」

Aレコードと同じ役割で、IPv6アドレスに対応したもの。

```
example.com.    300    IN    AAAA    2606:2800:220:1:248:1893:25c8:1946
```

「AAAA」は4つのAが並んでいる。IPv6はIPv4の「4倍の長さ」という覚え方でOK。

---

### CNAMEレコード — 「別名・あだ名」

あるドメインを、別のドメイン名に転送するためのレコード。

```
www.example.com.    3600    IN    CNAME    example.com.
```

> `www.example.com`にアクセスした人を `example.com` に転送する、あだ名のようなもの。

**注意：** ルートドメイン（`example.com` そのもの）にはCNAMEを設定できない（RFC規格の制約）。`www.example.com` など、サブドメインにのみ設定可能。

---

### MXレコード — 「メールはこのサーバーへ」

メール受信サーバーを指定する。`@example.com` 宛のメールをどこで受け取るかを決める。数字が小さい方が優先度が高い。

```
example.com.    3600    IN    MX    10    mail1.example.com.
example.com.    3600    IN    MX    20    mail2.example.com.（予備）
```

Gmailで独自ドメインのメールを使うときにこのレコードを設定する。

---

### TXTレコード — 「自由なメモ書き・証明書」

テキストを自由に書けるレコード。「このドメインは本物ですよ」という証明によく使われる。

| 用途 | 内容 |
|---|---|
| **SPF** | 「このドメインからのメールはこのサーバーからだけ」（メール詐欺防止） |
| **DKIM** | メールの電子署名の公開鍵 |
| **DMARC** | メール認証ポリシーの設定 |
| **ドメイン所有権の確認** | Google Search Console など各種サービスの認証 |

---

### NSレコード — 「このドメインの管理はここ」

「このドメインのDNS情報は、このネームサーバーが管理している」という案内板。

```
example.com.    86400    IN    NS    ns1.example.com.
example.com.    86400    IN    NS    ns2.example.com.
```

ドメイン登録サービスを変えたり、CloudflareにDNS管理を移すときにここが変わる。

---

### SOAレコード — 「このDNSの管理情報」

ゾーン（ドメイン全体の設定）の管理責任者・更新日時などの情報。通常は自分で触ることはなく、ドメイン登録サービスが自動管理してくれる。

---

### レコード一覧まとめ

| レコード | 役割 | よく使う場面 |
|---|---|---|
| A | ドメイン → IPv4アドレス | ウェブサイトの公開 |
| AAAA | ドメイン → IPv6アドレス | IPv6対応サーバーへの接続 |
| CNAME | ドメイン → 別のドメイン名（別名） | wwwの設定、GitHub Pages |
| MX | メールの受け取りサーバー指定 | 独自ドメインのメール |
| TXT | 自由なテキストメモ | 認証・メール詐欺対策 |
| NS | DNSを管理するサーバー指定 | DNS管理サービスの変更時 |
| SOA | DNS全体の管理情報 | 通常は自動管理 |

## 4. ドメインの選び方

ドメインの末尾（`.com` や `.jp` の部分）を **TLD（トップレベルドメイン）** という。

### 主なTLDと特徴

| TLD | 特徴 | おすすめな人 |
|---|---|---|
| `.com` | 世界で最も有名・信頼性が高い。迷ったらこれ。 | 誰にでも |
| `.net` | ネットワーク系の意味合い。`.com` の代替として広く使われる。 | `.com`が取れないとき |
| `.org` | 非営利団体に多いが制限なし。 | コミュニティ・OSS・非営利 |
| `.jp` | 日本を示す国別ドメイン。「日本の」という印象を与えられる。 | 日本向けのサービス・お店 |
| `.co.jp` | 日本の**法人のみ**取得可能。信頼感が増す。 | 企業・会社 |
| `.io` | IT系スタートアップに人気。本来はインド洋の国別TLD。 | エンジニア・テックサービス |
| `.dev` | 開発者向け（Google管理）。**HTTPS強制**（HTTPSが必須）。 | 技術ブログ・ポートフォリオ |
| `.app` | アプリ向け（Google管理）。`.dev` 同様HTTPS強制。 | アプリのランディングページ |
| `.ai` | AI関連サービスに多用。アンギラ国の国別TLDが国際的に普及。 | AIサービス |

### 選び方のポイント

1. **迷ったら `.com`** — 認知度・信頼性ともに最高
2. `.com` が取れなければ名前を変えるか `.net` / `.jp` を検討
3. 日本国内向けなら `.jp`、エンジニア向けなら `.dev` / `.io` も良い
4. **短くてシンプル・読みやすい名前**にする
5. 有名ブランドや商標と紛らわしい名前は**絶対に避ける**（法的トラブルになる）
6. **SEOへの影響**：`.jp` は日本向けのジオターゲティングシグナルになる

## 5. ドメイン登録サービスの比較

### 主要サービス比較

| サービス | 拠点 | `.com`年間費用（目安） | 特徴 |
|---|---|---|---|
| **お名前.com** | 日本 | 初年度1円〜、更新1,500円〜 | 国内最大手。日本語サポート充実。初年度と更新の価格差に注意。 |
| **ムームードメイン** | 日本 | 初年度160円〜 | GMOグループ。シンプルな管理画面。 |
| **Xserverドメイン** | 日本 | 1,500円〜 | Xserverとの連携が楽。安定した国内サービス。 |
| **Cloudflare Registrar** | 米国 | 約$10.44（原価） | **原価販売でマークアップなし**。WHOIS代行無料。管理UIが優秀。英語インターフェース。 |
| **Namecheap** | 米国 | 約$9〜$14 | 低価格帯。WHOIS代行無料。DNS管理機能が充実。 |
| **Porkbun** | 米国 | 約$9〜 | 新興サービスだが価格が安い。英語インターフェース。 |

### 費用について知っておくこと

- **初年度だけ激安**なサービスが多い（例：初年度1円 → 2年目から1,500円/年）
- **更新料金を必ず確認**する。登録時より更新時の方が高いケースがある
- Cloudflare Registrar は原価販売なので**更新料も安定して安い**

### 安心なサービスの選び方

| 優先したいこと | おすすめ |
|---|---|
| 日本語サポートが欲しい | お名前.com、Xserverドメイン |
| とにかく安くしたい（英語OK） | Cloudflare Registrar、Porkbun |
| 管理のしやすさ | Cloudflare Registrar |

**共通の注意点：**
- WHOIS代行（個人情報保護）が無料のサービスを選ぶ
- 2段階認証（2FA）が使えるサービスを選ぶ（ドメインはハイジャックリスクがある）
- 移管（サービス乗り換え）はいつでも可能。最初のサービスが気に入らなければ後から移せる

## 6. GitHub PagesでHTTPSが自動でつく理由

### HTTPSと証明書とは？

ブラウザのアドレスバーの `https://` は、通信が暗号化されている証拠。これには **SSL/TLS証明書** が必要で、通常は認証局（CA）から取得する必要がある。以前は有料・手動での取得が必要だった。

### Let's Encryptが無料化した

**Let's Encrypt** は、Mozilla（Firefoxの会社）などが支援する非営利団体で、**無料でSSL証明書を発行**してくれるサービス。現在3億以上のドメインに証明書を発行している。

### なぜGitHub Pagesでは自動でつくのか

GitHubがすべて代わりにやってくれるから。

```
ユーザーがすること
    └─ DNSレコードをGitHubのIPに向けるだけ

GitHubが自動でやること
    ├─ Let's Encryptに証明書を申請
    ├─ ドメイン所有権の検証（DNS設定がその証明になる）
    ├─ 証明書をサーバーに設定
    └─ 90日ごとの証明書の自動更新
```

### 技術的な仕組み（詳細）

Let's Encryptは **ACME（Automatic Certificate Management Environment）プロトコル**（RFC 8555）で証明書発行を自動化している。

ドメイン所有権の検証方法：
- **HTTP-01チャレンジ**：指定のパス（`/.well-known/acme-challenge/`）にトークンファイルを配置
- **DNS-01チャレンジ**：`_acme-challenge.example.com` にTXTレコードを設定

GitHub Pagesは HTTP-01チャレンジをインフラ側で処理。DNSがGitHubのIPを向いていることが「ドメイン所有者がGitHubを指定した」という証明になる。証明書の有効期限は90日だが、GitHubが期限切れ前に自動更新するためユーザーの作業は一切不要。

### まとめ

| 従来の手動運用 | GitHub Pages |
|---|---|
| 認証局と契約・支払い | 不要（Let's Encryptは無料） |
| 証明書の申請 | GitHubが自動実行 |
| サーバーへの証明書設定 | GitHubのインフラが処理 |
| 90日ごとの更新作業 | GitHubが自動更新 |

カスタムドメインでも同じ。必要なのは**DNSレコードを正しく設定するだけ**。設定後、数分〜最大24時間以内にHTTPSが有効になる。
