# アジャイル開発 完全ガイド

ソフトウェア開発の現場で標準となった「アジャイル開発」を、ゼロから実務レベルまで体系的に学べるガイド集です。歴史や考え方から、スクラム・カンバンといった具体的な手法、見積りや技術プラクティス、大規模化、導入の落とし穴までを章ごとにまとめています。

## 30秒サマリ

- **アジャイル開発**とは、短い期間（1〜4週間）で「動くソフトウェア」を繰り返し作り、フィードバックを得ながら計画を柔軟に見直していく開発アプローチの総称
- 2001年の「**アジャイルソフトウェア開発宣言（アジャイルマニフェスト）**」が原点。特定の手法名ではなく**価値観と原則**の名前
- 代表的な手法は **スクラム**（世界で最も普及）、**カンバン**、**XP（エクストリーム・プログラミング）**
- 「速く作る」手法ではなく、「**間違ったものを作り込むリスクを減らす**」手法。変化が激しく正解が見えない領域で特に強い

## 読者別おすすめルート

- **はじめての人**: ① → ② → ⑦ → ⑯（基礎 → 定番手法 → 落とし穴 → シミュレーションで疑似体験）
- **開発者**: ① → ② → ⑤ → ⑩ → ⑭（考え方 → チームの回し方 → 技術面 → テスト → レガシー）
- **プロダクトオーナー / 企画担当**: ① → ④ → ② → ⑨（考え方 → 要求と見積り → スクラムでの役割 → ディスカバリー）
- **マネージャー / 導入推進者**: ① → ⑦ → ⑥ → ⑫（考え方 → 導入と計測 → 組織展開 → 日本の実務）
- **スクラムマスター**: ② → ⑧ → ⑪ → ⑬（スクラム → チームと文化 → ふりかえり → コーチング）
- **デザイナー**: ① → ⑨ → ⑮（考え方 → ディスカバリー → 協働の実務）

## 用語ミニ辞典（最初に押さえる10語）

| 用語 | 意味 |
|---|---|
| イテレーション / スプリント | 開発を区切る短い固定期間（通常1〜4週間）。スプリントはスクラム用語 |
| プロダクトバックログ | 実現したいことを優先順位付きで並べた一覧 |
| ユーザーストーリー | 「誰が・何を・なぜ」の形式で書く要求の表現方法 |
| ベロシティ | 1スプリントでチームが完了できる作業量の実績値 |
| プロダクトオーナー（PO） | 何を作るかの優先順位に責任を持つ役割 |
| スクラムマスター | プロセスが機能するよう支援する役割。管理職ではない |
| デイリースクラム | 毎日15分の同期ミーティング |
| レトロスペクティブ | スプリント末にやり方自体を振り返る会 |
| WIP制限 | 同時に進める仕事の数に上限を設けること（カンバンの中核） |
| 完成の定義（DoD） | 「完了」と呼べる条件をチームで合意したチェックリスト |

その他の用語は[用語集・チートシート](glossary.md)で章への参照付きで引けます。
