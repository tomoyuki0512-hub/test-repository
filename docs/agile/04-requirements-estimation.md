# ④ 要求と見積り — ユーザーストーリー・バックログ・ストーリーポイント

[← 総合インデックスに戻る](README.md)

## 1. ユーザーストーリー

### 基本フォーマット

アジャイルでは要求を仕様書の箇条書きではなく、**「誰の・何のための機能か」が分かる短い文**で表現します。

> **\<ユーザーの種類\> として、\<実現したいこと\> がしたい。それは \<理由・価値\> のためだ。**

例:

- 「**サイト訪問者**として、**メールアドレスでパスワードを再設定**したい。**パスワードを忘れてもアカウントに戻れる**ようにするためだ」
- 「**経理担当者**として、**月次の売上をCSVでダウンロード**したい。**会計ソフトに取り込む**ためだ」

### なぜこの形式か

- **Why（理由）が残る**ため、実装時により良い代替案を提案できる
- 機能一覧と違い「誰も使わない機能」が混ざると**違和感に気づける**
- 詳細を書き切らないことで、**会話のきっかけ**として機能する（ストーリーは「会話の約束」と呼ばれる）

### 良いストーリーの条件 — INVEST

| 頭文字 | 条件 | 意味 |
|---|---|---|
| **I**ndependent | 独立している | 他のストーリーと順序依存が少ない |
| **N**egotiable | 交渉可能 | 実現方法を話し合う余地がある |
| **V**aluable | 価値がある | ユーザーか顧客に価値が届く |
| **E**stimable | 見積り可能 | 大きさの見当がつけられる |
| **S**mall | 小さい | 1スプリントに収まる（目安: 数日以内） |
| **T**estable | テスト可能 | 完了を確認する方法がある |

### 受け入れ条件（Acceptance Criteria）

ストーリーには「何を満たせば完了か」を付けます。**Given-When-Then**形式が定番です。

> - Given: 登録済みメールアドレスを入力した状態で
> - When: 「再設定リンクを送信」を押すと
> - Then: 60秒以内に再設定メールが届き、リンクは24時間有効である

### ストーリーの分割

大きすぎるストーリー（エピック）は縦に割ります。**「工程で割る（設計だけ・実装だけ）」のはNG**で、それぞれが単体で価値を届けられる単位に割るのが原則です。

| 分割の切り口 | 例（「商品を検索できる」を割る） |
|---|---|
| ワークフローのステップ | まずキーワード検索のみ → 絞り込み → 並べ替え |
| ビジネスルール | 通常商品のみ → セール品対応 → 在庫切れ表示 |
| データの種類 | 商品名のみ → カテゴリ → タグ |
| 操作(CRUD) | 閲覧のみ → 登録 → 編集 → 削除 |
| 品質の段階 | まず動く → 速くする → 美しくする |

## 2. プロダクトバックログの管理

### 構造

```
優先度高 ┌──────────────────────────────┐
   ↑    │ 詳細化済み・見積り済み（次の1〜2スプリント分）│ ← すぐ着手できる粒度
   │    │ ある程度具体化されたストーリー               │
   │    │ 粗い機能・エピック                         │
   ↓    │ アイデア・メモレベル                       │ ← 粗いままでよい
優先度低 └──────────────────────────────┘
```

上位ほど詳細で、下位ほど粗い形を保ちます（**DEEP**: Detailed appropriately / Estimated / Emergent / Prioritized）。全項目を最初に詳細化するのはムダ（多くは実装前に消えるか変わる）です。

### 優先順位付けの技法

| 技法 | 内容 |
|---|---|
| 価値 vs 労力マトリクス | 価値が高く労力が小さいものから。象限で粗く分ける |
| **MoSCoW** | Must / Should / Could / Won't の4分類 |
| **WSJF**（SAFe由来） | (ビジネス価値＋時間的緊急度＋リスク低減) ÷ 作業量 が大きい順 |
| 狩野モデル | 当たり前品質 / 性能品質 / 魅力品質に分類して考える |

### リファインメント（手入れ）

バックログは「一度書いたら終わり」ではなく、週1回程度チームで**分割・詳細化・見積り・並べ替え**を行います。目安として、常に1〜2スプリント分の「着手可能（Ready）」な項目を保ちます（**Definition of Ready** を決めるチームもあります）。

## 3. 見積り — ストーリーポイントと相対見積り

### なぜ時間で見積もらないのか

- 人によって作業速度が違うため「何時間」は**誰がやるか**で変わる
- 人間は絶対値の見積りが苦手だが、「AはBの2倍くらい」という**相対比較は得意**
- 時間見積りは「約束」と受け取られ、心理的な駆け引きが生まれやすい

そこで多くのチームは**ストーリーポイント**という相対値を使います。基準となるストーリーを「2」と決め、その何倍の大きさかで他を見積もります。

### フィボナッチ数列を使う理由

ポイントには **1, 2, 3, 5, 8, 13, 21...** の修正フィボナッチ数列を使うのが定番です。大きい作業ほど不確実性が高く、「12と13の違い」を議論するのは無意味なため、**数字の間隔を粗くして精度の錯覚を防ぎます**。8や13が付いたストーリーは分割を検討するシグナルです。

### プランニングポーカー

1. POがストーリーを説明し、チームが質問する
2. 各自が**同時に**カード（1,2,3,5,8...）を出す — 声の大きい人への同調を防ぐ
3. 最大と最小を出した人が理由を説明する — ここで認識のズレ（考慮漏れ・前提の違い）が炙り出される
4. 議論して再投票。だいたい収束したら決定

**見積りの最大の価値は数字ではなく、この「認識のズレの発見」にあります。**

### ベロシティと予測

- **ベロシティ** = 1スプリントで完了したポイントの合計（実績値）
- 直近3〜5スプリントの平均を使い、「残り120ポイント ÷ ベロシティ20 ≒ 6スプリント」のように**着地を予測**する
- ベロシティは**チームごとの内部指標**。チーム間比較や人事評価に使うと、数字を膨らませるインフレが起きて予測に使えなくなる（⑦章参照）

### ノーエスティメート／実務の簡略化

ストーリーを十分小さく均等に割れるなら、ポイントを付けず**枚数（スループット）**で予測する流儀もあります（#NoEstimates）。また近年は「Tシャツサイズ（S/M/L）で粗く見積る」「詳細見積りはやめて分割に注力する」チームも増えています。**見積りは目的（予測と会話）を果たす最小限でよい**、が実務的な結論です。

## 4. リリース計画とロードマップ

- スプリント単位の計画とは別に、四半期程度の**リリース計画／ロードマップ**を持つのが普通
- アジャイルのロードマップは日付と機能の確約リストではなく、**目標（アウトカム）ベース**で書くのがコツ（例:「Q3: 新規登録の離脱率を半減」）
- **スコープ・期日・品質のうち、固定するのは最大2つ**。期日固定なら、スコープを調整弁にする（優先順位の低いストーリーを落とす）
