# ⑩ アジャイルテスト — 品質をフェーズからチームの習慣へ

[← 総合インデックスに戻る](README.md)

ウォーターフォールではテストは「最後の工程」でしたが、アジャイルではテストは**開発と同時並行の継続的な活動**になります。⑤章の自動テスト・TDDを前提に、この章ではテスト戦略とQAの役割を深掘りします。

## 1. アジャイルテストの4象限

Brian Marick が提唱し、書籍『Agile Testing』（Lisa Crispin / Janet Gregory）で広まった、テストを整理する定番の地図です。

```
                        ビジネス面
            ┌───────────────┬───────────────┐
            │ Q2: 機能テスト     │ Q3: 探索的テスト    │
            │  ・受け入れテスト    │  ・ユーザビリティ    │
  チームを   │  ・ATDD/BDD       │  ・UAT(受入検証)    │  製品を
  支援する   │  （自動が主）       │  （手動が主）       │  批評する
            ├───────────────┼───────────────┤
            │ Q1: ユニット・      │ Q4: 品質特性テスト   │
            │     コンポーネント   │  ・性能・負荷       │
            │  ・TDDの成果物     │  ・セキュリティ      │
            │  （自動）          │  （ツール利用）      │
            └───────────────┴───────────────┘
                        技術面
```

- **Q1/Q2（左側）**: 「作る前・作りながら」チームを導くテスト。**自動化して回帰資産にする**
- **Q3/Q4（右側）**: 「作ったものを批評する」テスト。**人の知性（探索）や専門ツール（負荷・脆弱性）**が主役
- ありがちな偏り: Q1だけ（ロジックは固いがUXが壊れている）、Q3だけ（毎リリース手動総点検で疲弊）。**4象限をスプリント内でバランスさせる**のが戦略

## 2. シフトレフト — 「テストは最初に」

バグの修正コストは発見が遅いほど桁で増えます（要件段階なら会話1つ、本番なら障害対応＋信頼低下）。テスト活動を左（早い工程）へ移すのが**シフトレフト**です。

### スリーアミーゴス

ストーリー着手**前**に、**ビジネス（PO）・開発者・テスター**の三者で受け入れ条件を具体例ベースで会話する短いミーティング。

> PO: 「送料は5,000円以上で無料にしたい」
> テスター: 「ちょうど5,000円は？ クーポンで4,900円になったら？ 沖縄・離島は？」
> 開発者: 「割引適用の順序で結果が変わりますね。仕様を決めましょう」

**実装前に**このズレを潰すのが最も安いテストです。具体例はそのまま受け入れテストになります（Specification by Example / 実例による仕様）。

### BDD（振る舞い駆動開発）とGherkin

受け入れ条件を Given-When-Then の構造化テキスト（Gherkin記法）で書き、Cucumber や SpecFlow 等のツールで**そのまま自動テストとして実行**する手法。

```gherkin
シナリオ: 5,000円以上の注文は送料無料
  前提 カートに合計5,200円の商品が入っている
  もし 注文を確定する
  ならば 送料は0円である

シナリオ: クーポン適用後に5,000円を下回る場合は送料がかかる
  前提 カートに合計5,200円の商品が入っている
  かつ 500円引きクーポンを適用している
  もし 注文を確定する
  ならば 送料は450円である
```

価値の中心はツールではなく、**PO・開発・QAが同じ具体例を共有すること**。ツール導入だけして会話が伴わないBDDは維持コストだけが残ります。

## 3. 探索的テスト

台本（テストケース）通りの確認ではなく、**学習しながらテスト設計と実行を同時に行う**知的活動です。自動テストが「既知の壊れ方の再発防止」なら、探索的テストは「**未知の壊れ方の発見**」を担います。

### セッションベースドテスト

探索を仕組みにする方法。

1. **チャーター**（探索の目的）を決める: 例「割引と送料の組み合わせ周りを、金額の境界値を中心に60分探索する」
2. タイムボックス（60〜90分）で集中して探索し、気づき・バグ・新たな疑問をメモする
3. 短い報告（デブリーフィング）で共有し、次のチャーターに繋げる

### 探索の観点ヒント

- **境界と極端**: 0件・1件・最大件数、0円・マイナス・巨大な数、空文字・絵文字・長大文字列
- **中断と順序**: 戻るボタン、二重送信、途中でセッション切れ、想定外の画面遷移順
- **状態の組み合わせ**: 解約済みユーザーで購入、権限変更直後の操作
- **CRUDの裏**: 削除したものへの参照、編集の同時実行

## 4. スプリント内でテストを回す

### 「テストは翌スプリント」が生む破綻

実装スプリントの翌スプリントでQAが検証する方式（ステージング運用）は、一見合理的ですが次を招きます。

- バグ修正が常に割り込みになり、次スプリントの計画が壊れる
- 「完成」の定義が曖昧になり、仕掛りが積み上がる（ミニウォーターフォール化）

**原則はストーリー単位で同一スプリント内にテストまで完了**（DoDに含める)。終わらないなら、ストーリーが大きすぎるシグナルです（④章の分割へ）。

### QA・テスターの役割の変化

| 従来 | アジャイルチームでの姿 |
|---|---|
| 最後の関門・検品係 | **品質のコーチ・チームの相談役** |
| テストケースを大量に書いて実行 | スリーアミーゴスで**バグを作る前に防ぐ** |
| 開発と別部隊 | チームの一員としてスプリントに参加 |
| 手動回帰テストの反復 | 回帰は自動化へ移譲し、**探索的テストと戦略に集中** |

「品質はQAの責任」から「**品質はチーム全員の責任、QAはその専門家**」への転換が核心です（Whole Team Approach）。

## 5. 回帰テスト資産の運用

- **フレーキーテスト**（たまに落ちるテスト）は即隔離・修正する。放置すると「落ちてもリトライすればいい」文化になり、テストが信号の意味を失う
- テストコードも**リファクタリング対象**。重複だらけのテストは仕様変更のたびに膨大な修正を生む（テストヘルパー・ビルダーの整備）
- **テストデータ管理**: 本番データのコピー依存を避け、テストごとに必要データを組み立てる（Fixture/Factory）。個人情報はマスキング
- E2Eは**クリティカルパス（購入・ログイン等）に絞る**。画面の細部はユニット・コンポーネントテストへ押し下げる（⑤章のピラミッド）

## 6. 品質特性（非機能）テストのスプリントへの組み込み

- 性能・セキュリティを「リリース前の一大イベント」にしない。**CIに常設**する: 性能の自動ベンチマーク（劣化したら失敗）、依存パッケージの脆弱性スキャン、静的解析
- 大掛かりな負荷試験・侵入テストが必要な場合は、**専用のストーリー/スパイクとしてバックログに載せ**、リリース計画に組み込む
- 「非機能要件」も受け入れ条件として書ける: 「検索結果は95パーセンタイルで1秒以内に返る」
