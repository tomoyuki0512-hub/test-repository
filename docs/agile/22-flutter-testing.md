# ㉒ Flutterのテスト戦略 — テストの種類と「テストゼロ」からの現実的な導入

[← 総合インデックスに戻る](README.md)

Flutterアプリでのテストの種類を整理し、後半では「**既存アプリにテストが1本もない**」状態から、コストを爆発させずに導入する手順をまとめます。[⑩ アジャイルテスト](10-agile-testing.md)と[⑭ レガシーコード](14-legacy-code.md)のFlutter具体版です。

## 1. Flutterのテストの種類 — 全体マップ

| 種類 | 何を守るか | 実行速度 | 実行環境 | コスト |
|---|---|---|---|---|
| 静的解析（analyze / lints） | 型エラー・非推奨API・コード規約 | 数秒 | どこでも | ほぼゼロ（設定のみ） |
| **ユニットテスト** | ロジック（計算・変換・状態遷移） | 1件数ms | Dart VM（端末不要） | 低 |
| **ウィジェットテスト** | 1画面/1部品の表示と操作 | 1件数十ms〜 | Dart VM（端末不要） | 低〜中 |
| **ゴールデンテスト** | 見た目のデグレ（ピクセル比較） | 速い | Dart VM | 中（画像更新の運用） |
| **インテグレーションテスト** | 実機での一気通貫フロー | 1本数十秒〜分 | 実機/エミュレータ | 高 |
| 手動・探索的テスト | 未知の壊れ方・使い勝手 | 人間 | 実機 | 都度 |

**Flutterの最大の強みは「ウィジェットテストが軽い」こと**です。他のフレームワークではUIテスト＝重いE2Eになりがちですが、Flutterは画面の描画とタップ・入力を**端末なし・ミリ秒単位**で検証できます。テストピラミッド（[⑤章](05-engineering-practices.md)）の中段が異常に安く手に入る、と覚えてください。

### ① ユニットテスト（`test` パッケージ）

純粋なDartのロジックを検証します。最も速く、最も安定。

```dart
test('送料は5,000円以上で無料になる', () {
  expect(calcShipping(cartTotal: 5200), 0);
  expect(calcShipping(cartTotal: 4999), 450);
  expect(calcShipping(cartTotal: 5000), 0); // 境界値
});
```

- 依存（API・DB）は `mocktail` / `mockito` で偽物に差し替える
- Riverpod なら `ProviderContainer`、Bloc なら `bloc_test` で**状態管理のロジックを画面なしで**検証できる

### ② ウィジェットテスト（`flutter_test`）

画面や部品を仮想的に描画し、表示・タップ・入力を検証します。

```dart
testWidgets('キャンセルボタンを押すと確認ダイアログが出る', (tester) async {
  await tester.pumpWidget(const MaterialApp(home: OrderDetailPage(order: testOrder)));
  await tester.tap(find.text('キャンセルする'));
  await tester.pumpAndSettle();
  expect(find.text('本当にキャンセルしますか？'), findsOneWidget);
});
```

- 発送済み注文ではボタンが出ない、ローディング中はスピナー表示、といった**状態×表示の組み合わせ**を高速に網羅できる
- エミュレータ不要なのでCIでも数百本を数分で回せる

### ③ ゴールデンテスト（見た目の回帰）

ウィジェットの描画結果を画像として保存し、以後の変更でピクセル差分を検出します。

```dart
await expectLater(find.byType(PriceCard), matchesGoldenFile('goldens/price_card.png'));
```

- **共通コンポーネント（ボタン・カード等のデザインシステム部品: [⑮章](15-design-collaboration.md)）に特に有効**。意図しない見た目のデグレを機械が見張る
- 意図した変更のたびに基準画像を更新する運用が必要。乱発せず「壊れると困る部品」に絞る

### ④ インテグレーションテスト（`integration_test`）

実機・エミュレータでアプリ全体を起動し、実際のフローを通します。ネイティブ機能（権限ダイアログ等）まで触るなら `patrol`、多端末での実行は Firebase Test Lab などのデバイスファームが選択肢です。

- **遅く・不安定になりやすいので本数を絞る**のが鉄則。「ログイン→主要操作→購入」のようなクリティカルパス2〜3本だけに
- 課金など本物を叩けないものはサンドボックス・スタブサーバーで（[Flutterサブスクリプションのテスト・QAガイド](../flutter-subscription-testing.md)も参照）

### Flutter版テストピラミッド（推奨比率のイメージ)

```
        ▲ インテグレーション: クリティカルパス2〜3本だけ
      ▲▲▲ ウィジェット+ゴールデン: 主要画面の状態網羅・共通部品
  ▲▲▲▲▲▲ ユニット: ロジック・状態管理・変換処理（最多）
──────────
  土台: flutter analyze + lints + フォーマッタ（CIで常時）
```

## 2. 前提: テストしやすさはアーキテクチャで決まる

「テストが書けない」の大半は、テスト技法ではなく**ロジックがウィジェットに埋まっている**ことが原因です。

- `build()` メソッドや `onPressed` の中に計算・分岐・API呼び出しが直書きされていると、ユニットテストできない（＝レガシーコードの定義そのもの: [⑭章](14-legacy-code.md)）
- 対策は状態管理（Riverpod / Bloc 等）とリポジトリパターンで**「表示」と「ロジック」を分離**すること。分離した瞬間、ロジックは安いユニットテストで守れるようになる
- ただし**全面リアーキテクチャは不要**。次章の手順の通り、「これから触る場所」だけ分離していけばよい

## 3. 「テストが1本もない」からの現実的な導入手順

ここが本題です。**総工数の見積りから始めない**でください。「全部にテストを書いたら○人月」という計算は必ず巨大な数字になり、提案ごと却下されます。正解は⑭章と同じ——**全体を目指さず、これから触る場所だけを守る**。コストは一括投資ではなく、日々の開発に薄く溶かします。

### ステップ0: CIと静的解析（半日〜1日）

- `flutter analyze` と `flutter test` をPRごとに走らせるCIを作る（GitHub Actionsなら数十行）
- `flutter_lints`（または `very_good_analysis`）を入れ、**今より悪化したらCIが落ちる**状態を作る
- テストが0本でもこの土台だけで価値がある。**最初の1本が書かれたときから自動で守られ始める**

### ステップ1: 「触るコードにだけ書く」ルール（コストほぼゼロで開始）

- 「新規・変更するコードにはテストを書く」をDoD（[②章](02-scrum.md)）に追加する。**既存コードへの遡及はしない**
- バグ修正は最高の起点: **直す前にそのバグを再現するテストを書く**（レッド→修正でグリーン）。修正のついでなので追加コストが最小で、しかも再発が二度と起きない
- これだけで、変更頻度の高い場所（＝壊れると痛い場所）から自然にテストが生えていく（ホットスポット集中: [⑭章](14-legacy-code.md)）

### ステップ2: ロジックの「切り出し＋ユニットテスト」（触るついでに）

- 触る画面にロジックが埋まっていたら、**その部分だけ**関数・クラスに切り出してユニットテストを付ける（スプラウト: [⑭章](14-legacy-code.md)）
- 切り出しの単位は小さくてよい。「価格計算だけ」「バリデーションだけ」で十分
- 既存の挙動が正しいか不明なら、まず**現状の出力をそのまま期待値にする仕様化テスト**で固定してから変更する

### ステップ3: 主要画面にウィジェットテスト（スプリントに数本ずつ）

- アプリの**主要5〜10画面**について「正常表示・ローディング・エラー・空」の状態別テストを、スプリントごとに数本ずつ積む
- 共通コンポーネントにはゴールデンテストを併設
- 目安コスト: 1画面あたり0.5〜1日。改善枠（スプリント容量の10〜20%: [⑤章](05-engineering-practices.md)）に載せ、**専用の「テスト整備プロジェクト」は立てない**（日常に溶かす）

### ステップ4: クリティカルパスのインテグレーションテスト（2〜3本だけ）

- 「これが壊れたら事業が止まる」フロー（例: ログイン→商品閲覧→購入）だけを実機テスト化
- リリース前の手動確認チェックリストが既にあるなら、その**上位2〜3項目を自動化する**と考えると選びやすい

### 到達イメージ（数ヶ月後）

| 時期 | 状態 |
|---|---|
| 1週目 | CI稼働。テスト0本でも劣化が止まる |
| 1〜2ヶ月 | バグ修正・機能追加のたびにユニット/ウィジェットテストが数十本たまる。よく触る画面から安全になる |
| 3〜6ヶ月 | 主要画面の状態網羅＋クリティカルパス自動化。リリース前の手動確認が大幅短縮 |
| その後 | 「テストを書く方が速い」が体感になり、ルールなしでも書かれるようになる |

## 4. やってはいけないこと

| アンチパターン | なぜダメか |
|---|---|
| 「カバレッジ80%」を最初に掲げる | 無意味なテストの量産で応えられてしまう（[⑦章](07-metrics-adoption.md)グッドハートの法則）。数字でなく「触る場所ルール」で |
| テスト整備専用の大型プロジェクト化 | 機能開発が止まる提案は通らないし、通っても優先度で潰される。日常に溶かすのが唯一持続する形 |
| いきなり全画面のE2E自動化 | 遅く不安定なテスト群が「たまに落ちるが皆無視する」存在になり、信号として死ぬ（[⑩章](10-agile-testing.md)フレーキーテスト） |
| 古いコードへの一括遡及 | 変更されないコードのテストは投資対効果が最低。触らない汚いコードは触らないまま置く（[⑭章](14-legacy-code.md)） |
| テストを書かない前提の続行 | スプリントを重ねるほど変更が怖くなり失速する（[⑤章](05-engineering-practices.md)の片輪走行）。導入コストは「払うか」ではなく「いつ薄く払うか」の問題 |

## 5. コストの説明の仕方（上司・顧客向け）

- 「テスト整備に○人月ください」ではなく「**開発容量の10〜20%を品質に充てる運用にします。3ヶ月後にリリース前の手動確認時間と障害率で効果を示します**」と提案する（[⑰章ケースC](17-case-studies.md)と同じ構図）
- 効果の証拠に使える数字: リリース前の手動テスト時間・本番障害/ロールバック件数・バグ修正の平均時間（Four Keys: [⑦章](07-metrics-adoption.md)）
- 「バグ再現テスト」から始めると、**同じバグの再発がゼロになる**という誰にでも分かる成果が最初の1ヶ月で出る
