# ⑤ 技術プラクティス — TDD・CI/CD・リファクタリング

[← 総合インデックスに戻る](README.md)

アジャイルは「短いサイクルで変更し続ける」開発です。**変更し続けても品質と速度が落ちない技術的な土台**がないと、スプリントを重ねるほどバグと手戻りが増えて失速します。この章はその土台となるプラクティスを扱います（多くはXP由来です）。

## 1. なぜ技術プラクティスが必須なのか

```
技術的土台なしのアジャイル:
  スプリント1 ██████ 速い
  スプリント5 ████   バグ修正に追われ減速
  スプリント10 ██    変更が怖くて何も触れない ← 「アジャイルは失敗した」

技術的土台ありのアジャイル:
  スプリント1 ████   テスト整備分やや遅い
  スプリント5 █████  安定
  スプリント10 ██████ 自信を持って変更できる
```

鍵は「**変更のたびに壊れていないことを高速に確認できる**」こと。その中心が自動テストです。

## 2. 自動テストとテストピラミッド

```
        ▲  E2E/UIテスト（少なく）: 遅い・壊れやすい・本物に近い
       ▲▲  統合テスト（中くらい）: API・DB連携の確認
     ▲▲▲▲  ユニットテスト（多く）: 速い・安定・ロジック単位
```

- **ユニットテストを土台に厚く**、上に行くほど数を絞るのが原則（逆三角形になると遅く不安定な「アイスクリームコーン」アンチパターン）
- テストは「後で書く」と書かれない。**ストーリーの完成の定義（DoD）にテストを含める**のが定石

## 3. テスト駆動開発（TDD）

実装より先にテストを書き、**レッド → グリーン → リファクタ**の小さなサイクルを回します。

1. **レッド**: これから作る振る舞いのテストを書く → 失敗することを確認
2. **グリーン**: テストを通す**最小限**の実装をする
3. **リファクタ**: テストが通る状態を保ったまま、コードを整理する

### 効果と誤解

- 効果: 設計が使う側の視点から始まる／テストが仕様のドキュメントになる／常に動く状態が保たれる
- TDDは「テスト技法」というより**設計と進め方の技法**。カバレッジのためにやるものではない
- 派生として、受け入れ条件を自動化する**ATDD/BDD**（Given-When-Then をそのままテストにする）がある

## 4. リファクタリング

**外から見た振る舞いを変えずに、内部構造を改善する**こと。アジャイルでは設計を最初に完成させない代わりに、**リファクタリングで設計を継続的に育てます**（進化的設計）。

- 自動テストがある前提の活動（テストがないリファクタリングは単なる「祈り」）
- 「あとでまとめて」ではなく、**機能追加のついでに小さく**行う（ボーイスカウトルール: 来た時よりきれいにして帰る）
- 大きな構造変更は、ストーリーとしてPOと交渉するのではなく、**関連する機能開発に織り込む**のが現実的

### 技術的負債

締切のために意図的に取った近道や、学習の結果として見えてきた設計のズレは**技術的負債**として蓄積します。負債は「返済」（リファクタリング）しなければ利息（変更コスト増）を払い続けることになります。負債の存在を**見える化**（バックログに載せる・マーカーコメント）し、毎スプリント一定割合（例: 10〜20%）を改善に充てるチームが多いです。

## 5. 継続的インテグレーション（CI）と継続的デリバリー（CD)

### CI（Continuous Integration）

- 全員が**1日に何度も**共有ブランチに統合し、そのたびに自動でビルド＋テストが走る
- 統合を遅らせるほど衝突（コンフリクト・結合バグ）は複利で膨らむ。**小さく頻繁に**が原則
- 壊れたら**直すことを最優先**する（壊れたまま放置するとCIは信号の意味を失う）
- 代表的ツール: GitHub Actions、GitLab CI、CircleCI、Jenkins

### CD（Continuous Delivery / Deployment）

- **Continuous Delivery**: いつでもリリース**できる**状態を常に保つ（リリース判断は人間）
- **Continuous Deployment**: テストを通れば**自動で本番デプロイ**まで行う
- リリースが「一大イベント」でなくなることで、小さく頻繁に出せる → 1回あたりのリスクが下がる → さらに頻繁に出せる、という好循環が生まれる

### ブランチ戦略とフィーチャーフラグ

- アジャイルと相性が良いのは**トランクベース開発**（長命ブランチを作らず、小さな変更を頻繁にmainへ統合する）
- 未完成機能は**フィーチャーフラグ**で本番コード内に隠し、統合とリリースを分離する（「デプロイ」と「リリース」の分離）
- 長命のフィーチャーブランチ＋まとめてマージは、CIの効果を打ち消すアンチパターン

## 6. コードレビューとペアプログラミング

| 方式 | 特徴 |
|---|---|
| プルリクエストレビュー | 非同期・記録が残る。レビュー待ちがリードタイムを延ばしやすい（小さいPRが対策） |
| ペアプログラミング | 2人で1画面。レビューが常時行われ、知識移転が速い。疲れるので時間を区切る |
| モブプログラミング | チーム全員で1画面。設計判断の共有と合意形成が速い。難所・共通基盤に向く |

どれか一つが正解ではなく、変更の重要度や目的（品質確認か知識移転か）で使い分けます。**コードの共同所有**（誰でもどこでも直せる）を支えるのがレビューとテストです。

## 7. その他の重要プラクティス

- **YAGNI**（You Aren't Gonna Need It）: 将来使うかもしれない機能・抽象化は作らない。必要になった時に作る
- **KISS / シンプルデザイン**: 動く最もシンプルな設計を選ぶ。シンプルさは「変更のしやすさ」への投資
- **スパイク**: 技術的な不確実性を、時間を区切った調査・プロトタイプで先に潰す（成果はコードでなく「知見」）
- **インフラのコード化（IaC）・自動化**: 環境構築やデプロイを再現可能にし、「動く環境を作る」コストをゼロに近づける
- **監視・可観測性**: リリース後の挙動（エラー率・レイテンシ・利用状況）を見て初めてフィードバックループが閉じる

## 8. まとめ — プロセスと技術は両輪

| プロセス面（②〜④章） | 技術面（本章） |
|---|---|
| 短いサイクルで計画を見直す | 短いサイクルで安全に変更できるコードとパイプライン |
| スプリントごとに出荷可能に | CI/CD・自動テストで「出荷可能」を機械的に保証 |
| 変化を歓迎する | リファクタリングと進化的設計で変化のコストを下げる |

どちらか片方だけでは持続しません。「スクラムはやっているが技術プラクティスがない」チームが数ヶ月後に失速するのは、この片輪走行が原因です。
