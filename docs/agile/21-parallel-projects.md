# ㉑ 並行案件とブランチ戦略 — 同じ画面・ソースが被り、先に始まった案件が後に終わるとき

[← 総合インデックスに戻る](README.md)

2つの案件が同じ画面・同じソースコードを触り、しかも**先に始まった案件Aが後にリリースされ、後から始まった案件Bが先にリリースされる**——並行開発で最も事故が起きやすいパターンです。この章では、ブランチ戦略・スケジュール・品質・工数管理の4面から標準フローをまとめます。

## 1. 問題の構造を正確に捉える

```
時間 ──────────────────────────────────────▶
案件A（大型改修）  ├────────── 開発 ──────────────┤ リリース(後)
案件B（小型・急ぎ）        ├──── 開発 ────┤ リリース(先)
                          ▲同じ画面・同じソースを変更▲
```

何もルールを決めないと、次の事故が起きます。

| 事故 | 起きること |
|---|---|
| 統合地獄 | Bリリース後、Aのブランチが数ヶ月分乖離しておりマージ不能級のコンフリクト |
| 先祖返り（デグレード） | AのリリースでBの変更が消える（Aが古いコードベースのまま上書き） |
| 見えない工数 | Aチームの「Bへの追従作業」が計画外で発生し、残業で吸収されて表面化しない |
| テスト漏れ | 統合後の再テスト範囲が曖昧で、被った画面のバグが本番で発覚 |

**原則はただ1つ: 「開始順」は忘れ、「リリース順」で考える。** 先にリリースされるものが先にmainに入り、後からリリースされるもの（A）が**追従する側**になります。追従のコスト・リスクをどう最小化するかが、この章の全てです。

## 2. ブランチ戦略 — 3つの選択肢と使い分け

### 選択肢① トランクベース＋フィーチャーフラグ（本命）

A・B両方の変更を**小さく分割して常時mainへマージ**し、未完成のAの機能は**フィーチャーフラグ**で隠します（[⑤章](05-engineering-practices.md)）。

```
main ──●──●──●──●──●──●──●──●──▶
        A1   B1  A2  B2(リリース)  A3  …  AフラグON(リリース)
        （Aの変更はフラグOFFで本番に入り続けるが、ユーザーには見えない）
```

- **コンフリクトが構造的に消える**: 乖離が最大でも数日分なので、衝突は小さいうちに解消される
- Bのリリース時、Aのコードは既にmainに同居してテストされている → 先祖返りが起きようがない
- 代償: フラグの管理コスト（リリース後の削除を忘れない）、未完成コードが本番に入ることを許容する文化・テストが必要
- **同じ画面のUIが大きく変わる場合**もフラグで分岐可能（旧UI/新UIをフラグで切替）。分岐が複雑になりすぎるならコンポーネントを新旧2つ用意してフラグで差し替える

### 選択肢② リリース順マージ＋定期同期（フラグが使えない場合の現実解）

案件ごとのブランチを維持する場合の規律です。

```
main ────────────●(Bマージ・リリース)──────────●(Aマージ・リリース)──▶
branch B ──┬───┘
branch A ──┴──────── ◀─週次でmainを取り込む ──┴─
                      （Bリリース直後は必ず即日取り込み）
```

- **mainへのマージはリリース順**（B→A）。Aが先にマージ済みでも、Bが先に出るならAは待つかフラグで隠す
- Aブランチは**週1回以上、mainを取り込む**（逆マージ/rebase）。「最後にまとめて」が統合地獄の直接原因。Bのリリース直後は必ず即日取り込む
- 取り込みのたびに**Aブランチ上で自動テストを全実行**。CIは案件ブランチにも適用する
- 被るファイルの変更は**小さいコミット・早めのプッシュ**で互いに見える化する

### 選択肢③ 画面・モジュールの排他ロック（最後の手段）

「この画面はBが終わるまでAは触らない」という調整。単純ですが、**Aのスケジュールを人質に取る**方法であり、待ち時間という形で最も高くつきます。数日で終わる被りにだけ限定使用してください。

### 使い分けの目安

| 状況 | 推奨 |
|---|---|
| 自社プロダクト・CI/自動テストがある | ①トランクベース＋フラグ |
| 受託・検収の都合で案件ブランチが必要 | ②リリース順マージ＋週次同期 |
| 被りが小さく数日で解消する | ③一時的な排他 or ②の簡易運用 |
| 被りが巨大（同一画面の全面改修が衝突） | §3の「共通部分の先出し」で被り自体を減らす |

## 3. 設計でコンフリクトを減らす — 「共通部分の先出し」

ブランチ運用の工夫より効くのが、**被り自体を小さくする設計**です。

1. **共通リファクタリングを最初に切り出して単独リリースする**: A・B両方が触る画面・関数があるなら、まず「両案件が乗れる形への整理」（コンポーネント分割・関数の分離）だけを小さな案件0としてmainに入れる。以後AとBは**別々のファイル・別々のコンポーネント**を触る形になり、コンフリクトが激減する
2. **触るファイルの重なりを事前に予測する**: 着手前に両案件の変更対象ファイル一覧を突き合わせる（変更予定のディレクトリを洗い出すだけでも有効）。重なりが大きい箇所は担当を寄せる（同じ人・同じペアが両案件のその部分を書く）
3. **インターフェースを先に合意する**: 被る境界（APIの形・コンポーネントのprops）だけ先に両案件で合意して固定し、中身はそれぞれ自由に進める

## 4. スケジュール管理

- **統合をタスクとして計画に載せる**: 「Bリリース後のAへの取り込み・再テスト」は必ず発生する作業。Aの計画に**統合スプリント（または統合タスク数日分）を最初から積む**。載っていない統合作業は残業で吸収される（§6）
- **リリース順を疑う**: そもそもBの内容がAの前提を壊すなら、リリース順の入れ替え・Aの分割リリース（先に出せる部分を出す）を検討する。スコープの分割（[④章](04-requirements-estimation.md)）はスケジュール衝突の最強の解決策
- **依存の見える化**: 両案件のタイムラインと「被りイベント」（Bリリース日・Aの取り込み日・共同テスト日）を1枚の線表にし、双方のチームと関係者で共有する
- **Bの遅延はAに波及する**と認識する: Bが2週間遅れると、Aの統合・再テストも後ろにずれる。Aの締切がタイトなら、Bの遅延がAのリスクとして管理台帳に載っていなければならない

## 5. 品質管理

- **被った画面の回帰テストを自動化する**: 統合のたびに手動で全画面を確認するのは持続しない。被る画面のクリティカルパスはE2E自動テスト（[⑩章](10-agile-testing.md)）にして、取り込みのたびに機械で検証する
- **「統合後の再テスト範囲」を事前に定義する**: Bリリース後にAが取り込んだら何を再テストするか（被り画面全部？ Aの全機能？）を、両案件のQA観点で**着手前に**合意しておく。曖昧なままだと「多分大丈夫」で本番に出る
- **AはBの受け入れテストを引き継ぐ**: Bが作った自動テストはAブランチでも常時走らせる。「Aのリリースで Bの機能が壊れていないこと」を機械で保証するのが先祖返りへの根本対策
- **統合直後は探索的テストを1セッション**: 自動テストは既知の壊れ方しか守らない。被り画面を対象にチャーターを立てて60分探索する（[⑩章](10-agile-testing.md)）

## 6. 工数管理

並行案件で最も見落とされるのが**統合・追従の工数**です。

- **追従作業を明示的なチケットにする**: 「mainの取り込み」「コンフリクト解消」「統合後の再テスト」をバックログ項目にして計測する。目安として、被りが大きい場合**Aの総工数の10〜20%が統合系に消える**前提で計画する（実績で補正）
- **コスト負担のルールを決める（受託・部門別予算の場合）**: 原則は「**後からリリースする側（A）が追従コストを持つ**」。ただしBが共通部分を大きく作り替えた場合の追加コストは案件間で協議する——これを**着手前に**合意しておかないと、発生時に押し付け合いになる
- **統合工数を記録して次に活かす**: 「今回の並行開発で統合に何人日使ったか」を残す。次に並行案件の話が出たとき、「並行させるコスト」を数字で示して、そもそも並行を避ける（順番にやる・チームを分ける・共通部分を先出しする）判断材料にする
- **見えない残業に吸収させない**: 追従作業がチケット化されていないと、Aチームの疲弊として現れる。ふりかえり（[⑪章](11-retrospectives.md)）で「計画外の統合作業に何時間使ったか」を確認する

## 7. 体制とコミュニケーション

- **同一チームが両案件を持つ場合**: バックログを1本にし、WIP制限で同時着手を絞る（[③章](03-kanban-xp.md)）。「Aを止めてBに全振り→Bリリース→Aに戻る」の直列化も、実は最速の選択肢になり得る（コンテキストスイッチと統合コストが消えるため）
- **別チームの場合**: 週次の**合同同期**（15〜30分。被り箇所の進捗・次週触るファイル・リリース日程だけ）＋被りコードの**共有オーナー**（両案件のレビューに必ず入る人）を置く
- **レビューをクロスさせる**: 被りファイルのPRは相手案件のメンバーもレビューする。「知らないうちに変わっていた」の予防として最も安い手段

## 8. 標準フロー（チェックリスト）

```markdown
## 着手前（両案件のキックオフ時）
- [ ] 変更対象ファイル・画面の一覧を突き合わせ、被りを特定した
- [ ] ブランチ戦略を選定した（フラグ / リリース順マージ+週次同期 / 一時排他）
- [ ] 共通部分の先出しリファクタリングを検討した
- [ ] リリース順を確認し、後リリース側（A）に統合タスクを計画として積んだ
- [ ] 追従コストの負担ルールを合意した（受託・部門別予算の場合）
- [ ] 統合後の再テスト範囲をQA観点で合意した

## 開発中
- [ ] A ブランチは週1回以上 main を取り込んでいる（選択肢②の場合）
- [ ] 被りファイルのPRは相手案件メンバーがレビューしている
- [ ] 週次の合同同期で「次週触るファイル」を共有している
- [ ] 被り画面のE2E自動テストが両ブランチのCIで走っている

## Bリリース時
- [ ] Aは即日〜数日内にmainを取り込み、全自動テスト＋被り画面の探索的テストを実施した
- [ ] Bの自動テストがAブランチでも走る状態にした（先祖返り防止）

## Aリリース時
- [ ] Bの機能が壊れていないことを自動テストで確認した
- [ ] 統合に使った実工数を記録した（次回の並行判断の材料に）
```
