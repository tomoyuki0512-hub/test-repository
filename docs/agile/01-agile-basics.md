# ① アジャイルの基礎

[← 総合インデックスに戻る](README.md)

## 1. アジャイル開発とは何か

アジャイル（Agile）は「俊敏な・機敏な」という意味の英単語です。アジャイル開発とは、**短いサイクルで「動くソフトウェア」を繰り返し届け、実際の利用やフィードバックから学びながら、計画・要求・設計を継続的に見直していく開発アプローチの総称**です。

重要なのは、アジャイルは特定のツールや手順書の名前ではなく、**価値観と原則の体系**だという点です。スクラムやカンバンは、その価値観を実現するための具体的な手法（フレームワーク）にあたります。

### よくある誤解

| 誤解 | 実際 |
|---|---|
| アジャイル＝速く作る手法 | 速さより「**間違ったものを作るムダの削減**」が本質。結果的に価値の提供は速くなる |
| ドキュメントを書かない | 「包括的なドキュメント**より**動くソフトウェア」であり、書かないとは言っていない。必要なものは書く |
| 計画を立てない | 計画は立てる。ただし「計画に従うこと」より「変化への対応」を優先し、計画を**更新し続ける** |
| 設計しない | 設計はする。最初に全部決めず、**進化的に設計する**（⑤章参照） |
| 何でも途中で変えてよい | スプリント内のゴールは原則固定。変更は次のスプリントの計画で取り込む（スクラムの場合） |

## 2. なぜ生まれたか — 歴史的背景

### ウォーターフォール開発の限界

1970〜90年代の主流は、**要件定義 → 設計 → 実装 → テスト → リリース**と工程を一方向に進める「ウォーターフォール型」でした。この方式は、要件が最初に確定できるプロジェクトでは機能しますが、次の問題が繰り返し起きました。

- 要件定義時点では**ユーザー自身も本当に欲しいものが分かっていない**
- 動くものを見られるのが数ヶ月〜数年後で、**手戻りのコストが巨大**になる
- 市場や技術の変化に**計画が追いつかない**
- 工程間の引き継ぎドキュメントが膨らみ、**作ること自体が目的化**する

### 1990年代 — 軽量プロセスの登場

こうした問題意識から、1990年代に「軽量（lightweight）プロセス」と呼ばれる手法が各地で生まれました。

- **スクラム**（1993年頃〜、Jeff Sutherland / Ken Schwaber）
- **XP: エクストリーム・プログラミング**（1996年頃〜、Kent Beck）
- **FDD、Crystal、DSDM** など

### 2001年 — アジャイルソフトウェア開発宣言

2001年2月、これらの手法の提唱者17名が米国ユタ州スノーバードに集まり、共通する価値観を「**アジャイルソフトウェア開発宣言（Agile Manifesto）**」としてまとめました。「アジャイル」という呼び名はここで決まりました。

## 3. アジャイルマニフェスト — 4つの価値

> 私たちは、ソフトウェア開発の実践あるいは実践を手助けをする活動を通じて、よりよい開発方法を見つけだそうとしている。この活動を通して、私たちは以下の価値に至った。

| 左側（も価値がある） | | 右側（**より価値をおく**） |
|---|---|---|
| プロセスやツール | よりも | **個人と対話** |
| 包括的なドキュメント | よりも | **動くソフトウェア** |
| 契約交渉 | よりも | **顧客との協調** |
| 計画に従うこと | よりも | **変化への対応** |

ポイントは「左側にも価値があるが、右側に**より**価値をおく」という相対的な表現です。プロセス・ドキュメント・契約・計画を否定しているわけではありません。

## 4. 背後にある12の原則（要約）

1. **顧客満足を最優先**し、価値あるソフトウェアを早く継続的に届ける
2. 要求の変更は**開発後期でも歓迎**する。変化を顧客の競争力に変える
3. 動くソフトウェアを**2〜3週間から2〜3ヶ月の短い間隔**でリリースする
4. ビジネス側と開発者は**毎日一緒に**働く
5. **意欲ある人々**を集め、環境と支援を与え、仕事を任せる
6. 最も効率的な情報伝達は**対面での会話**である
7. **動くソフトウェアこそが進捗の最も重要な尺度**である
8. **持続可能な開発**を促進する。一定のペースを維持できるようにする
9. **技術的卓越性と優れた設計**への継続的な注意が機敏さを高める
10. **シンプルさ**（ムダなく作れる量を最大限にすること）が本質である
11. 最良のアーキテクチャ・要求・設計は**自己組織化したチーム**から生み出される
12. チームは定期的に**やり方を振り返り**、行動を調整する

## 5. ウォーターフォールとの比較

| 観点 | ウォーターフォール | アジャイル |
|---|---|---|
| 進め方 | 工程を一方向に順番に進める | 短いサイクルを反復する |
| 要求 | 最初に固定する | 継続的に見直し、優先順位を並べ替える |
| 成果物の確認 | 最終工程でまとめて | 毎サイクルの終わりに動くもので |
| 変更への対応 | 変更管理手続きで抑制 | 歓迎し、次サイクルで取り込む |
| リスクの現れ方 | 後半に集中（結合テストで発覚など） | 早期に小さく現れる |
| 進捗の尺度 | 工程・ドキュメントの完了率 | 動くソフトウェア |
| 契約との相性 | 一括請負と相性が良い | 準委任・ラボ型と相性が良い |
| 向く領域 | 要件が固定的・規制で工程が定められている | 不確実性が高い・市場の変化が速い |

### どちらが優れているかではなく「不確実性」で選ぶ

- **要件・技術の両方が明確**（例: 既存システムの法令対応の一部）→ ウォーターフォールでも十分機能する
- **要件が不明確、または変化が速い**（例: 新規サービス、競争の激しい市場のプロダクト）→ アジャイルが強い
- 実務では、全体の節目管理はマイルストーンで行いつつ開発はアジャイルで回す**ハイブリッド**も多い

## 6. アジャイルが向くケース・向きにくいケース

### 向くケース

- 作るべきものの「正解」が事前に分からない（仮説検証が必要）
- ユーザーフィードバックを得やすい（Webサービス・アプリなど）
- 優先順位が頻繁に変わる
- 小さくリリースして価値を早く届けたい

### 向きにくい・工夫が必要なケース

- 人命に関わる等で**規制上、工程と文書が厳密に定められている**（医療機器・航空など）→ 規制対応と両立させる工夫が必要
- **リリースが物理的に1回きり**（組み込みの一部、ハードウェア同梱）→ 内部イテレーションは可能だが検証コストが高い
- 発注側が**一括請負で仕様固定**を強く求める → 契約形態から見直さないと形だけのアジャイルになる
- チームが頻繁に解散・結成される → チームの学習が蓄積されずアジャイルの強みが出にくい

## 7. 「アジャイルで速くなる」の正しい理解

アジャイルで速くなるのは、タイピングや実装が速くなるからではありません。

1. **作らないものを決められる** — 優先順位の低い機能に手を付ける前にやめられる
2. **手戻りが小さい** — 間違いに2週間で気づけば、損失は2週間分で済む
3. **統合・リリースの摩擦が減る** — 毎サイクル出荷可能な状態を保つため、最後の「結合地獄」がない
4. **チームが学習する** — 振り返りでプロセス自体が毎サイクル改善される

逆に言えば、**フィードバックを見ない・振り返りをしない・優先順位を見直さない**「形だけのスプリント」では速くなりません（⑦章のアンチパターン参照）。
