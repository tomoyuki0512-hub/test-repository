# ⑱ テンプレート集 — そのままコピーして使える道具箱

[← 総合インデックスに戻る](README.md)

現場でそのままコピーして使えるテンプレートを集めました。すべてMarkdownなので、チケットシステム・Wiki・議事録にそのまま貼れます。**テンプレートは出発点です。数スプリント使ったら、ふりかえりで自チーム用に改造してください。**

## 1. ユーザーストーリー（チケットテンプレート）

```markdown
## ストーリー
<ユーザーの種類> として、<実現したいこと> がしたい。
それは <理由・価値> のためだ。

## 受け入れ条件
- [ ] Given <前提> / When <操作> / Then <期待結果>
- [ ] Given <前提> / When <操作> / Then <期待結果>
- [ ] （エッジケース: 0件・上限・権限なし・通信断 を検討したか？）

## 補足
- デザイン: <Figmaリンク等>
- 決まっていないこと: <あれば明記。実装前にスリーアミーゴスで解消>
- 計測: <この機能の効果をどの指標で見るか>

## 見積り: <ポイント>
```

→ 解説: [④章](04-requirements-estimation.md)・[⑩章](10-agile-testing.md)

## 2. 完成の定義（DoD）ひな形

```markdown
# 完成の定義 (チーム: ____ / 最終更新: ____)
このリストを全て満たしたものだけを「完了」と呼ぶ。

- [ ] 受け入れ条件をすべて満たす
- [ ] コードレビューを1名以上が承認済み
- [ ] 追加・変更ロジックに自動テストがあり、CI が全て緑
- [ ] ステージング環境で動作確認済み
- [ ] デザイン確認済み（UI変更がある場合）
- [ ] 監視・ログ・フィーチャーフラグの設定済み（本番リリースを伴う場合）
- [ ] ドキュメント更新済み（API仕様・運用手順に変更がある場合）

※「デモできる」「本番に出せる」の2段階に分けたくなったら、それは
  リリースプロセスに障害物がある兆候。ふりかえりで扱うこと。
```

→ 解説: [②章](02-scrum.md)

## 3. ワーキングアグリーメント ひな形

```markdown
# チームの働き方合意 (作成: ____ / 次回見直し: ____)

## 時間
- コアタイム: 10:00-16:00（この間は同期的な相談OK）
- デイリー: 10:00-10:15、遅刻時は事後にスレッドで共有
- 会議のない日: 毎週水曜午後（集中作業日）

## コミュニケーション
- 仕様の質問: #proj-qa チャンネルへ。PO は4時間以内に一次反応
- 緊急連絡の定義: 本番障害・セキュリティのみ。それ以外は非同期
- 決定事項は必ずチケット/ADRに記録（口頭・DMで完結させない）

## 仕事の進め方
- WIP: 1人同時2枚まで
- レビュー依頼から24時間以内に着手（滞留したらペア作業に切替）
- 手が空いたら新しい着手より、詰まっているカードの支援を優先

## この合意の扱い
- 破られても責めない。ふりかえりで「守れない理由」を構造として扱う
```

→ 解説: [⑧章](08-team-and-culture.md)

## 5. ふりかえりアジェンダ（90分・5ステップ）

```markdown
# スプリント__ ふりかえり (日付: ____ / ファシリテーター: ____)

1. チェックイン (5分): 今の気分をひとこと
2. 前回のTryの確認 (5分): 実行した? 効果は? 続ける/やめる?
3. データ収集 (25分): 手法: [タイムライン / KPT / 帆船 / 4Ls]
   ※まず各自5分書いてから共有
4. 深掘り (20分): ドット投票で1テーマに絞り、なぜを繰り返す
   ※人ではなく仕組みの言葉で
5. Try決定 (20分): 1〜2個。担当・期限・効果の確認方法を決めて
   スプリントバックログへ
6. クローズ (5分): ROTI（この時間の価値を1-5で挙手）

## 決定事項
- Try:
- 担当:
- 効果確認: 次回ふりかえり冒頭で
```

→ 解説: [⑪章](11-retrospectives.md)

## 6. 仮説キャンバス（ディスカバリー用）

```markdown
# 仮説: <短い名前>

- 私たちは <対象ユーザー> が <課題> を持っていると信じている
- 根拠: <データ・インタビュー・問い合わせ件数など>
- <この機能・変更> を提供すれば <計測可能な変化> が起きるはずだ
- 成功基準: <指標> が <期間> で <値> になること
- 棄却基準: <期間> で <値> に届かなければこの仮説を捨てる
- 最小の検証方法: [インタビュー / LP / モック / Wizard of Oz / A/B]
- 検証コスト: __人日 / 検証期限: ____
```

→ 解説: [⑨章](09-product-discovery.md)

## 7. ADR（アーキテクチャ決定記録）

```markdown
# ADR-___: <決定の短いタイトル>
日付: ____ / 状態: [提案中 / 承認 / 廃止(→ADR-___に置換)]

## 文脈
<どんな課題・制約があったか。なぜ今決める必要があるか>

## 決定
<何をすることに決めたか（1〜3文で明確に）>

## 検討した代替案
- 案B: <内容> → 不採用の理由:
- 案C: <内容> → 不採用の理由:

## 結果として受け入れるトレードオフ
<この決定で失うもの・将来の制約>
```

→ 解説: [⑤章](05-engineering-practices.md)・[⑯章の会話例](16-sprint-simulation.md)

## 8. インセプションデッキ 記入シート（簡易版）

```markdown
# インセプションデッキ: <プロダクト名> (日付: ____ / 参加者: ____)

1. 我々はなぜここにいるのか:
2. エレベーターピッチ:
   「<課題> を抱える <対象顧客> 向けの <プロダクト名> は <カテゴリ> である。
    これは <重要な利点> ができ、<競合・代替手段> とは違って <差別化要素> が備わっている」
3. やらないことリスト:
   - やる: / やらない: / 後で決める:
4. ご近所さん: <関係部署・関係者と、それぞれに必要な関わり>
5. 夜も眠れない問題: <リスク上位3つと兆候の見張り方>
6. 期間の見極め: <ざっくり○ヶ月。根拠>
7. トレードオフスライダー: 品質[1-5] 期日[1-5] スコープ[1-5] 予算[1-5]
   ※同点禁止。優先度に順位をつける
```

→ 解説: [⑧章](08-team-and-culture.md)

## 9. 障害ふりかえり（ポストモーテム）

```markdown
# ポストモーテム: <障害の短い説明> (発生: ____ / 作成: ____)
※このドキュメントは blameless（非難なし）。人名は役割名で書く。

## 影響
- 影響範囲: <ユーザー数・機能・金額> / 継続時間: __分

## タイムライン（時刻・出来事・判断を事実だけ）
- HH:MM
- HH:MM

## 寄与要因（「根本原因は1つ」と考えない）
- 検知: なぜ__分かかったか
- 拡大: 何が被害を広げたか
- 復旧: 何が復旧を遅らせたか

## うまくいったこと
-

## 再発防止アクション（仕組みで塞ぐ。担当と期限必須）
| アクション | 種別[検知/予防/復旧] | 担当 | 期限 |
|---|---|---|---|
```

→ 解説: [⑪章](11-retrospectives.md)・[⑧章](08-team-and-culture.md)

## 10. スプリントレビュー 進行台本

```markdown
# スプリント__ レビュー (日付: ____ / 参加: チーム＋____)

1. スプリントゴールの再共有 (2分)
2. デモ（動くもので。スライド禁止） (20分)
   - 完了項目: / 未完了と理由（隠さない）:
3. 計測結果の共有 (10分)
   - 前回リリース分の指標: <アウトカムの数字>
4. フィードバック・対話 (20分)
   - 問い:「これを見て、次に何をすべきだと思いますか？」
5. 今後の見通し (5分)
   - バックログ上位: / 予測(確約ではない):

## 出たフィードバックと扱い
| 意見 | 扱い [バックログ追加/既存項目へ/やらない+理由] |
|---|---|
```

→ 解説: [②章](02-scrum.md)・[⑯章の会話例](16-sprint-simulation.md)
