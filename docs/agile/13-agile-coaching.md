# ⑬ アジャイルコーチング — チームと組織の変化を支援する技術

[← 総合インデックスに戻る](README.md)

スクラムマスターやアジャイルコーチの仕事は「スクラムのルールを教えること」だけではありません。**人と組織の行動が変わるのを支援する**という、それ自体が専門技術のある仕事です。この章では、SM・コーチ・リーダーが使う支援の技術を深掘りします。

## 1. 守破離 — 学習段階に合わせて支援を変える

日本の武道・芸道の学習モデル「守破離」は、アジャイルコミュニティで学習段階の共通言語として世界的に使われています。

| 段階 | チームの状態 | コーチの関わり方 |
|---|---|---|
| **守**（型を守る） | 初めてで判断基準がない | **ティーチング**が中心。型どおりにやらせる。「なぜこの型か」も添える |
| **破**（型を破る） | 型の意図を理解し、状況に合わせ調整できる | **メンタリング/コーチング**へ移行。調整の相談相手になり、実験を促す |
| **離**(型を離れる) | 原則を体得し、自分たちの型を作れる | **手を放す**。求められた時だけ関わる。依存を作らないことが成功 |

### よくある失敗

- **守の省略**: 初心者チームが最初から「うちに合わせてイベント削減」→ 型の意図を知らないままの自己流で崩壊（⑦章の形骸化に直結）
- **守への固執**: 成熟したチームにルール遵守を強要し続ける → コーチがボトルネック化し、チームの思考停止を招く
- 段階の見極めがコーチの技術。「型を変えたい」という提案が**意図の理解に基づくか**（破)、**面倒の回避か**（守の未達）を対話で見分ける

## 2. 4つの帽子 — 状況で立場を切り替える

アジャイルコーチは1日の中でも複数のスタンスを使い分けます。混同が事故のもとになります。

| 帽子 | 何をするか | 使う場面 |
|---|---|---|
| ティーチャー | 知識・型を教える。答えを渡す | 新しい概念の導入時（守） |
| メンター | 自分の経験から助言する。選択肢を渡す | 実務の壁に当たった時 |
| **コーチ** | 問いによって相手自身の答えを引き出す。**答えを渡さない** | 相手に答えがありそうな時・成長させたい時 |
| ファシリテーター | 中立の立場で場と対話の構造を作る | 会議・ワークショップ・対立の調停 |

### コーチングの基本技術（コーチの帽子）

- **開かれた問い**: 「テスト書きましたか？」(Yes/No)より「品質にどんな不安が残っていますか？」
- **スケーリングクエスチョン**: 「今のチームの状態は10点満点で何点？」「1点上げるとしたら何が変わっている？」— 抽象論を具体的な次の一歩に変換する
- **沈黙に耐える**: 問いのあと相手が考える時間を奪わない。沈黙を埋めたくなるのはコーチ側の不安
- **観察のフィードバック**: 評価でなく観察を返す。「デイリーが報告会でしたね（評価）」ではなく「今日のデイリーで、発言の8割がSMに向いていました（観察）。どう見えますか？」

## 3. 抵抗と変化のマネジメント

### 「抵抗」は情報である

変化への抵抗は敵ではなく、**満たされていない懸念のシグナル**です。よくある正体:

| 表面の言葉 | 背後にある懸念（例) | アプローチ |
|---|---|---|
| 「うちには合わない」 | 過去の失敗体験・失うものへの不安 | 何を失うと感じているかを聞く。小さく安全な実験として提案 |
| 「忙しくてそれどころではない」 | 現在の評価軸で損をする恐れ | 評価者を巻き込む。既存業務の何を減らすかを先に決める |
| 「前もやったが駄目だった」 | 学習性無力感 | 前回と何が違うかを明確に。過去の失敗要因を一緒に分析 |
| ベテランの沈黙・不参加 | 専門性が無価値になる不安 | その人の専門性が新しいやり方でこそ活きる役割を設計 |

### 変化を広げる実務原則

- **小さく始めて見せる**: 説得より実演。パイロットチームの成果（動くデモ・数字）が最強の説得材料（⑦章）
- **アーリーアダプターから**: 全員一斉でなく、乗り気な人・チームから。懐疑派は「証拠が出てから」で構わない
- **公式の権力者を巻き込む**: 評価制度・予算・人事に触る変化はコーチだけでは動かない。スポンサー（役員等）を確保し、定期的に進捗と障害を報告する
- **変化の理由を語り続ける**: 「何をするか」だけ伝わり「なぜか」が抜けると、形だけの実施（カーゴカルト）になる

## 4. システム思考 — 個人ではなく構造を見る

チームの問題行動の多くは、個人の資質ではなく**構造（インセンティブ・制約・情報の流れ）の産物**です。

- 例:「開発者がテストを書かない」→ 個人の怠慢と見る前に構造を疑う: 締切が個人コミット扱いになっていないか？ テストを書く時間が見積りに入っているか？ レビューでテストが要求されるか？ 書かなくても誰も困らない仕組みになっていないか？
- 介入は**行動に一番近いレバー**から: DoD にテストを含める（ルール）→ CI で強制する（仕組み）→ ペア作業で書き方を移転する（能力）
- 「犯人探し」をやめ「**この構造なら誰でもそうなる。構造のどこを変えるか**」を口癖にする。これは⑧章の心理的安全性、⑪章の blameless と同じ思想の実践

## 5. コーチ自身の成長とセルフマネジメント

- **自分がボトルネックにならない**: コーチの成功指標は「自分がいなくてもチームが回ること」。介入するたび「これはチーム自身にできたのでは？」と自問する
- **観察の記録**: イベントでの発言分布・意思決定の速さ・ボードの滞留などを定点観測し、感覚でなくデータで変化を語る
- **スーパービジョン/ペアコーチング**: コーチ自身も第三者の視点が必要。コーチ仲間との相互観察・ふりかえりの場を持つ
- **越えてはいけない線**: コーチングは心理療法ではない。メンタルヘルスの問題が疑われる場合は専門家・産業医へつなぐ

## 6. スクラムマスターからのキャリアパス

| 段階 | 範囲 | 主な仕事 |
|---|---|---|
| スクラムマスター | 1チーム | チームの立ち上げ・イベント支援・障害除去 |
| シニアSM / チームコーチ | 2〜3チーム | SMの育成・チーム間の調整・技術プラクティス導入支援 |
| アジャイルコーチ | 部門 | 管理職への働きかけ・組織設計・導入戦略（⑥⑦⑫章の実行主体） |
| エンタープライズコーチ | 全社 | 経営層との協働・評価/予算制度の変革・変革プログラム設計 |

上に行くほど、仕事の中心は「スクラムの知識」から「**組織開発・変化のマネジメント**」に移ります。技術プラクティス（⑤章）を教えられるコーチは希少で市場価値が高い、という実務的な傾向も覚えておく価値があります。
