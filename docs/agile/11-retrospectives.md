# ⑪ ふりかえりの技法 — 継続的改善のエンジンを回し続ける

[← 総合インデックスに戻る](README.md)

レトロスペクティブ（ふりかえり）は、アジャイルの「検査と適応」をプロセス自体に適用する装置であり、**チームが自力で良くなっていけるかどうかを決める最重要イベント**です。一方で最も形骸化しやすいイベントでもあります。この章では、運営の技法を実践レベルまで深掘りします。

## 1. ふりかえりの5ステップ構造

書籍『アジャイルレトロスペクティブズ』（Esther Derby / Diana Larsen）の定番構造。KPTだけを毎回やるより、この骨格に多様な手法を当てはめる方が長持ちします。

| ステップ | 目的 | 時間目安(90分の場合) | 手法例 |
|---|---|---|---|
| 1. 場を設定する | 全員の口を開かせ、目的を確認する | 5分 | チェックイン(一言)、今の気分を天気で |
| 2. データを収集する | 事実と感情を出し切る | 25分 | タイムライン、Mad/Sad/Glad |
| 3. アイデアを出す | 原因を掘り、改善案を発散する | 25分 | 5回のなぜ、フィッシュボーン |
| 4. 何をするか決める | 実行するTryを絞り込む | 20分 | ドット投票、SMARTなアクション化 |
| 5. 終了する | 決定を確認し、ふりかえり自体を評価 | 10分 | ROTI(この時間の価値を5段階で) |

**ポイント**: いきなり「問題は？」から始めない。ステップ1で全員が一度声を出す（最初の5分で発言しなかった人はその後も黙る傾向がある）、ステップ2で**事実を先に**共有する（事実の共有前に意見を戦わせると水掛け論になる）。

## 2. データ収集の手法カタログ

### タイムライン

スプリントの時系列を壁に描き、出来事を付箋で貼っていく。**記憶の偏り（直近の出来事だけで語る）を防ぐ**効果が大きい。感情の浮き沈みを折れ線で重ねる「感情曲線」を組み合わせると、数字に出ない疲弊・盛り上がりが見える。

### KPT（Keep / Problem / Try）

日本で最も普及。続ける・問題・試すの3枠。**マンネリ化したら**: Keepを「なぜうまくいったか」まで掘る、Problemを出す前にタイムラインで事実を出す、Tryに担当者と期限を付ける、の3点で締め直すと復活することが多い。

### Fun / Done / Learn

楽しかった・完了した・学んだの3円ベン図。ポジティブ起点なので、**疲弊しているチーム・非難合戦になりがちなチーム**の立て直しに向く。

### 4Ls / Mad-Sad-Glad / Starfish

- **4Ls**: Liked / Learned / Lacked / Longed for（好き・学び・足りない・望む）
- **Mad-Sad-Glad**: 感情ベースで出す。感情の裏に必ず事実があるので、掘る素材になる
- **Starfish**: Keep doing / Less of / More of / Stop doing / Start doing の5分類。「やめる」を明示的に扱えるのが強み

### 帆船（Sailboat）

絵で考える手法。帆船（チーム）・追い風（推進力）・錨（足かせ）・岩（先のリスク）・島（ゴール）を描いて付箋を貼る。**リスク（岩）を未来志向で扱える**のが特徴で、比喩のおかげで率直な指摘が和らぐ。

## 3. 原因分析 — 症状ではなく構造を直す

### 5回のなぜ（Five Whys)

> 「リリースが遅れた」→ なぜ? 「結合テストでバグが多発」→ なぜ? 「APIの仕様齟齬」→ なぜ? 「仕様変更が別チームに伝わっていない」→ なぜ? 「変更の通知経路が個人のDM頼み」→ **対策: 変更はADR＋チャンネル通知をルール化**

- 途中で「〇〇さんが忘れた」に到達したら**掘り方を誤っている**。人ではなく仕組みに向かって掘る（なぜ忘れられる構造だったのか？）
- 5回は目安。仕組みで対策できる深さに達したら止める

### 影響の輪で仕分ける

出てきた課題を「**チームで変えられること / 働きかければ変わること / 変えられないこと**」に仕分ける。変えられないこと（会社の制度など）への愚痴に時間を溶かさず、変えられることに集中する。働きかけ枠はSMが組織へエスカレーションする。

## 4. Tryを「実行される改善」にする

ふりかえり形骸化の最大要因は「**Tryが実行されない**」ことです。

- **絞る**: Tryは1〜2個まで。10個決めるのは0個と同じ
- **具体化する**: 「コミュニケーションを増やす」ではなく「毎朝のデイリー後に、詰まっているタスクのペア作業相手をその場で決める」
- **タスク化する**: Tryをスプリントバックログに載せ、ボードで管理する（ふりかえりの外に置くと消える）
- **効果を確認する**: 次回ふりかえりの冒頭は「前回のTryはどうだったか」から始める。効果がなければやめてよい（やめる判断も改善)
- **計測できるなら測る**: 「レビュー待ちが長い」→ レビュー着手までの時間を1スプリント測ってから対策を選ぶ

## 5. ファシリテーターの引き出し

- **書いてから話す**: 付箋やボードに各自書く時間（3〜5分の沈黙）を先に取る。声の大きい人への同調を防ぎ、全員の視点が出る
- **ラウンドロビン**: 1人ずつ順番に1枚ずつ出す。偏りを防ぐ
- **ドット投票**: 議論する課題を投票で絞る。全部は扱わない
- **ファシリテーターを輪番にする**: SM固定だと視点が固定化する。全員が回すとイベントへの当事者意識も上がる
- **手法を定期的に変える**: 同じ手法の連続はマンネリの温床。3〜4回ごとに変える、四半期に1回は長めの「スプリントを跨いだふりかえり」を入れる
- **安全度チェック**: 冒頭に「今日どこまで率直に話せそうか」を無記名で5段階投票。低ければ、その事実自体を扱う（⑧章の心理的安全性）

## 6. 状況別の処方箋

| 症状 | 処方箋 |
|---|---|
| 意見が出ない・沈黙が続く | 書く時間を先に取る。Fun/Done/Learnなどポジティブ起点に変える。1on1で背景を探る（安全性の問題かも） |
| 毎回同じProblemが出る | 「なぜ前回のTryで解決しなかったか」を主題にする。影響の輪で「変えられないこと」なら扱いを変える（エスカレーション） |
| 愚痴大会になる | 事実（タイムライン）から始める。感情は受け止めた上で「では何を変えるか」へ構造的に移す |
| 特定個人への非難が出る | ルールを事前合意（「人ではなく仕組みを話す」）。出たら即、仕組みの言葉に翻訳して板書する |
| 時間が長引く・疲れる | タイムボックスを画面に表示。扱う課題をドット投票で1つに絞る。そもそも90分を60分に |
| 盛り上がるが何も変わらない | Try のタスク化・担当・次回冒頭での確認をルーチンにする（本章4節） |

## 7. スプリントの外のふりかえり

- **プロジェクト/四半期ふりかえり**: 複数スプリントを俯瞰し、大きな構造（チーム編成・アーキテクチャ・リリース戦略）を扱う。半日〜1日かける
- **ポストモーテム（障害ふりかえり）**: 障害対応後に非難なし（blameless）で行う。タイムライン再構成 → 寄与要因の列挙 → 再発防止アクション。「根本原因は1つ」と考えず、複数の寄与要因を仕組みで塞ぐ
- **プレモーテム**: 開始前に「このプロジェクトは失敗した。何が起きたか？」を想像で書き出す。リスクを事前に炙り出す逆向きのふりかえり
