# ⑧ チームと文化 — 自己組織化・心理的安全性・チームの成長

[← 総合インデックスに戻る](README.md)

アジャイルの手法・イベント・メトリクスは「見える部分」ですが、それが機能するかどうかは**チームの状態と組織文化**という「見えない部分」でほぼ決まります。この章では、その土台を深掘りします。

## 1. 自己組織化（自己管理）チームとは

アジャイルの原則11は「最良のアーキテクチャ・要求・設計は**自己組織化したチーム**から生み出される」と述べています。

### 何が「自己」で決められるのか

| 決めること | 従来型 | 自己組織化チーム |
|---|---|---|
| 何を作るか（What） | 上位が決める | PO がバックログで決める（チームと対話しながら） |
| どう作るか（How） | リーダー・アーキテクトが割り振る | **チームが決める**（設計・分担・進め方） |
| 誰がやるか（Who） | マネージャーがアサイン | **チーム内で自律的に引き受ける**（プル型） |
| いつまでか（When） | 上から降ってくる | 見積りとベロシティに基づきチームが予測を示す |

- 自己組織化＝放任ではない。**ゴール（スプリントゴール・プロダクトゴール）と制約（DoD・品質基準）は明確に与えられ**、その中での手段が自由
- マネージャーの役割は指示から**環境整備**（障害除去・権限委譲・成長支援）へ変わる

### タスクは「割り振る」のではなく「取る」

デイリースクラムでリーダーがタスクを配るのは自己組織化ではありません。ボードから**自分でプルする**（サインアップする）のが基本形です。プル型は次の効果を生みます。

- 手が空いたら詰まっている作業を**助けに行く**動きが自然に生まれる
- 「自分のタスクだけ終わればよい」から「**スプリントゴールをチームで達成する**」への意識転換
- 属人化の緩和（得意な人が常に同じ領域を取り続けないよう、あえて不得意領域を取る文化を作る）

## 2. 心理的安全性

Google の Project Aristotle（2015年公表）は、成果の高いチームの最重要因子は**心理的安全性**（対人リスクを取っても罰されないという共有信念、Amy Edmondson の概念）であると報告しました。アジャイルとの関係は直接的です。

| アジャイルの活動 | 心理的安全性が低いと起きること |
|---|---|
| デイリースクラム | 「順調です」しか言わなくなる。ブロッカーが隠される |
| プランニングポーカー | 最大・最小を出した人が黙る。見積りのズレ（＝認識のズレ）が発見できない |
| レトロスペクティブ | 当たり障りのない Keep しか出ない。本当の問題が扱われない |
| スプリントレビュー | 悪いニュース（未完了・仮説の失敗）が報告されず、検査と適応が壊れる |

### 高める実践

- **失敗の扱いを変える**: 障害・失敗のふりかえりを**非難なし（blameless）ポストモーテム**で行う。「誰が悪いか」ではなく「どの仕組みが失敗を許したか」を問う
- **リーダーが先に弱さを見せる**: 「自分はここが分からない」「この判断は間違いだった」を上位者が先に言う
- **発言の均等化**: 振り返りで付箋に書いてから話す（声の大きい人に引きずられない）、1人ずつ順番に話すラウンドロビン
- **「質問を歓迎する」を仕組みにする**: モブプログラミング・ペア作業は「質問して当然」の場として機能する

心理的安全性は「ぬるさ」ではありません。**基準が高く、かつ率直に話せる**状態が学習するチームです（Edmondson の「学習ゾーン」）。

## 3. チームの成長段階 — タックマンモデル

チームは結成直後から機能するわけではなく、段階を経ます（Bruce Tuckman, 1965）。

| 段階 | 状態 | この時期に必要なこと |
|---|---|---|
| 形成期（Forming） | 様子見。表面的に和やか、本音は出ない | ゴール・役割・ワーキングアグリーメントの明確化 |
| 混乱期（Storming） | 意見の衝突・主導権争い・手法への不満が噴出 | **衝突を避けずに扱う**。ファシリテーションと対話 |
| 統一期（Norming） | チームなりのやり方・規範が確立する | 規範の明文化・振り返りでの微調整 |
| 機能期（Performing) | 自律的に高い成果。役割が流動的に補完し合う | 権限委譲を進め、邪魔をしない |

### 実務への示唆

- **混乱期は健全な通過点**。ここで「アジャイルは失敗だ」と手法のせいにして解散するのが最悪手
- **メンバーの入れ替えごとに段階は逆戻りする**。「プロジェクトに人を集める」より「**安定したチームに仕事を流す**」が推奨されるのはこのため
- 結成時に**ワーキングアグリーメント**（コアタイム・レビューのSLA・割り込みの扱い・Slackの返信期待値など）を明文化すると混乱期が軽くなる

## 4. インセプションデッキ — チームの立ち上げ道具

『アジャイルサムライ』で紹介された、プロジェクト開始時に**チームと関係者の認識を揃える10の問い**。数日かけて全員で作ります。

1. **我々はなぜここにいるのか**（プロジェクトの根本理由）
2. **エレベーターピッチ**（30秒でプロダクトを説明する定型文）
3. **パッケージデザイン**（顧客に見せる架空の商品パッケージを作る）
4. **やらないことリスト**（スコープ外を明文化 — しばしば最も価値がある）
5. **「ご近所さん」を探せ**（関わる部署・関係者の一覧）
6. **解決案を描く**（技術的なアーキテクチャ概要）
7. **夜も眠れなくなる問題は何か**（リスクの共有）
8. **期間を見極める**（ざっくりのサイズ感）
9. **何を諦めるのか**（スコープ・予算・期日・品質のトレードオフスライダー）
10. **何がどれだけ必要か**（体制・費用）

特に「やらないことリスト」と「トレードオフスライダー」（期日と品質どちらを優先するか等を**事前に**合意する）は、後々の意思決定の拠り所として効果が大きい道具です。

## 5. ファシリテーションの基本

スクラムイベントの質はファシリテーションで大きく変わります。SM に限らずチーム全員が持つべきスキルです。

- **問いで進める**: 「進捗どうですか」より「スプリントゴール達成のために今日一番重要なことは？」
- **見える化しながら話す**: ホワイトボード・付箋・ボードを囲む。空中戦（口頭だけの議論）は堂々巡りになる
- **発散と収束を分ける**: アイデア出し（批判禁止・量重視）と絞り込み（ドット投票など）を混ぜない
- **タイムボックスを守る**: 時間で区切る勇気。終わらなければ「続きをいつやるか」だけ決めて終える
- **沈黙を恐れない**: 問いのあと数秒待つ。書く時間（1人で考える時間）を先に取ると質が上がる

## 6. 分散・リモートチームでのアジャイル

- **ボードとドキュメントをシングルソースにする**: 口頭で決まったことも必ずボード・チケットに反映（「その場にいた人しか知らない」を作らない）
- **同期と非同期を設計する**: デイリーは同期を維持しつつ、リファインメントの一部はドキュメントの事前コメントで非同期化するなど
- **雑談の場を意図的に作る**: 対面で自然発生していた偶発的な会話（設計の相談・違和感の共有）が消えるため、モブ作業・バーチャルコーヒーなどで補う
- タイムゾーンが分かれる場合、**重なる時間帯にイベントを固定**し、それ以外は非同期前提のワークフローにする
