# ⑥ 大規模アジャイル — 複数チーム・組織への展開

[← 総合インデックスに戻る](README.md)

スクラムは10人以下の1チームを前提に設計されています。プロダクトが大きくなり複数チームで開発する場合、**チーム間の依存関係・統合・優先順位の整合**という新しい問題が生まれます。この章では、その解決アプローチである主要なスケーリングフレームワークを解説します。

## 1. スケーリングの前に — 大前提

大規模フレームワークの導入より先に確認すべきことがあります。

1. **本当に大人数が必要か？** — 人数を増やすほどコミュニケーションコストは組み合わせ的に増える（n人の連絡経路は n(n-1)/2）。少数精鋭で足りるならそれが最速
2. **1チームのアジャイルが機能しているか？** — 1チームで回らないものを拡大しても、問題が拡大するだけ
3. **アーキテクチャが分業に耐えるか？** — チーム間依存の多くはコードの依存。疎結合な構造（後述のコンウェイの法則）が先
4. **技術的土台（⑤章）があるか？** — 複数チームの成果を統合するには、CI/CDと自動テストがほぼ必須

### コンウェイの法則と「逆コンウェイ戦略」

> システムの構造は、それを作る組織のコミュニケーション構造を写し取る（コンウェイの法則）

逆に言えば、**望むアーキテクチャに合わせてチームを編成する**（逆コンウェイ戦略）のが大規模化の定石です。機能横断で自己完結した**フィーチャーチーム**（1チームでUI〜DBまで縦に持つ）が理想とされ、レイヤー別チーム（UIチーム・APIチーム・DBチーム）は1つの機能に全チームの調整が必要になるため避けるべきとされます。

## 2. 主要フレームワークの比較

| | Scrum of Scrums | **LeSS** | **SAFe** | **Scrum@Scale** | Spotifyモデル |
|---|---|---|---|---|---|
| 規模感 | 2〜5チーム | 2〜8チーム（LeSS Huge で拡張） | 数十〜数百チーム | 可変（フラクタル） | （参考事例） |
| 思想 | 最小限の連絡会 | スクラムを崩さず最小限の追加 | 組織全体の包括的な体系 | スクラムの構造を再帰的に拡大 | 文化とチーム自治の事例 |
| 追加ロール | 特になし | 少ない（POは全体で1人） | 多い（RTE、各種マネージャ層） | SoSマスター等 | 章・ギルド等の呼称 |
| 導入コスト | 低 | 中（組織変更が必要） | 高（研修・認定体系が充実） | 中 | — |

### Scrum of Scrums（SoS）

最も軽量な方法。各チームの代表が集まる**チーム横断のデイリー/定例**を行い、依存とブロッカーを同期します。2〜5チーム程度ならこれで十分なことが多いです。

### LeSS（Large-Scale Scrum）

「スクラムをそのまま大きくする」思想。**プロダクトバックログもPOも全体で1つ**のまま、複数チームが同じスプリントを並走します。

- スプリントプランニングの前半は全チーム合同、後半はチーム別
- スプリントレビューは全チーム合同で1つ
- 「役割・成果物・プロセスを**増やす**のではなく、組織をシンプルに**減らす**」ことを重視し、コンポーネントチームの廃止・フィーチャーチーム化などの組織変更を伴う

### SAFe（Scaled Agile Framework）

エンタープライズ向けの最も包括的（かつ重量級）なフレームワーク。

- 50〜125人を**ART（Agile Release Train）**という単位に束ね、**PI（Program Increment、約8〜12週）**という大きなサイクルで計画する
- PIの冒頭に全員参加の**PIプランニング**（2日間）を行い、チーム間の依存を一気に可視化・調整する
- ポートフォリオ管理・リーン予算など経営レイヤーまで体系化されており、大企業の既存階層と接続しやすい
- 批判もある: 階層とロールが多く「アジャイルの皮を被ったウォーターフォール」になりやすい、という指摘は導入時に必ず意識すべき

### Scrum@Scale

スクラム共同考案者Jeff Sutherlandによる。**スクラムチームの構造をフラクタル（再帰的）に積み上げる**発想で、「SMサイクル（どう作るか）」と「POサイクル（何を作るか）」の2系統で組織をスケールさせます。規定が少なく柔軟な分、自分たちで設計する部分が多くなります。

### Spotifyモデル（注意付き）

Squad（チーム）/ Tribe / Chapter / Guild という用語で知られる事例。ただしSpotify自身が「これはある時点のスナップショットで、フレームワークではない」と述べており、**そのままコピーして導入するものではない**とされます。「チームの自治と緩やかな横のつながり」という考え方だけ参考にするのが適切です。

## 3. スケーリング共通の実務課題

### 依存関係の管理

- 依存はボードで**可視化**し、PIプランニングやSoSで事前に調整する
- 恒常的に同じチーム間で依存が発生するなら、**チーム編成かアーキテクチャを直す**シグナル（依存の調整をうまくやるより、依存自体を減らす）

### 統合(インテグレーション)

- 全チームが**同一のメインブランチに頻繁に統合**し、常に全体が動く状態を保つ（⑤章のCIを全体に拡大）
- 「チームごとに完成 → 最後に結合」は大規模で最も高くつく失敗パターン

### 優先順位の一元化

- チームごとに別々の発注元から仕事を受けると、全体最適が崩れる
- **プロダクト全体のバックログ（または明確なポートフォリオ優先順位）を1本化**するのが原則（LeSSはこれを最重視）

### コミュニケーション設計

- 全員会議を増やすのではなく、**チームAPI**（各チームが何をどう受け付けるかの明文化）や社内プラットフォームで「調整不要で進める」仕組みを作る
- 参考: 書籍『チームトポロジー』のストリームアラインドチーム／プラットフォームチームという整理

## 4. どれを選ぶか（実務指針）

| 状況 | 推奨 |
|---|---|
| 2〜3チーム | フレームワーク導入前にSoS＋バックログ1本化で様子を見る |
| 〜8チーム・組織を変える覚悟あり | LeSS |
| 大企業・多階層・数百人規模、経営層の巻き込みが必要 | SAFe（形骸化への警戒とセットで） |
| スクラム文化が既に強く、柔軟に設計したい | Scrum@Scale |

共通する結論: **フレームワークは出発点であり、目的は「チーム間の依存と待ちを減らし、価値の流れを速くすること」**。導入自体をゴールにしないことが最大の成功要因です。
