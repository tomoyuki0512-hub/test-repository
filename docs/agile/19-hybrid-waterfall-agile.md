# ⑲ ハイブリッド開発 — 新規構築はウォーターフォール、保守・改善はアジャイルで回すスマホアプリ

[← 総合インデックスに戻る](README.md)

「新規開発（0→1）はウォーターフォールで計画的に作り切り、リリース後の保守・改善（1→10）はアジャイルで回したい」— 1つのスマホアプリをこの組み合わせで開発する場合の設計を、フェーズごとにまとめます。この組み合わせは実務で広く使われており、**切り替えの設計を誤らなければ**十分に機能します。

## 1. 全体像 — 3つの期間で考える

```
フェーズ1: 新規構築期          フェーズ2: 移行期       フェーズ3: 保守・改善期
（ウォーターフォール主体）      （切り替えの準備）       （アジャイル主体）
┌─────────────────────┐ ┌─────────────┐ ┌──────────────────────┐
│ 企画 → 要件定義 → 設計   │ │ βテスト・審査  │ │ 計測 → 改善バックログ →    │
│ → 実装 → テスト         │ │ 体制の組み替え  │ │ スプリント/カンバン →       │
│ （中身は反復を推奨 §3）   │ │ 運用準備      │ │ 定期リリース → 計測…（反復） │
└─────────────────────┘ └─────────────┘ └──────────────────────┘
        〜リリースまで          リリース前後1ヶ月        リリース後ずっと
```

最重要ポイントは2つです。

1. **フェーズ1を「純粋な」ウォーターフォールにしない** — 外側の節目（マイルストーン）はWF、内側の作り方はイテレーティブにする（§3）
2. **フェーズ2（移行期）を明示的に設計する** — ここを設計しないと「作った人が去り、直せない人が残る」典型的な失敗になる（§5）

## 2. なぜこの組み合わせが合理的か（そして何が危険か）

### 合理的な理由

| 特性 | 新規構築期 | 保守・改善期 |
|---|---|---|
| 要求の性質 | ストア公開に必要な最低限の機能セットは事前に決めやすい | ユーザーの実際の行動・レビュー・クラッシュから次々発生 |
| 締切 | 公開日・キャンペーン日など**日付固定**が多い | 固定締切は少なく、継続的な流れ |
| 予算 | 初期投資として一括で確保しやすい（WF的な稟議と相性が良い） | 運用費として月額・体制維持で確保 |
| 検証手段 | リリース前はβテストのみ（フィードバックが限定的） | 本番の計測・A/Bテストが使える（アジャイルの強みが最大化） |

リリース前は「フィードバックを得る手段」が乏しいためアジャイルの強みが出にくく、リリース後は要求が流動化するためWFでは追いつかない — つまり**各フェーズの不確実性の性質に手法を合わせる**のがこのハイブリッドの本質です（[①章](01-agile-basics.md)の「不確実性で選ぶ」の応用）。

### 危険な点（先に知っておく）

- **初回リリースの一発勝負リスク**: WF構築の弱点はそのまま残る。「作ったが使われない」をβテストとMVP思考で緩和する必要がある（§4）
- **移行期の断絶**: 構築チーム解散→別会社・別チームで保守、という分断は技術情報とテスト資産の喪失を招く
- **「Water-Scrum-Fall」の悪い形**: 中身だけスクラムの真似をして、要件も予算もリリースも硬直したままの「名ばかりハイブリッド」になりやすい。**どこをWFで固定し、どこをアジャイルに委ねるかを明文化する**のが対策

## 3. フェーズ1: 新規構築期の設計

### 決め切るもの／決めないもの

WFの「最初に決める」を全部に適用しないのがコツです。

| 最初に決め切る（WF的に固定） | 走りながら決める（余地を残す） |
|---|---|
| リリース日・予算・体制 | 画面の細部・文言・微調整 |
| 初回リリースのスコープ（機能一覧をMoSCoWで: [④章](04-requirements-estimation.md)） | Could（あれば良い）機能の採否 |
| アーキテクチャ・技術選定（ADRに記録: [⑱章](18-templates.md)） | 内部実装の詳細 |
| 非機能要件（対応OS・オフライン動作・セキュリティ・計測基盤） | 改善の優先順位（フェーズ3のバックログへ） |

**スコープ・期日・品質のうち固定するのは2つまで**の原則（④章）はここでも生きます。期日と品質を固定し、**スコープ（Could機能）を調整弁**にするのがスマホアプリの定石です。

### 内側はイテレーティブに作る

工程を「全画面の設計完了→全画面の実装→最後に結合」と進めるのではなく、**設計・実装・テストを機能単位で縦に回します**（外から見ればWFのマイルストーン、中は2週間の反復）。

- マイルストーン例: M1 認証と基盤（動くアプリの骨格）→ M2 コア機能一気通貫 → M3 課金・周辺機能 → M4 β版 → リリース
- 各マイルストーンで**実機で動くアプリをステークホルダーにデモ**する。ここが最大の保険 — 紙の進捗率でなく動くもので判断する（[①章](01-agile-basics.md)原則7）
- **CI/CDと自動テストはフェーズ1の最初に作る**（[⑤章](05-engineering-practices.md)）。「保守フェーズになったら整備する」は手遅れ。フェーズ3の改善速度はここで決まる

### リリース前の検証を最大化する

- 社内配布（TestFlight / Firebase App Distribution 等）で**M2以降は常に触れる状態**を保つ
- クローズドβ→オープンβと段階を踏み、**本物のユーザーの操作データとクラッシュログ**を集めてから公開する
- ストア審査は初回が最も読めない。**審査提出をリリース日の2〜3週間前**に置き、リジェクト対応の予備日を計画に入れる

## 4. フェーズ2: 移行期の設計（最重要）

リリース前後の約1ヶ月。ここで次の5つを済ませます。

1. **体制の連続性を確保する** — 構築メンバーの少なくとも2〜3名（アーキテクチャを知る人・ドメインを知る人）を保守チームの中核として残す。全員入れ替えは禁じ手。受託の場合は契約段階で保守フェーズの体制を合意しておく（[⑫章](12-agile-in-japan.md)）
2. **資産を引き継ぐ** — コードだけでなく、自動テスト・CI/CD・ADR・運用手順（Runbook）・既知の技術的負債リストをセットで。負債リストはそのまま**改善バックログの初期在庫**になる
3. **計測を立ち上げる** — フェーズ3の燃料はデータ。クラッシュ監視（Crashlytics等）・行動分析・ストアレビュー監視・Four Keys（[⑦章](07-metrics-adoption.md)）をリリース時点で動く状態に
4. **リリースプロセスを定義する** — 定期リリースの周期・hotfixの手順・ロールバック（スマホは即時ロールバック不可: §5参照）を文書化
5. **保守フェーズの働き方を立ち上げる** — 最初のバックログ（負債＋β期の積み残し＋初期レビュー対応）を作り、ふりかえり（[⑪章](11-retrospectives.md)）を第1週から開始する

## 5. フェーズ3: 保守・改善期の設計

### 手法は「スクラムバン」が第一候補

保守・改善期の仕事は2種類が混ざります。

- **改善（計画的）**: 新機能・UX改善・A/Bテスト → スプリントで計画的に
- **保守（割り込み）**: クラッシュ対応・OSアップデート追従・ストア規約対応・問い合わせ起因の修正 → カンバンで流す

このため**スクラム＋カンバンのハイブリッド（スクラムバン: [③章](03-kanban-xp.md)）**が最も収まりが良い構成です。

- スプリントの計画時に**割り込み用バッファ（容量の20〜30%）**を確保する。実績を計測して比率を調整
- 障害・セキュリティのみ即時割り込み可とし、それ以外はバックログ経由 — 「本当の緊急」の定義を[⑱章のワーキングアグリーメント](18-templates.md)に明文化
- 改善テーマは仮説駆動で: 計測→仮説→変更→計測のループ（[⑨章](09-product-discovery.md)）。ストアレビューとクラッシュ率は最優先の入力

### スマホアプリ特有のリリース設計

Webと違い「いつでも即リリース・即ロールバック」ができないため、リリースの規律が重要です。

| 論点 | 推奨プラクティス |
|---|---|
| リリース周期 | **リリーストレイン方式**（例: 隔週で定期便を出す）。乗り遅れた機能は次の便へ。締切前の駆け込みマージを防ぐ |
| 審査リスク | 審査落ちを前提に、定期便の提出は公開予定日の数日前に |
| ロールバック不可 | **フィーチャーフラグ**（[⑤章](05-engineering-practices.md)）で機能をリモートON/OFF可能にしておく。問題があればアプリ更新なしで無効化 |
| 段階公開 | ストアの段階的ロールアウト（1%→10%→100%）でクラッシュ率を見ながら展開 |
| 旧バージョン | 強制アップデートの仕組みを**フェーズ1で**実装しておく（後から入れると行き渡らない）。API後方互換の維持方針を決める |
| OS追従 | 毎年のOSメジャーアップデート対応を**恒例行事としてバックログに事前確保**（毎年6〜9月は容量を空けておく等） |

### 体制とガバナンスの接続

- 経営・予算側にはWF的な報告が引き続き必要なことが多い。**四半期のアウトカム目標（OKR: ⑨章）＋Four Keys**で報告し、機能一覧の約束はしない — 「ロードマップは目標ベース」（[④章](04-requirements-estimation.md)）
- 保守・改善の契約は**準委任（ラボ型）**が基本形（[⑫章](12-agile-in-japan.md)）。「月あたりの改善チーム一式」として確保し、中身の優先順位はPOが動かす

## 6. よくある失敗と対策

| 失敗パターン | 何が起きるか | 対策 |
|---|---|---|
| フェーズ1で全部作り込む | Could機能まで盛り込み、リリース延期の連鎖 | MoSCoWで初回スコープを絞る。「リリース後に学んでから作る」が合言葉 |
| 移行期の設計なし | 構築チーム解散→保守チームがコードを読めず改善が止まる | §4の5項目を計画に組み込む。中核メンバーの継続を契約・人事で担保 |
| テスト・CI を保守フェーズに先送り | 保守期の変更が怖く、改善速度が出ない（[⑤章](05-engineering-practices.md)の片輪走行） | フェーズ1の最初のマイルストーンに含める |
| 保守もWFのまま | 四半期ごとの大型リリースになり、不具合対応が遅く、学習ループが回らない | リリーストレイン＋スクラムバンへ。まず隔週便から |
| 改善バックログがない | 保守＝言われたバグを直すだけの受け身運用に | 移行期に負債・計測・レビューから初期バックログを作る |
| 名ばかりハイブリッド | 「スプリント」と呼ぶだけで要件・予算・リリース全部が硬直 | どこをWFで固定しどこを委ねるか（§3の表）を関係者と明文化 |

## 7. 立ち上げチェックリスト

```markdown
## フェーズ1（構築）開始前
- [ ] 初回スコープをMoSCoWで分類し、Won't/Couldを明文化した
- [ ] 固定するもの（期日・品質）と調整弁（スコープ）を関係者と合意した
- [ ] マイルストーンごとに「実機デモ」を置いた
- [ ] CI/CD・自動テスト・計測基盤・強制アップデート機構を初期マイルストーンに入れた
- [ ] ストア審査の予備期間を計画に入れた

## フェーズ2（移行）
- [ ] 構築メンバーの継続者（2名以上）が決まっている
- [ ] テスト・ADR・Runbook・負債リストの引き継ぎが完了した
- [ ] クラッシュ監視・行動分析・レビュー監視が稼働している
- [ ] リリーストレインの周期とhotfix手順を文書化した
- [ ] 初期バックログとふりかえりの定例が立ち上がった

## フェーズ3（保守・改善）
- [ ] 割り込みバッファの比率を計測して調整している
- [ ] フィーチャーフラグと段階的ロールアウトを使えている
- [ ] 改善は仮説→計測のループで回っている（作りっぱなしがない）
- [ ] OS追従・規約対応の恒例行事がバックログに確保されている
```
