# ⑨ プロダクトディスカバリー — 「正しく作る」の前に「正しいものを見つける」

[← 総合インデックスに戻る](README.md)

スクラムを完璧に回しても、**バックログに入っているものが間違っていれば、間違ったものが高速に出来上がるだけ**です。この章は「何を作るべきか」を見つける活動＝プロダクトディスカバリーを深掘りします。

## 1. デュアルトラックアジャイル

現代のプロダクト開発では、2つのトラックを**並行して**回すモデルが主流です。

```
ディスカバリートラック（何を作るべきか検証する）
  アイデア → 仮説 → 調査・プロトタイプ・実験 → 検証済みの項目
                                        │
                                        ▼ 検証済みのものだけが流れ込む
デリバリートラック（検証済みのものを高品質に作る）
  バックログ → スプリント → リリース → 計測 ──→ 学びがディスカバリーに還流
```

- ディスカバリーは「フェーズ」ではなく**常時走る活動**。デリバリーの2〜3スプリント先を探索するイメージ
- 担当を分業（企画チーム vs 開発チーム）にすると受発注関係に逆戻りする。**同じチームの活動の2側面**として、開発者もディスカバリーに参加する（インタビュー同席・プロトタイプ作成など）

## 2. 仮説駆動開発

機能を「要望」ではなく**検証可能な仮説**として扱います。

> 私たちは【あるユーザー】が【ある課題】を持っていると信じている。
> 【この機能・変更】を提供すれば、【この計測可能な変化】が起きるはずだ。
> それは【この指標がこの値になる】ことで確認できる。

例:

> 新規登録ユーザーの多くが初回設定の途中で離脱していると信じている。設定をスキップ可能にして後から促す方式にすれば、登録完了率が現在の55%から70%以上に上がるはずだ。

### アウトプットとアウトカム

| | 定義 | 例 |
|---|---|---|
| アウトプット | 作ったもの・出荷したもの | 「検索機能をリリースした」 |
| **アウトカム** | ユーザー・事業に起きた変化 | 「目的の商品到達時間が半減し、購入率が8%上がった」 |
| インパクト | 長期的な事業成果 | 「年間売上が◯億円増えた」 |

アジャイルの原則7「動くソフトウェアこそ進捗の尺度」の先には「**アウトカムこそ成果の尺度**」があります。ロードマップ・スプリントゴール・レビューを、アウトプット（機能一覧）ではなく**アウトカム（変化させたい指標）**で語れるかがチームの成熟度の分かれ目です。

## 3. 検証の道具箱

コストの低い順に並べます。**高コストな検証（実装）に進む前に、低コストな検証で仮説を絞る**のが原則です。

| 手法 | コスト | 何が分かるか |
|---|---|---|
| ユーザーインタビュー | 低 | 課題の実在・深さ・現在の代替手段（※「欲しいですか？」と聞かない — 過去の行動を聞く） |
| ペーパープロトタイプ / Figmaモック | 低 | UIの理解可能性・フローの妥当性 |
| ランディングページテスト | 低〜中 | 需要の強さ（登録・クリックという行動で測る） |
| **Wizard of Oz** | 中 | 裏側を人力で偽装して機能価値を検証（例: 「AI推薦」を人が手動で返す） |
| コンシェルジュMVP | 中 | サービス全体を人力で提供し、価値提案そのものを検証 |
| **A/Bテスト** | 中〜高 | 実装済み機能の効果を因果として計測（十分なトラフィックが必要） |
| 段階的ロールアウト＋計測 | 中 | 本実装の効果と副作用を小さく確認 |

### ユーザーストーリーマッピング

Jeff Patton 考案。バックログを「一次元のリスト」ではなく**二次元の地図**にする技法です。

```
（横軸＝ユーザーの行動の流れ / 時間軸）
 商品を探す → 比較する → 購入する → 受け取る → 困ったら問い合わせる
 ─────────────────────────────────────────────
（縦軸＝優先度）
 キーワード検索   価格で並べ替え   カード決済      配送状況表示     FAQ        ← リリース1（最小の一気通貫）
 カテゴリ絞り込み  レビュー比較     コンビニ払い    再配達依頼      チャット    ← リリース2
 検索候補表示     お気に入り比較    ポイント払い    置き配指定      電話       ← リリース3
```

- **横に一列すべて薄く通す**のがリリース1（=歩けるスケルトン）。「検索だけ完璧」より「全行程が最低限動く」が先
- 一覧では見えない**抜け漏れ（行動の流れの欠落）**が見える
- POとチームとステークホルダーが**同じ地図を見て**リリース範囲を交渉できる

## 4. OKRとアジャイルの接続

**OKR**（Objectives and Key Results）は、アウトカムベースの目標設定手法で、アジャイルのロードマップと相性が良い組み合わせです。

- **O（目標）**: 定性的で野心的な方向（例: 「新規ユーザーが迷わず価値に到達できる」）
- **KR（主要な結果）**: Oの達成を示す計測可能な指標（例: 「初回設定完了率 55%→80%」「登録翌週の継続率 20%→35%」）

接続のパターン: 四半期OKR（アウトカム）→ それを動かす仮説群（ディスカバリー）→ 検証済み仮説がバックログへ（デリバリー)→ スプリントレビューでKRの動きを確認、という流れ。**「今スプリントの仕事はどのKRを動かすためか」を全員が言える**状態が理想です。

注意: KRを個人評価に直結させると、達成確実な低い目標を置くようになりOKRが壊れます（⑦章のグッドハートの法則と同じ構図)。

## 5. POを支える実務ツール

### プロダクトビジョンとリーンキャンバス

- **ビジョン**: 「誰のどんな世界を実現するか」の1〜2文。バックログの並び順に迷った時の最終判断基準
- **リーンキャンバス**: 課題・顧客セグメント・独自の価値提案・解決策・指標・収益/コスト構造を1枚で整理。事業仮説の全体像をチームと共有する道具

### 優先順位づけの深掘り — RICEスコア

④章の技法に加え、ディスカバリー文脈では **RICE** がよく使われます。

> スコア = （**R**each 到達ユーザー数 × **I**mpact 影響度 × **C**onfidence 確信度）÷ **E**ffort 工数

Confidence（その見立てをどれだけ確信できるか）が入っているのが特徴で、**確信が低い項目は「作る」前に「検証する」へ**回す判断につながります。

### 捨てる勇気 — サンクコストとの戦い

- 検証で否定された仮説・使われない機能は**捨てる（作らない・消す）**。ここまでかけた調査コストは戻らない（サンクコスト）
- 機能は増えるほど保守コスト・UI複雑性という**負債**になる。「追加」だけでなく「削除」もバックログ項目にする
- レビューで「何を学び、何をやめるか」を必ず扱うと、捨てる文化が根付く

## 6. アンチパターン（ディスカバリー編）

| アンチパターン | 症状 | 処方箋 |
|---|---|---|
| フィーチャーファクトリー | 機能を出し続けるが誰も効果を測らない | 機能ごとに仮説と成功指標を事前定義。レビューで計測結果を扱う |
| 御用聞きPO | 営業・役員の要望を右から左にバックログへ | 要望の背後の課題を聞く（5回のなぜ）。ビジョンとRICEで優先度を説明 |
| 検証ごっこ | インタビューはするが結論が最初から決まっている | 仮説を「棄却条件」込みで事前に書く（どうなったら諦めるか） |
| ビッグバンMVP | 「最小」のはずのMVPが半年がかりの全部入り | ストーリーマッピングで一気通貫の最薄スライスを切り直す |
| 分業ディスカバリー | 企画が仕様書を作り開発に投げる（受発注の復活） | 開発者をインタビュー・実験に巻き込む。同じ地図（マップ）を見る |
