# ⑮ デザイナーとの協働 — デザインをスプリントに溶かし込む

[← 総合インデックスに戻る](README.md)

「デザインが終わってから開発が始まる」体制のままスクラムを導入すると、デザインが**チーム内のミニウォーターフォール**になります。この章では、デザイナー（UI/UX）とエンジニアが同じリズムで働くための実務パターンを扱います。

## 1. ありがちな断絶とその症状

| 体制 | 症状 |
|---|---|
| デザインチームが別部門で「納品」する | 完成カンプ至上主義。実装で発覚した矛盾の手戻りが大きい。デザイナーにフィードバックが還らない |
| デザイナーが複数チーム掛け持ちで薄く関与 | レビュー待ちがボトルネック化。「デザイン確認待ち」レーンにカードが滞留 |
| デザインなしでエンジニアが作り、後から「化粧」 | UXが構造レベルで壊れており、表面の修正では直らない |

**目指す形**: デザイナーが**チームの一員として**スプリントに参加し、「納品物のやり取り」ではなく「**共同作業**」になっている状態。⑨章のデュアルトラック（ディスカバリー/デリバリー）は、まさにデザイナーの主戦場です。

## 2. リードタイムをずらす — 「1スプリント先行」モデル

デザインと実装を同一ストーリーで完全同時進行するのは難しいため、実務では**デザイン活動が半〜1スプリント先行**する形が定番です。

```
スプリント N   : ストーリーAのUI設計・ユーザテスト ── ストーリーBを実装
スプリント N+1 : ストーリーAを実装（デザイナーは実装中の質問に即応）── ストーリーCのUI設計
```

- 先行しすぎ（3スプリント以上先のデザイン在庫）は**作り置きのムダ**になる。実装から得た学びが反映されない在庫は腐る
- デザイナーの仕事はデザイン先行分だけではない。**実装スプリント中の協働**（後述）と**リリース後の計測・学習**（⑨章）にも時間を割り当てる
- リファインメント（④章）にデザイナーが参加し、ストーリーの受け入れ条件にUI観点を入れるのが先行活動の起点

## 3. 「完成カンプ」から「会話と基準」へ

### 忠実度を段階的に上げる

最初から高忠実度カンプを作り込まず、**議論の段階に合った忠実度**を使い分けます。

| 段階 | 道具 | 決めること |
|---|---|---|
| 構造 | 紙のスケッチ・ホワイトボード | 画面の目的・情報の優先順位・フロー |
| 挙動 | ワイヤーフレーム・クリック可能プロトタイプ | 遷移・状態変化・エラー時の振る舞い |
| 外観 | 高忠実度モック（必要な画面だけ） | 視覚的な仕上げ・トーン |

エンジニアを構造段階から巻き込むと、「実装都合でこの構造は高くつく」「既存コンポーネントならこの形が安い」という**トレードオフの会話が手戻りゼロの段階**でできます（⑩章のスリーアミーゴスのデザイン版。デザイナー・PO・エンジニアで行う）。

### 曖昧さはカンプでなく基準で潰す

全状態のカンプを網羅するのは不可能です（ローディング・空・エラー・超長文・翻訳後の文字幅…）。網羅の代わりに**基準**を共有します。

- 「一覧が0件のときは共通のEmptyStateコンポーネントを使う」
- 「テキストは全て折り返し前提。省略記号は原則使わない」
- 「非同期処理は300ms超でスピナー表示」

この基準の集大成が次節のデザインシステムです。

## 4. デザインシステム — 協働のインフラ

**デザインシステム**＝再利用可能なコンポーネント群＋使い方の原則（色・余白・タイポグラフィのトークン、Button/Form/Modal等の部品、ライティングガイド）。

### アジャイルとの関係

- ストーリーごとのデザイン作業が「ゼロから描く」から「**部品を組み合わせる**」に変わり、デザインの先行リードタイムが劇的に縮む
- エンジニアは既存コンポーネントで**デザイナーの詳細関与なしに8割の画面を組める**ようになり、デザイナーは新規性の高い2割に集中できる
- Figma のコンポーネントとコードのコンポーネント（Storybook 等）を対応させ、**「デザインと実装の差分」を構造的に減らす**

### 運用の注意

- デザインシステム自体もプロダクト。**専任オーナーかギルド**（各チームの有志横断体）でバックログを持って育てる
- 「システムにない表現は禁止」と硬直させない。逸脱は**新パターンの候補**として回収する仕組みにする

## 5. スプリント内の協働の実際

- **デイリーにデザイナーも出る**: 「実装してみたらこの余白では収まらない」の解決が15分後にできるか、3日後のレビュー指摘になるかの分かれ目
- **ペアデザイン/デザインとのペア実装**: 難しい画面はデザイナーとエンジニアが並んで(または画面共有で)その場で調整しながら作る。カンプの往復より一桁速い
- **受け入れ確認にデザイナーを含める**: DoD に「デザイナーによるUI確認」を含めるチームは多い。ただし**確認待ちが滞留するならペア実装に切り替える**シグナル
- **デザインQAの観点を明文化**: 余白・整列・状態網羅・アクセシビリティ（コントラスト比・フォーカス順・代替テキスト）をチェックリスト化し、エンジニアが一次チェックできるようにする

## 6. ユーザーテストをリズムに組み込む

- **軽量ユーザビリティテスト**: 5人程度へのテストで主要な問題の大半が見つかるとされる（Nielsen の経験則）。毎スプリントまたは隔スプリントで定例化し、「テストする日」を固定すると準備が習慣化する（例: 毎週水曜は必ず誰かユーザーに会う）
- 対象はプロトタイプでも実装済み機能でもよい。**「作る前」(⑨章の検証)と「作った後」(改善の発見)の両方**で回す
- 録画・気づきはチーム全員で見る会を短く持つ。デザイナーだけが知っている状態にしない（エンジニアがユーザーの躓きを直接見ると、細部の実装判断が変わる）

## 7. アンチパターンまとめ

| アンチパターン | 症状 | 処方箋 |
|---|---|---|
| デザイン納品制 | カンプ完成までストーリー着手不可。手戻り大 | 1スプリント先行モデル＋構造段階からの共同作業 |
| ピクセルパーフェクト検品 | 実装後にデザイナーが数px単位の差分指摘を大量に出す | デザイントークン・コンポーネント共有で構造的に差分をなくす。基準で会話 |
| デザイン在庫の作り置き | 数スプリント先までカンプ完成済み。学びが反映されない | 先行は1スプリントまで。ディスカバリー活動に時間を回す |
| デザイナー抜きのバックログ | 実装可能性・工数だけで優先順位が決まる | リファインメント・プランニングにデザイナー参加 |
| 「あとでデザインシステム化」 | 同じボタンが7種類。修正が全画面個別対応 | 小さくても今から: 色とボタンのトークン化だけでも始める |
