# ㉓ ハイブリッド開発の技術構成 — ベストプラクティス・システム適性・技術スタック

[← 総合インデックスに戻る](README.md)

[⑲章](19-hybrid-waterfall-agile.md)（スマホアプリ）・[⑳章](20-hybrid-web.md)（Web）がプロセス面からハイブリッド開発（新規構築＝WF、保守・改善＝アジャイル）を扱ったのに対し、この章は**技術構成・アーキテクチャの面**から答えます。「どんなシステムならやり易いのか」「具体的にどう組めばよいのか」をまとめた総集編です。

## 1. ベストプラクティス10箇条（⑲⑳の総まとめ）

1. **不確実性で区間を分ける** — 事前に決められるもの（基盤）はWF、学びながら決めるもの（機能・UI）はアジャイル（[①章](01-agile-basics.md)）
2. **「開始順」でなく「リリース順」で統合を考える**（[㉑章](21-parallel-projects.md)）
3. **WFで決め切るのは「後から変えるコストが高いもの」だけ** — アーキテクチャ・認証・DB設計・URL設計・非機能。画面と機能は決め切らない（[⑲章](19-hybrid-waterfall-agile.md)§3）
4. **CI/CD・自動テスト・計測基盤は構築期の最初に作る** — 保守期に後付けは手遅れ（[⑤章](05-engineering-practices.md)）
5. **デプロイとリリースをフィーチャーフラグで分離する** — これがWFとアジャイルの技術的な接着剤（本章§3）
6. **移行期を明示的に設計する** — 体制の連続性・資産の引き継ぎ・計測の立ち上げ（[⑲章](19-hybrid-waterfall-agile.md)§4）
7. **保守期はスクラムバン＋割り込みバッファ20〜30%**（[③章](03-kanban-xp.md)）
8. **リリースの重さに周期を合わせる** — 審査があるアプリはリリーストレイン、Webは継続的デプロイ（[⑳章](20-hybrid-web.md)）
9. **経営報告はアウトカム＋Four Keysで** — 機能一覧の確約はしない（[⑦章](07-metrics-adoption.md)・[⑨章](09-product-discovery.md)）
10. **統合・追従の工数を明示的に計画へ載せる** — 見えない残業に吸収させない（[㉑章](21-parallel-projects.md)§6）

## 2. どんなシステムがハイブリッドをやり易いか

### 判定基準は3つ

| 基準 | 問い | なぜ効くか |
|---|---|---|
| ① 層の分離可能性 | 「固定する基盤」と「変化する機能」を境界で切れるか | WF区間とアジャイル区間を**別の場所**に割り当てられる |
| ② リリース可分性 | 一部だけ先に出せるか。小さく頻繁に出せるか | アジャイル区間の学習ループが回る |
| ③ 計測可能性 | 出した結果を数字で観測できるか | 保守・改善期の燃料。無いとアジャイル区間が回らない |

### 適性マトリクス

| システムタイプ | 適性 | 理由と補足 |
|---|---|---|
| **SaaS / 業務系Webアプリ** | ◎ | 3基準すべて満たしやすい。基盤（認証・テナント・課金）をWFで固め、機能をアジャイルで回す王道（[⑳章](20-hybrid-web.md)） |
| **ECサイト** | ◎ | 決済・在庫基盤は固定、売場UI・導線は高速改善。A/Bテストの効果も最大級 |
| **スマホアプリ + API構成** | ○ | 審査でリリースが重い分Web より一段WF寄りになるが、フラグと定期便で十分回る（[⑲章](19-hybrid-waterfall-agile.md)） |
| **社内業務システム** | ○ | ユーザーが社内なので限定公開・巻き戻しが容易。ただし「全部要件で決めたがる」文化との戦いが本題になりがち（[⑫章](12-agile-in-japan.md)） |
| **データ基盤・バッチ中心** | △ | リリース可分性は作れる（パイプライン単位）が、UIがなく計測がコスト・品質指標中心になる。WF比率高めが現実的 |
| **基幹システム刷新** | △ | 一括刷新はWFでも失敗率が高い。ストラングラーフィグで周辺から切り出せばハイブリッド化できる（本章§4パターンC・[⑭章](14-legacy-code.md)） |
| **規制産業（医療機器・金融勘定系等）** | △ | 文書・工程が規制で固定される部分はWF。規制対象外のUI・周辺はアジャイルに分離する二層構え |
| **組み込み / ハード同梱** | ✗〜△ | リリースがほぼ1回・ロールバック不可。内部イテレーション＋シミュレータ検証はできるが、ハイブリッドの旨味は薄い |

**結論**: 「**固定できる基盤と、変化する機能・UIを境界で分離できるシステム**」がやり易い。逆に言えば、分離できない一枚岩はまず分離（§4パターンC）から始めます。

## 3. 構成の設計原則 — 「固定層」と「可変層」を境界で切る

ハイブリッドを技術構成に翻訳すると、こうなります。

```
┌─────────────────── 可変層（アジャイルで高速変更）───────────────────┐
│  画面・UIコンポーネント / 機能ロジック / 文言・導線 / 実験(A/B)      │
│  → フィーチャーフラグで出し分け・毎週〜毎日変わる                    │
├──────────────────── 境界（契約として固定・変更は管理）──────────────┤
│  API契約(OpenAPI/GraphQLスキーマ) / イベント定義 / コンポーネントI/F │
├─────────────────── 固定層（WFで設計し切る・変更は重い）─────────────┤
│  認証・認可 / 課金 / DBスキーマ / マルチテナント設計 /              │
│  インフラ(IaC) / セキュリティ / 監視・計測基盤                      │
└──────────────────────────────────────────────────────────────┘
```

- **固定層**: 構築期（WF区間）に設計レビュー・ADR（[⑱章](18-templates.md)）付きで決め切る。保守期の変更は「重い変更」として別管理
- **境界**: API・スキーマは**契約テスト**（OpenAPIからの自動検証等）で守る。境界が守られている限り、可変層は自由に動ける
- **可変層**: トランクベース＋フラグ＋自動テストで毎日変えられる状態を維持
- この分離は[⑥章](06-scaling.md)の逆コンウェイと同じ話 — **層の分離はチームの分離（基盤チーム/機能チーム）にも対応**させられる

## 4. 具体的なシステム構成 3パターン

### パターンA: スマホアプリ（小規模〜中規模・スタートアップ向け）

**Flutter + BaaS（Firebase）構成**。サーバー実装を最小化し、可変層の速度を最大化する構成です。

```
[Flutter アプリ]──┬── Firebase Auth(認証)           ←固定層
 ├ UI/機能(可変層)  ├── Cloud Firestore(DB)          ←スキーマ設計は固定層
 ├ Remote Config   ├── Cloud Functions(サーバーロジック)
 │  (フラグ/出し分け)├── FCM(プッシュ通知)
 └ 課金SDK ────────┴── RevenueCat 等(サブスク管理)    ←固定層
                        ↓
        Crashlytics / Google Analytics(計測)・Firebase A/B Testing
```

| 層 | 技術スタック例 |
|---|---|
| フロントエンド | Flutter（状態管理: Riverpod / Bloc）＋テスト（[㉒章](22-flutter-testing.md)） |
| バックエンド | Firebase（Auth / Firestore / Cloud Functions）または Supabase |
| 課金 | RevenueCat + ストアIAP（[Flutterサブスク構成](../flutter-subscription-system-design.md)参照） |
| CI/CD | GitHub Actions + Fastlane（ビルド・署名・TestFlight/内部テスト配布・ストア提出） |
| フラグ/実験 | Firebase Remote Config + A/B Testing（アプリ更新なしで機能ON/OFF） |
| 計測・監視 | Crashlytics（クラッシュ）+ GA4（行動）+ ストアレビュー監視 |

- WF区間で固める: Firestoreのデータ設計（後からの変更が最も痛い）・認証方式・課金モデル・Remote Configのキー設計
- アジャイル区間: 画面・機能をフラグ配下で毎スプリント追加、隔週の定期便でリリース（[⑲章](19-hybrid-waterfall-agile.md)§5）

### パターンB: Webアプリ / SaaS（継続的デプロイ前提）

```
[ブラウザ] ── [Next.js(SSR/フロント)] ── [BFF/API (REST or GraphQL)] ── [アプリケーションサーバー]
                    │                         │OpenAPIスキーマ=契約(境界)      │
                    │                         │                        [マネージドDB(PostgreSQL)]
              [フラグSDK]                [認証基盤(IDaaS)]                [IaC(Terraform)]
                    ↓
      Vercel/クラウドへ継続的デプロイ(カナリア→100%) + Sentry/Datadog(監視) + A/Bテスト基盤
```

| 層 | 技術スタック例 |
|---|---|
| フロントエンド | Next.js / React（or Vue+Nuxt）。デザインシステム（Storybook）併設（[⑮章](15-design-collaboration.md)） |
| バックエンド | Node.js(NestJS) / Go / Rails 等 + OpenAPIで契約管理（モジュラーモノリスから開始が推奨） |
| DB / インフラ | マネージドPostgreSQL（RDS / Cloud SQL）+ Terraform(IaC) |
| 認証 | IDaaS（Auth0 / Firebase Auth / Cognito）— 自作しないのが定石 |
| CI/CD | GitHub Actions → コンテナ（Cloud Run / ECS）or Vercel。マージで自動デプロイ＋カナリア |
| フラグ/実験 | LaunchDarkly / Unleash(OSS) / PostHog。A/BテストはPostHog / Optimizely |
| 計測・監視 | Sentry（エラー）+ Datadog/Grafana（メトリクス）+ GA4/Amplitude（行動） |

- WF区間で固める: URL設計・レンダリング戦略・マルチテナント/認可モデル・DBスキーマ・OpenAPI契約（[⑳章](20-hybrid-web.md)§3）
- アジャイル区間: マージ→自動デプロイ→カナリア→計測のループを毎日回す。「効果測定込みでリリース」を標準に

### パターンC: 既存基幹 × 新周辺（大企業・レガシー環境向け）

基幹（変えられない・WFの塊）を**そのまま固定層として扱い**、周辺に可変層を增築する構成。[⑭章](14-legacy-code.md)のストラングラーフィグの構成版です。

```
[ユーザー] ── [新フロント(Web/アプリ)] ── [APIゲートウェイ/BFF] ─┬─ [新マイクロサービス群] ←アジャイル(可変層)
                                          (境界・振り分け)      │      │
                                                              │  [新DB(サービス毎)]
                                                              └─ [連携層(API化/CDC)] ── [既存基幹システム] ←WF管理のまま(固定層)
                                                                                          │
                                                                                     [基幹DB(触らない)]
```

| 層 | 技術スタック例 |
|---|---|
| 新フロント | Web: Next.js / アプリ: Flutter（パターンA/Bと同じ） |
| 境界 | APIゲートウェイ（Kong / Apigee / API Gateway）+ BFF。**基幹を直接触らせない**のが最重要 |
| 新サービス | コンテナ（Kubernetes / Cloud Run）上の小さなサービス群。最初はBFF内のモジュールで十分 |
| 基幹連携 | 基幹のAPI化（薄いラッパー）、またはCDC（Debezium等）でデータを新側へ複製 |
| CI/CD | 新側だけ継続的デプロイ。基幹は従来のリリース管理のまま（無理に変えない） |
| 計測・監視 | 新側にSentry/Datadog。基幹との突き合わせ（リコンサイル）ジョブを監視 |

- **基幹のリリース管理はWFのまま残してよい**のがこのパターンの肝。変えるのは境界の外側だけ
- 移行はスプリントごとに新側の担当範囲を拡大（[⑭章](14-legacy-code.md)§4）。シャドーイングで新旧の出力を突き合わせてから切替

## 5. 横断的な選定基準 — ハイブリッドを支える4つの道具

どのパターンでも、この4つが揃っているかがハイブリッド成立の技術条件です。

| 道具 | 選択肢（左ほど手軽） | 選定の目安 |
|---|---|---|
| フィーチャーフラグ | Firebase Remote Config → Unleash(OSS) → LaunchDarkly | まず無料圏で開始。実験・段階公開・権限管理が要るなら専用SaaSへ |
| CI/CD | GitHub Actions → CircleCI / GitLab CI | リポジトリと同じ場所が原則。モバイルはFastlane併用 |
| エラー・クラッシュ監視 | Crashlytics(アプリ) / Sentry(Web) → Datadog | リリース直後の異常検知が段階公開の前提条件 |
| 行動計測・A/B | GA4 → Amplitude / PostHog | 「改善の効果を数字で示す」ための燃料。構築期に埋め込む |

**選定の共通原則**: ①固定層は「枯れて実績のあるもの」を選ぶ（冒険しない）②可変層の道具は「変更の速さ」で選ぶ ③自作は最後の手段（認証・課金・フラグ基盤は買う/借りるのが定石）。

## 6. 立ち上げチェックリスト（技術版）

```markdown
## 構築期（WF区間）の終わりまでに
- [ ] 固定層/境界/可変層の区分を図にして関係者と合意した
- [ ] 固定層の設計（認証・課金・DBスキーマ・API契約）がADRとして記録されている
- [ ] API契約がスキーマ（OpenAPI等）で管理され、契約テストがCIにある
- [ ] マージ→自動テスト→デプロイのパイプラインが動いている
- [ ] フィーチャーフラグ基盤が導入され、1機能以上で実際に使った
- [ ] エラー監視・行動計測が本番相当環境で動いている

## 保守・改善期（アジャイル区間）に入ったら
- [ ] 可変層の変更はフラグ配下で小さくマージされている（長命ブランチがない）
- [ ] 段階公開（カナリア/%ロールアウト）と即時無効化の手順を訓練した
- [ ] 固定層の変更は「重い変更」として別レーンで管理されている
- [ ] 使い終わったフラグを削除する運用が回っている
- [ ] 改善の効果が計測データでレビューに出てくる（⑨章のループ）
```
