# ⑳ Webアプリでのハイブリッド開発 — スマホアプリとの違いと設計の変更点

[← 総合インデックスに戻る](README.md)

[⑲章](19-hybrid-waterfall-agile.md)（新規構築＝ウォーターフォール、保守・改善＝アジャイル）の**Webアプリ版**です。3フェーズの骨格は同じですが、Webは「配布」の性質が根本的に違うため、**ハイブリッドの最適な配合比が変わります**。結論から言うと、**Webはスマホアプリよりさらにアジャイル寄りに倒せる**（そして倒すべき）です。

## 1. 何が根本的に違うのか

| 観点 | スマホアプリ | Webアプリ | ハイブリッド設計への影響 |
|---|---|---|---|
| 配布 | ストア経由。**審査あり**・反映まで日単位 | 自社サーバーへのデプロイのみ。**分単位** | Webはリリーストレイン不要。いつでも出せる |
| ロールバック | 不可（前バージョンに戻せない） | **即時可能**（直前ビルドに戻すだけ) | Webは失敗のコストが桁違いに低い → 小さく頻繁に |
| バージョン共存 | 旧バージョン利用者が残り続ける | 全ユーザーが**常に最新版** | WebはAPI後方互換の重荷が軽い（自社ブラウザ分のみ） |
| 初回リリースの重さ | 一発勝負感が強い（審査・DL障壁・レビュー） | **URLを知らせるだけ**。限定公開も自在 | Webは「こっそり出して育てる」が可能 → WF区間を短縮できる |
| 段階公開 | ストアの段階的ロールアウト機能 | **自前で自由に**（カナリア・Blue-Green・%指定） | Webの方が細かく制御できる |
| 実験 | A/Bテストはフラグ実装頼み・反映遅い | A/Bテスト・計測が**最も強力に使える** | フェーズ3の仮説検証ループ（⑨章）が高速に回る |
| 環境差異 | OSバージョン・端末の分断 | ブラウザ・画面サイズの分断（ただし自動更新で近年は緩い） | 検証マトリクスの組み方が変わる（§4） |
| 常時変化する外部要因 | OSメジャーアップデート（年1回） | ブラウザ更新（6週間ごと）・依存パッケージ脆弱性 | Webは「小さく常時追従」型の保守が必要 |

## 2. 結論: 配合比をこう変える

### スマホアプリ（⑲章）: WF区間を長めに取る合理性があった

審査・一発勝負・ロールバック不可という**リリースの摩擦**が大きいため、構築期に決め切る量を増やし、リリースを重いイベントとして計画する価値があった。

### Webアプリ: WF区間は「基盤と骨格」だけに絞る

リリースの摩擦がほぼゼロなので、**「完成してから公開」を待つ理由が構造的に薄い**。推奨する配合は:

```
フェーズ1（WF的に決め切る・短め）      フェーズ1.5（限定公開で育てる）      フェーズ2〜3（アジャイル）
┌─────────────────────┐ ┌─────────────────────┐ ┌────────────────┐
│ 要件の骨格・アーキテクチャ    │ │ 社内公開 → 招待制β → 一部  │ │ 正式公開後の保守・改善  │
│ 認証/課金/DB設計・非機能     │→│ 顧客に公開。本番データで学び  │→│ （⑲章フェーズ3と同じ  │
│ セキュリティ・計測基盤       │ │ ながら機能を足していく       │ │  仕組みで運用）       │
└─────────────────────┘ └─────────────────────┘ └────────────────┘
     1〜3ヶ月目安                「リリースの日」を薄く分散させる
```

- スマホの「M4 β版→審査→公開」という**崖**が、Webでは**スロープ**（限定公開の範囲を広げていく）に置き換わる
- 「正式リリース日」はマーケティング上のイベントとしては残してよい。ただし**技術的にはその日が初稼働にならない**ようにする（公開日の何週間も前から本番環境で実ユーザーが使っている状態が理想）
- 発注・稟議の都合でWF計画が必要な場合も、**マイルストーンの中身を「公開範囲の拡大」で定義**すると両立しやすい（M1: 社内50人 → M2: 招待100社 → M3: 全公開）

## 3. フェーズ1で決め切るもの — Web特有の項目

⑲章の「決め切るもの」リスト（期日・予算・体制・アーキテクチャ・非機能）に加えて、Webでは以下を最初に固定します。**後から変えるコストが高いものだけをWFで決める**という原則は同じです。

| 項目 | なぜ最初か |
|---|---|
| レンダリング戦略（SPA / SSR / SSG） | SEO・初期表示性能・アーキテクチャ全体を規定。後からの転換は事実上の作り直し |
| URL設計 | SEO資産・共有リンク・ブックマークは蓄積型。後の変更はリダイレクト地獄 |
| 認証・マルチテナント設計 | データ分離の方式（B2Bなら特に）は後から変えられない代表格 |
| 対応ブラウザ・画面サイズの下限 | 検証コストとCSS設計を規定。「IE対応やっぱり必要」は最悪の後出し |
| セキュリティ要件（OWASP対応・監査ログ） | 後付けの認可制御は漏れる。設計段階で |
| 計測・A/Bテスト基盤 | フェーズ1.5以降の「学びながら育てる」の前提装置 |

逆に**画面・機能はスマホ以上に決め切らない**。フェーズ1.5で本番の実ユーザーから学べるからです。

## 4. フェーズ3（保守・改善期）の違い

⑲章のスクラムバン＋割り込みバッファの構成はそのまま使えます。違いは以下です。

### リリース: トレインから継続的デプロイへ

- 隔週トレインではなく、**マージされたら自動で本番へ**（継続的デプロイ: [⑤章](05-engineering-practices.md)）が到達目標。少なくとも「出したい時にいつでも出せる」状態を維持する
- 安全装置はスマホと共通の**フィーチャーフラグ**に加え、**カナリアリリース**（新版を数%のトラフィックだけに向けて監視）と**即時ロールバック**。この3点があれば「リリース＝怖いイベント」が消える
- デプロイ頻度・変更障害率などFour Keys（[⑦章](07-metrics-adoption.md)）はWebだと素直に測れる。改善の主要指標にする

### 保守の中身: 「年1回の大物」から「常時の小物」へ

| スマホ（⑲章） | Web |
|---|---|
| OSメジャーアップデート対応（年1回の恒例行事として容量確保） | ブラウザ更新は自動追従が基本。代わりに**依存パッケージの脆弱性対応が常時発生** |
| 旧バージョン互換の維持 | 不要（常に最新版）。代わりに**キャッシュ起因の新旧混在**（デプロイ直後だけ旧JSが動く）への対処 |
| ストア規約変更への対応 | 法規制対応（Cookie同意・アクセシビリティ・個人情報）が相当品 |

- 脆弱性対応は Dependabot / Renovate 等で**自動PR化し、カンバンの割り込みレーンで日常処理**する。溜めて四半期ごとにやるのは最悪手（差分が巨大化して当てられなくなる）
- ブラウザ検証は全組み合わせ網羅ではなく、**アクセス解析上位＋自動E2E（クリティカルパスのみ: [⑩章](10-agile-testing.md)）**に絞る

### 改善: A/Bテストを主兵装に

- Webはランダム出し分け・即時反映・十分なサンプルが揃いやすく、**仮説駆動ループ（[⑨章](09-product-discovery.md)）が最速で回る**。「リリース後に効果測定」ではなく「効果測定込みでリリース」を標準にする
- SEO・表示速度（Core Web Vitals）という**Web特有の継続改善テーマ**をバックログの定常枠として持つ

## 5. スマホ版とWeb版を両方持つ場合（実務で最も多いケース）

1つのサービスでアプリとWebを並行提供する場合の指針です。

- **バックエンドとAPIは1つ**。リリース周期が違う2つのクライアント（Web=毎日 / アプリ=隔週便）がぶら下がるため、**APIの後方互換はアプリ側の都合で維持**する（Webだけなら不要だった重荷がアプリのせいで復活する点に注意）
- 新機能の検証は**Webで先行**させるのが定石: A/Bテストで当たりを確認してからアプリの定期便に載せると、審査リスクとフラグ実装コストを当たり確定後に払える
- チーム編成は「Web班/アプリ班」より**機能単位のフィーチャーチーム**（[⑥章](06-scaling.md)の逆コンウェイ）が理想。ただしモバイル専門性は Chapter/ギルド的な横串で担保する
- バックログは1本に保つ（[⑥章](06-scaling.md)の優先順位の一元化）。「Web用バックログ」「アプリ用バックログ」に分けると全体最適が崩れる

## 6. よくある失敗（Web版で追加されるもの）

| 失敗パターン | 何が起きるか | 対策 |
|---|---|---|
| Webなのに「完成後一斉公開」 | スマホの崖をわざわざ再現。学べたはずの数ヶ月を捨てる | フェーズ1.5（限定公開のスロープ）を計画に組み込む |
| いつでも出せるのに月1リリース | ロールバック可という最大の強みを放棄。変更が溜まり障害率上昇 | デプロイの自動化→頻度を上げる→さらに安全になる好循環へ |
| 脆弱性対応の貯金化 | 半年分の依存更新がまとめて来て、当てられず塩漬けに | 自動PR＋割り込みレーンで日常化 |
| URL・レンダリング戦略の後決め | SEO崩壊・作り直し級の手戻り | §3の表をフェーズ1の必須決定事項に |
| アプリとWebのバックログ分裂 | 同じ機能を二重に別仕様で実装、優先順位も別々 | バックログ1本化＋Web先行検証の定石（§5） |

## 7. 追加チェックリスト（⑲章のリストに足す）

```markdown
## Web版で追加確認するもの
- [ ] レンダリング戦略・URL設計・対応ブラウザ下限を最初に固定した
- [ ] 「正式公開日」より前に本番で実ユーザーが使う計画（限定公開のスロープ）がある
- [ ] カナリアリリースと即時ロールバックの手順が整備・訓練済み
- [ ] 依存パッケージの脆弱性対応が自動PR＋日常処理になっている
- [ ] A/Bテスト基盤が動いており、主要な改善は効果測定込みでリリースしている
- [ ] （アプリ併用時）バックログは1本・APIの後方互換方針が明文化されている
```
