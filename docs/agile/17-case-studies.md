# ⑰ ケーススタディ集 — 4つの導入物語と教訓

[← 総合インデックスに戻る](README.md)

タイプの異なる4つの組織の導入ストーリーです。**いずれも典型的な事例パターンを組み合わせた架空のケース**ですが、起きる問題と打ち手は現場で繰り返し観察される現実的なものを再構成しています。各ケースの末尾に、対応する章への参照を付けています。

---

## ケースA: スタートアップ — 「アジャイル以前」からの立ち上げ

### 状況

創業2年目のヘルスケア系スタートアップ。エンジニア4名＋代表＋デザイナー1名。プロセスは特になく、代表の思いつきがSlackで飛んできて、みんなが手を止めて対応する日々。リリースは「できたら出す」。

### 何が起きていたか

- 全員が常に3〜4案件を抱え、どれも進まない（WIP無制限）
- 「先週頼んだあれどうなった？」が頻発。誰も全体像を知らない
- 大型の新機能を3ヶ月作り込んでリリースしたが、利用率は数%だった

### 打ち手

1. **カンバンから開始**（スクラムではなく）。全仕事をボードに出し、WIP制限を1人1枚に設定。「これ以上は積めません。どれと入れ替えますか？」と代表に**選ばせる**構造を作った
2. 代表を**POとして週1の優先順位決め**に固定。思いつきは全てバックログへ（直接Slackで依頼するのは禁止）
3. 3ヶ月の作り込みの反省から、**仮説キャンバスとMVP検証**を導入。次の大型機能は、まずLPテストと20人のインタビューで需要を確認してから作った

### 結果と教訓

- リードタイム（依頼→リリース）が平均6週間→9日に短縮
- **教訓1**: 小さい組織はイベントの重いスクラムより**カンバン＋週1優先順位決め**が立ち上がりやすい（③章の選定指針どおり)
- **教訓2**: 創業者の「思いつき直通ルート」を塞ぐことが最重要かつ最難関。プロセスでなく**構造（バックログ経由の一本化）**で解決する
- **教訓3**: 「作る速さ」より先に「作るものの検証」（⑨章）。3ヶ月の空振りはスタートアップには致命傷になり得る

→ 関連: [③カンバン](03-kanban-xp.md)・[⑨ディスカバリー](09-product-discovery.md)

## ケースB: 受託SIer — 契約の壁を段階的に越える

### 状況

従業員300名のSIer。主要顧客の物流会社から基幹周辺システムを一括請負で受注し続けて8年。毎回、検収直前の仕様齟齬と手戻りで炎上し、利益率が低下。「アジャイルでやりたい」が、顧客は「金額と納期が決まらない契約は稟議が通らない」。

### 打ち手（3段階・2年がかり）

1. **フェーズ1（契約はそのまま、内部だけ変える）**: CI/自動テスト・週次の社内デモ・ふりかえりを導入。さらに顧客に「**月1回、動くものを見せる会**」を無償で提案。「進捗報告が資料でなく実物になるだけです」と説明した
2. **フェーズ2（デモが信頼を生む）**: 月1デモで仕様齟齬が早期発見され、検収時の手戻りが激減。顧客側の担当課長が「毎回見るなら、優先順位もこちらで並べたい」と言い出す — **事実上のPO誕生**
3. **フェーズ3（契約を変える）**: 次期案件で、IPAモデル契約を添えて**基本契約＋3ヶ月ごとの準委任個別契約**を提案。「3ヶ月ごとに継続判断できるため、御社のリスクはむしろ下がる」という説明で顧客の購買部門を通した

### 結果と教訓

- 検収時の重大不具合が前案件比8割減。顧客満足度調査で過去最高
- **教訓1**: 契約変更を最初に持ち出さない。**「動くものを見せる」は契約に関係なく始められ**、それが契約を変える信頼を作る（⑫章パターンB そのもの）
- **教訓2**: 顧客側POは任命するものではなく、デモの場が**育てる**もの
- **教訓3**: 準委任への切り替えは「発注側のリスクが下がる」と翻訳して初めて購買・法務に通る

→ 関連: [⑫日本の実務](12-agile-in-japan.md)・[⑦導入ステップ](07-metrics-adoption.md)

## ケースC: 大企業の内製部門 — レガシーと兼務の沼から

### 状況

製造業大手の社内システム部門、40名。15年物の基幹システム（テストなし・手作業リリース）を保守しつつ、DX号令で「アジャイルで新機能を作れ」と指示が降りた。全員が3〜5プロジェクト兼務。過去に一度スクラム導入を試み、「会議が増えただけ」と不評のまま自然消滅した過去がある。

### 何が起きていたか

- デイリーに出ても頭は別プロジェクト。ベロシティは毎スプリント乱高下し予測に使えない
- スプリントで作っても、リリースは四半期に1回の「リリース祭り」待ち。手作業3日がかり
- 前回導入失敗の記憶から「またか」という冷めた空気（学習性無力感）

### 打ち手

1. **1チームだけ、専任で編成**（5名）。部門長と交渉し「このチームの成果指標はFour Keysで四半期報告する」と約束して専任を勝ち取った
2. スクラムイベントより先に**リリース自動化に2スプリント投資**。デプロイ3日→40分に
3. レガシー部分は**「触る場所にだけ仕様化テスト」ルール**を導入。カバレッジ目標は設定しなかった
4. 過去の失敗経験者には「前回と違う点」を明示: 専任であること・技術投資が先であること・成果を数字で示すこと

### 結果と教訓

- 6ヶ月でデプロイ頻度が四半期1回→週2回、変更障害率は横ばい（悪化させずに高速化）
- その数字を見た他部門から「うちも」と声がかかり、2チーム目が発足
- **教訓1**: 兼務の解消は快適さの問題ではなく**成立条件**。ここを妥協した導入は前回同様に死ぬ
- **教訓2**: 大企業レガシー環境では**技術的土台（⑤⑭章）を先に**。イベントだけ入れた前回の失敗の直接の敗因
- **教訓3**: 再挑戦では「前回と何が違うか」を明示的に語る（⑬章の抵抗マネジメント）。数字（Four Keys）は冷めた組織への最強の説得材料

→ 関連: [⑤技術プラクティス](05-engineering-practices.md)・[⑭レガシーコード](14-legacy-code.md)・[⑬コーチング](13-agile-coaching.md)

## ケースD: 失敗事例 — 「全社アジャイル宣言」の崩壊と再起

### 状況

急成長したWebサービス企業、エンジニア120名。役員が外部セミナーに感化され「来期から全社アジャイル」を宣言。研修会社に依頼して全チーム一斉にスクラム導入。専任コーチなし、各チームのリーダーが3日研修を受けてSM兼任。

### 何が起きたか（転落の記録）

- **1ヶ月目**: 全チームがイベントを開始。「会議が3倍になった」という不満が噴出
- **3ヶ月目**: 経営がチーム別ベロシティのランキングを月次会議で表示し始める。ポイントのインフレが始まり、あるチームは3ヶ月でベロシティが「4倍」になった（もちろん生産性は変わっていない）
- **6ヶ月目**: レビューは役員向け報告会と化し、悪いニュースが消えた。デモは「うまくいく画面だけ」を見せる劇場に
- **9ヶ月目**: 主力エンジニアが複数退職。退職面談で「スクラムごっこに疲れた」。役員は「アジャイルは失敗だった。元に戻す」と宣言

### 何が敗因だったか

| 敗因 | 違反した原則 |
|---|---|
| 全チーム一斉導入 | パイロットで小さく学ぶ（⑦章）— 導入自体にアジャイルを適用しなかった |
| ベロシティのランキング化 | グッドハートの法則（⑦章）。指標を評価に使った瞬間に全指標が汚染された |
| SM教育の不足・兼任のみ | 守破離の「守」を支える人がいない（⑬章） |
| 悪いニュースが消えた | 心理的安全性の崩壊（⑧章）。検査と適応が両方止まった |
| 技術投資ゼロ | イベントだけ増え、リリースは速くならず「成果が見えない」評価に |

### 再起

新任VPoEが1年後に再挑戦。今度は**1チーム・専任コーチ・ベロシティ非公開・Four Keysのみ計測**で開始し、18ヶ月かけて8チームまで拡大した。全社宣言はしなかった。「アジャイルという言葉を使わない」方針すら取った — 前回の言葉が汚染されていたからである。

### 教訓

- **教訓1**: 導入の失敗はほぼ**手法以外**（評価制度・一斉展開・支援不足）で起きる。にもかかわらず「アジャイルは失敗した」と手法が葬られる
- **教訓2**: 指標の扱いは導入設計の最重要事項。**測る前に「何に使わないか」を宣言する**
- **教訓3**: 一度失敗した組織での再起は、規模を落とし、言葉を変え、数字で語る。文化の傷は制度より治りが遅い

→ 関連: [⑦アンチパターン](07-metrics-adoption.md)・[⑧心理的安全性](08-team-and-culture.md)

## 4ケースの横断比較

| | A: スタートアップ | B: 受託SIer | C: 大企業内製 | D: 全社一斉（失敗→再起） |
|---|---|---|---|---|
| 入口の手法 | カンバン | 社内実践＋月1デモ | スクラム（1チーム専任） | スクラム（全社一斉）✗ |
| 最大の壁 | 創業者の直通依頼 | 契約形態 | 兼務とレガシー | 評価制度と焦り |
| 効いた打ち手 | WIP制限で選ばせる | デモが信頼と PO を育てた | 技術投資が先・Four Keys報告 | （再起時）小さく・言葉を変え・数字で |
| 共通する教訓 | **小さく始め、構造を変え、数字で語り、指標を評価に使わない** | | | |
