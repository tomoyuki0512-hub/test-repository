# ⑫ 日本の実務とアジャイル — 契約・組織構造・現実的な始め方

[← 総合インデックスに戻る](README.md)

アジャイルの原則は普遍的ですが、実際の導入は**契約慣行・組織構造・雇用慣行**という土地の事情に大きく左右されます。この章では日本のIT実務に固有の論点を深掘りします。

## 1. なぜ日本でアジャイルは難しいと言われるのか — 構造要因

### 受託開発中心の産業構造

日本のIT投資はユーザー企業自身の内製ではなく、SIer への外注比率が歴史的に高く、多重下請け構造も残っています。これがアジャイルと衝突します。

| 構造 | アジャイルとの衝突点 |
|---|---|
| 発注側にIT人材が少ない | PO を務められる人が発注側にいない。「丸投げ」になりがち |
| 多重下請け | チームと顧客の間に何層も挟まり、「顧客との協調」「対面の会話」が物理的に困難 |
| 一括請負契約が標準 | 仕様固定・完成責任前提の契約と「変化への対応」が矛盾する |
| 人月ベースの商習慣 | 価値(アウトカム)ではなく工数(アウトプット)で対価が決まる |

### それでも変わりつつある

- ユーザー企業の内製化・エンジニア採用の拡大（金融・小売・製造でも内製組織の事例が増加）
- 官公庁でもデジタル庁がアジャイル前提の調達ガイドを整備
- IPA・経産省が契約モデルや導入ガイドを公開し、「契約の壁」への公式な解が示されつつある

## 2. 契約 — アジャイルと契約形態

### 基本の整理

| 契約形態 | 性質 | アジャイルとの相性 |
|---|---|---|
| 請負契約 | 完成物に対する責任（契約不適合責任）。仕様の確定が前提 | ×〜△ 仕様変更のたびに変更契約が必要になり、変化への対応を殺す |
| **準委任契約** | 業務の遂行（善管注意義務）に対する対価。成果物完成責任はない | **◎ アジャイルの基本形**。スプリントごとに優先順位を変えられる |

### IPA「アジャイル開発版 モデル取引・契約書」の要点

IPA（情報処理推進機構）が公開しているモデル契約は、準委任を前提に次を整理しています。

- **発注者がプロダクトオーナーを出す**ことを契約上の前提として明記（丸投げの防止）
- スクラムの役割・イベント・成果物を契約の別紙で定義し、双方の責任分担を明確化
- 「アジャイル開発に適したプロジェクトか」を判定する**事前チェックリスト**付き（経営層の関与・PO の確保・チーム体制など）

発注側・受注側どちらの立場でも、社内の法務・調達を説得する際の一次資料として非常に有効です。

### 実務で使われる契約の工夫

- **フェーズ分割**: 初期は短期の準委任（例: 3ヶ月）で始め、相互の信頼とベロシティの実績ができてから更新していく
- **基本契約＋個別契約**: スプリント群を個別契約の単位にし、継続判断のポイントを定期的に設ける
- **成果報酬・レベニューシェア**: 一部をアウトカムに連動させる形。運用は難しいが価値ベースへの一歩
- 発注側の心得: 準委任は「成果が出なくても払う契約」ではなく、**POとして毎スプリント成果を検査し方向を正す権利と責務**を持つ契約と捉える

## 3. 偽装アジャイル（Fake Agile）の見分け方

形だけアジャイルの語彙を使う状態は日本に限りませんが、受託構造下で起きやすいパターンがあります。

| 兆候 | 実態 |
|---|---|
| 「アジャイルなので仕様書はありません」 | ドキュメント軽視の言い訳。必要な文書まで放棄している |
| 「アジャイルなのでいつでも何でも変更できます」 | スプリントゴールも計画もない行き当たりばったり |
| 全工程・全機能の詳細見積りを出した上で「スプリントで作ります」 | 中身はウォーターフォールの工程をスプリントと呼んでいるだけ |
| デモがなく、進捗は資料で報告 | 動くソフトウェアが尺度になっていない |
| POが受注側にいる（発注側が不在) | 「顧客との協調」が構造的に欠落。要件の伝言ゲーム化 |
| ふりかえりがない・議事録だけある | 検査と適応が止まっている |

**チェックの核心**: 「①動くものを短い間隔で見せているか」「②見せた結果で計画が変わることがあるか」。②が一度もないなら、それはアジャイルの形をした逐次開発です。

## 4. 日本組織での現実的な始め方

### パターンA: 内製チームから始める（推奨）

1. 影響範囲を限定できる**社内向けツール・新規サービス**で1チーム立ち上げる（⑦章のパイロット）
2. 経営層には「Four Keys などの指標で四半期ごとに報告する」と**先に測り方を約束**する（「アジャイルだから計画がない」という誤解を防ぐ）
3. 成功体験と社内事例を作ってから横展開する

### パターンB: 受託の関係の中で始める

1. まず**受注側チーム内部**の実践（CI/CD・自動テスト・カンバン・ふりかえり）から始める — 契約を変えずにできる
2. 顧客に**スプリントレビュー相当のデモ**を定例化する — 「途中で動くものを見せる」だけでも手戻りが激減し、信頼が生まれる
3. 信頼ができた段階で、次期フェーズを**準委任＋PO 参画**の形に提案する（IPAモデル契約を添えて）

### パターンC: 大企業の既存部門で始める

- 稟議・予算の年度サイクルとスプリントの整合が課題になる。**予算はテーマ（アウトカム）単位で確保し、中身の機能はバックログで柔軟に**という「リーン予算」的な運用を財務と交渉する
- 品質保証部門・監査との調整: DoD・CI の証跡が従来の工程完了判定の代替になることを説明する（証跡は自動化の方がむしろ充実する）
- 兼務だらけのチームは失敗の最大要因。**専任率を上げる交渉**を最優先にする

## 5. 日本の商習慣とスクラムの意外な親和性

アジャイルの源流には日本企業の研究があります。

- スクラムの名前の由来は、野中郁次郎・竹内弘高の論文 **“The New New Product Development Game”**（1986年, Harvard Business Review）。ホンダ・キヤノン等の製品開発の「ラグビー型」チーム（部門横断・自律的・重なりのあるフェーズ）を分析したもので、Sutherland らがこれを参照した
- **トヨタ生産方式**（ジャストインタイム・自働化・カイゼン・現地現物）はリーン・カンバンの直接の源流
- つまり「アジャイルは外来思想で日本に合わない」は歴史的には逆で、**自律チームと継続的カイゼンはもともと日本企業の強みだった**。障壁は文化ではなく、契約・組織構造・人事制度という制度側にある

## 6. 個人のキャリアとしてのアジャイル（日本市場）

- 求人市場では「スクラム経験」はほぼ標準要件化し、**スクラムマスター・アジャイルコーチ**の専門職求人も増加
- 資格: **CSM**（Certified ScrumMaster, Scrum Alliance）、**PSM**（Professional Scrum Master, Scrum.org）が2大資格。日本語受験可。資格は入口であり、現場での実践経験が本体
- コミュニティ: Regional Scrum Gathering Tokyo、スクラムフェス（大阪・新潟・仙台ほか各地）、Agile Japan など活発なカンファレンスが多数。現場の生きた事例に触れる最短ルート
