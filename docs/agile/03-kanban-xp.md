# ③ カンバン・XP・その他手法

[← 総合インデックスに戻る](README.md)

スクラム以外の代表的なアジャイル手法を解説します。実務では「スクラム＋XPの技術プラクティス」「スクラムにカンバンの流れ管理を足す（スクラムバン）」のように**組み合わせて使う**のが一般的です。

## 1. カンバン（Kanban）

### 概要

カンバンは、トヨタ生産方式の「かんばん」に着想を得た**フロー（流れ)を管理する手法**です。スクラムと違い、スプリントという固定の区切りやロールの定義を持ちません。**今のやり方を変えずに始められ、流れを可視化して漸進的に改善していく**のが特徴です。

### カンバンの基本原則

1. **可視化する** — 仕事の流れをボード上のカードで見える化する
2. **WIP（仕掛り作業）を制限する** — 各ステージで同時に進められる枚数に上限を設ける
3. **流れを管理する** — リードタイムを計測し、滞留を検知して改善する
4. **ポリシーを明示する** — 「完了の条件」「レーン間の移動条件」を明文化する
5. **フィードバックループを設ける** — 定例の振り返り・ボード見直し
6. **協調的に改善する** — 小さな変更を継続する

### カンバンボードの例

```
| バックログ | 準備完了 | 開発中 (WIP:3) | レビュー中 (WIP:2) | テスト (WIP:2) | 完了 |
|   □ □ □   |  □ □    |  ■ ■ ■        |  ■              |  ■ ■          | ■ ■ |
```

### WIP制限がなぜ効くのか

- 同時作業が多いほど**切り替えコスト**と**滞留**が増え、1件あたりの完了は遅くなる
- WIP制限があると、新しい仕事を「始める」より、詰まっている仕事を「終わらせる」ことが優先される（**"Stop Starting, Start Finishing"**）
- 詰まり（ボトルネック）がボード上で即座に見えるため、チームで助け合う行動が生まれる

### カンバンの主要メトリクス

| 指標 | 意味 |
|---|---|
| リードタイム | 依頼から完了までの時間 |
| サイクルタイム | 着手から完了までの時間 |
| スループット | 単位期間あたりの完了数 |
| 累積フロー図（CFD） | 各ステージの仕掛り量の推移。帯の膨らみ＝ボトルネック |

### スクラムとカンバンの使い分け

| 観点 | スクラム | カンバン |
|---|---|---|
| リズム | 固定スプリント | 連続フロー（区切りなし） |
| 計画 | スプリントごとにまとめて | 都度補充（プル型） |
| 変更 | スプリント内は原則固定 | いつでも優先順位変更可 |
| ロール | PO / SM / 開発者を定義 | 既存の役割のまま始められる |
| 向く仕事 | プロダクト開発（まとまった機能単位） | 運用・保守・問い合わせ対応など割り込みが多い仕事 |

**スクラムバン**: スクラムのイベント構造を保ちつつ、スプリント内の流れをカンバン（WIP制限・プル）で管理するハイブリッド。開発チームの実務では非常によく見られます。

## 2. XP（エクストリーム・プログラミング）

### 概要

Kent Beckが提唱した、**技術プラクティスに最も踏み込んだ**アジャイル手法です。「良いとされる実践（レビュー・テスト等）を極端（エクストリーム）なまでに徹底する」ことが名前の由来です。スクラムが「チームの回し方」を定義するのに対し、XPは「**日々のコードの書き方・品質の保ち方**」を定義します。

### 5つの価値

**コミュニケーション・シンプリシティ・フィードバック・勇気・リスペクト**

### 主要プラクティス

| 分類 | プラクティス | 内容 |
|---|---|---|
| 計画 | 計画ゲーム | ストーリー単位で見積り、優先順位はビジネス側が決める |
| 計画 | 小さなリリース | 短い間隔で頻繁にリリースする |
| 設計 | シンプルデザイン | 今必要な最小の設計にする（YAGNI: You Aren't Gonna Need It） |
| 設計 | メタファー | システムを共通の比喩で語れるようにする |
| 開発 | **テスト駆動開発（TDD）** | テストを先に書き、実装 → リファクタリングを小刻みに回す |
| 開発 | **リファクタリング** | 動作を変えずに内部構造を継続的に改善する |
| 開発 | **ペアプログラミング** | 2人1組で1つのコードを書く。レビューが常時行われる状態 |
| 開発 | **継続的インテグレーション（CI）** | 頻繁に統合し、常にビルド・テストが通る状態を保つ |
| 開発 | コードの共同所有 | 誰でもどのコードも直してよい |
| 開発 | コーディング規約 | チームで統一したスタイルで書く |
| チーム | 週40時間 | 持続可能なペース。恒常的な残業をしない |
| チーム | オンサイト顧客 | 質問にすぐ答えられる顧客（業務の分かる人）が常にそばにいる |

TDD・リファクタリング・CIなどの詳細は[⑤ 技術プラクティス](05-engineering-practices.md)で扱います。

### XPの現在地

「XPを丸ごと採用している」と名乗るチームは少ないものの、TDD・CI・リファクタリング・ペアプロといった**XP由来のプラクティスは現代の開発の標準装備**になっています。「プロセスはスクラム、エンジニアリングはXP」が実質的なデファクトです。

## 3. その他の手法・概念

### リーンソフトウェア開発

トヨタ生産方式の原則をソフトウェアに翻訳したもの（Mary & Tom Poppendieck）。7原則: **ムダをなくす / 品質を作り込む / 知識を作り出す / 決定を遅らせる / 速く提供する / 人を尊重する / 全体を最適化する**。「部分最適でなく全体のフローを見る」という視点はカンバンやDevOpsに通じます。

### リーンスタートアップ

Eric Riesが提唱。**構築（Build）→ 計測（Measure）→ 学習（Learn）**のループを最速で回し、**MVP（Minimum Viable Product: 実用最小限の製品）**で仮説を検証する。アジャイル開発が「作り方」なら、リーンスタートアップは「**何を作るべきかの検証方法**」で、両者は補完関係にあります。

### DevOps

開発（Dev）と運用（Ops）の壁を取り払い、**ビルド・テスト・デプロイ・監視までを自動化・高速化**する文化と実践の総称。アジャイルが開発チーム内のループを速くするのに対し、DevOpsは**リリースと運用まで含めたループ全体**を速くします。CI/CD（⑤章）はその中核です。

### FDD / Crystal / DSDM（歴史的手法）

- **FDD**（ユースケースより細かい「フィーチャー」単位で駆動）
- **Crystal**（チーム規模・重要度でプロセスの重さを変える手法ファミリー）
- **DSDM**（タイムボックスと優先度ルール MoSCoW を特徴とする英国発の手法）

いずれもマニフェスト起草者が関わった手法ですが、現在の採用例は少数派です。用語（MoSCoW法など）は今も使われます。

## 4. 手法の選び方（実務指針)

| 状況 | おすすめ |
|---|---|
| 新規プロダクトをチームで開発 | スクラム（＋XPの技術プラクティス） |
| 運用・保守・ヘルプデスクなど割り込み中心 | カンバン |
| 開発とサポートが混在 | スクラムバン、またはチームを分ける |
| 何を作るべきか自体が不明（0→1） | リーンスタートアップ＋軽量スクラム |
| まず現状を変えずに改善を始めたい | カンバン（現行プロセスの可視化から） |
