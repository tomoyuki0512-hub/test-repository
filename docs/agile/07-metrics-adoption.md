# ⑦ メトリクスと導入・アンチパターン

[← 総合インデックスに戻る](README.md)

## 1. アジャイルのメトリクス

### スプリントの計測

| 指標 | 内容 | 使い方 |
|---|---|---|
| ベロシティ | スプリントで完了したポイント合計 | チーム内の**予測**用。比較・評価には使わない |
| バーンダウンチャート | 残作業量の推移を右下がりの線で見る | スプリント内の進み具合の検知（理想線との乖離） |
| バーンアップチャート | 完了量とスコープ総量を2本の線で見る | スコープ追加が見えるため、リリース計画に向く |
| スプリントゴール達成率 | ゴールを達成できたスプリントの割合 | ポイント消化より本質的な成否の指標 |

### フローの計測（カンバン系）

| 指標 | 内容 |
|---|---|
| リードタイム / サイクルタイム | 依頼〜完了 / 着手〜完了までの時間。短縮がそのまま顧客価値 |
| スループット | 期間あたりの完了項目数 |
| 累積フロー図（CFD） | ステージ別仕掛りの推移。帯の膨らみでボトルネックを特定 |

### Four Keys（DORAメトリクス)

Google のDORA研究に基づく、開発組織のパフォーマンス指標として現在最も広く使われる4指標です。

| 指標 | 意味 | エリートの目安 |
|---|---|---|
| デプロイ頻度 | 本番リリースの頻度 | オンデマンド（1日複数回） |
| 変更のリードタイム | コミットから本番までの時間 | 1日未満 |
| 変更障害率 | デプロイが障害を起こす割合 | 0〜15% |
| サービス復元時間（MTTR） | 障害からの復旧時間 | 1時間未満 |

**速度と安定性はトレードオフではなく両立する**（速いチームほど壊れにくい）ことがこの研究の最重要の発見です。

### メトリクスの鉄則 — グッドハートの法則

> 指標が目標になると、それは良い指標であることをやめる

- ベロシティを目標にする → ポイントのインフレが起き、予測に使えなくなる
- カバレッジを目標にする → アサーションのない無意味なテストが増える
- 指標は**改善の会話の材料**であり、**個人・チームの評価に使った瞬間に壊れる**

## 2. よくあるアンチパターン

### プロセス面

| アンチパターン | 症状 | 処方箋 |
|---|---|---|
| **形だけスクラム（ゾンビスクラム）** | イベントは全部やるが、フィードバックも改善も起きない | 各イベントの「目的」に立ち返る。レビューで本当にフィードバックを得ているか点検 |
| **ミニウォーターフォール** | スプリント内が「設計週→実装週→テスト週」になっている | ストーリー単位で設計〜テストまで完結させる。DoDにテストを含める |
| **報告会化したデイリー** | マネージャーに向かって進捗報告する15分 | 開発者同士がゴール達成のために話す場に戻す。ボードを見ながら話す |
| **形骸化した振り返り** | 同じ問題が毎回出るが何も変わらない | Tryを1個に絞り、次スプリントのタスクとして積む。効果を次回確認 |
| **要求固定のままの導入** | 「アジャイルだが仕様変更は受け付けない」 | 変更の受け皿（バックログの並べ替え）が機能しているか点検 |
| **完了の水増し** | 「実装は終わった（テスト以外）」が完了扱い | DoDを明文化し、満たさないものはベロシティに数えない |

### 組織面

| アンチパターン | 症状 | 処方箋 |
|---|---|---|
| **ベロシティ評価** | チーム比較・人事評価にベロシティを使用 | フローの指標（リードタイム）やアウトカムで会話する |
| **PO不在・弱いPO** | 優先順位を決められない／ステークホルダーの伝書鳩 | 決定権を持てる人をPOにする。無理なら組織課題として扱う |
| **チームの頻繁な組み替え** | プロジェクトごとに解散・再結成 | 「プロジェクトに人を割り当てる」から「安定チームに仕事を流す」へ |
| **アジャイル導入自体が目的化** | 「導入完了」がゴールでその後の改善がない | 何のためか（リードタイム短縮？品質？）を指標で定義してから始める |
| **技術プラクティスなき導入** | スプリントを重ねるほど遅くなる | ⑤章参照。CI・自動テストへの投資をロードマップに組み込む |

## 3. 導入ステップ（実務ガイド)

### ステップ0: 目的を定義する

「なぜアジャイルにするのか」を計測可能な形で決めます（例:「機能リリースのリードタイムを3ヶ月→2週間に」「手戻りによる障害を半減」）。これがないと成功も失敗も判定できません。

### ステップ1: パイロットチームを1つ選ぶ

- 全社一斉ではなく、**1チームで小さく始める**（アジャイルの原則を導入自体にも適用する）
- 条件が良いチーム: 新規性のあるプロダクト／協力的なステークホルダー／メンバーが固定できる

### ステップ2: 型どおりに始める（守破離の「守」）

- まずスクラムガイド通りに3〜5スプリント回す。最初から独自アレンジ（イベント省略など）をしない
- 経験者（社内外のコーチ・スクラムマスター経験者）が1人いると立ち上がりが大きく違う

### ステップ3: 技術的土台を並行して整備する

- CIパイプライン、自動テスト、デプロイ自動化を最初の数スプリントの目標に含める
- レガシーコードしかない場合も「これから触る部分にテストを書く」から始める

### ステップ4: 振り返りで自分たちの型に育てる（「破」「離」）

- 3〜5スプリント回した後、レトロスペクティブを起点にプロセスを調整する
- 調整の判断基準は常に「目的（ステップ0）に近づくか」

### ステップ5: 展開する

- パイロットの学び（うまくいった点・組織側の障害）を踏まえて2チーム目以降へ
- 複数チームの調整が必要になったら⑥章（スケーリング）へ

### 契約・発注側の論点（日本の実務）

- 一括請負（仕様固定・完成責任）とアジャイルは構造的に相性が悪く、**準委任契約**が基本形とされる
- 参考資料: IPA「アジャイル開発版 情報システム・モデル取引・契約書」— 発注側・受注側の役割分担やPO の責務が整理されており、社内説得の材料としても有用
