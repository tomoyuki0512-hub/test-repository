# ⑭ レガシーコードとアジャイル — テストのない既存システムで戦う

[← 総合インデックスに戻る](README.md)

⑤章の技術プラクティスは「これから書くコード」には素直に適用できますが、現実の多くのチームは**テストのない巨大な既存コード**の上で仕事をしています。Michael Feathers の定義では「**レガシーコード＝テストのないコード**」。この章は、その現場でアジャイルを成立させる技術を扱います。

## 1. レガシーコードのジレンマ

> 安全に変更するにはテストが要る。
> テストを書くにはコードの変更（分離）が要る。
> しかし変更は危険だ。テストがないから。

この循環を破るのが本章の技術です。原則は「**完璧を目指さず、これから触る場所だけを少しずつ安全にする**」こと。全面書き直しでも全面テスト整備でもなく、**変更する箇所の周りにだけ足場を組む**アプローチです。

## 2. 仕様化テスト（Characterization Test）

レガシーコードにまず書くのは「正しさを確かめるテスト」ではなく、「**現在の振る舞いを記録するテスト**」です。

1. 対象コードをテストから呼べるようにする（最低限の分離）
2. 適当な入力で呼び、**実際に返ってきた値をそのまま期待値として**テストに書く
3. これを主要な入力パターン分繰り返し、現在の振る舞いの「網」を作る
4. この網に守られた状態でリファクタリング・機能変更を行う

- 「この挙動、実はバグでは？」と思っても、**まず現状を固定してから**直す（記録と修正を混ぜない）。既存の挙動に依存している利用者がいるかもしれない
- ゴールデンマスターテスト（大量の入出力をファイルに保存して差分比較する）は、出力が複雑な帳票・変換系で特に有効

## 3. 縫い目（Seam）を作る — テスト可能にする最小の手術

テストできない原因の多くは、**依存が固定されている**こと（DBに直接接続、現在時刻を直接取得、静的呼び出し等）。依存を差し替えられる点＝**縫い目（Seam）**を最小の変更で作ります。

| 技法 | 内容 |
|---|---|
| スプラウトメソッド/クラス | 新しいロジックを既存の巨大メソッドに**書き足さず**、新しいメソッド/クラスとして書き（ここはTDDできる）、既存側からは1行呼ぶだけにする |
| ラップメソッド | 既存メソッドを新メソッドで包み、前後に新処理を差し込む |
| 依存の抽出 | `new Database()` や `DateTime.now()` を引数・コンストラクタ注入に変え、テストでは偽物（スタブ）を渡す |
| Extract & Override | テストでだけ問題の依存部分をサブクラスで上書きする（過渡期の技として割り切る） |

**ポイント**: 理想の設計への大改造を一気にやらない。「今日の変更に必要な縫い目」だけを作る。縫い目は積み重なって、数ヶ月後には自然にテスト可能な設計へ近づいていきます。

## 4. ストラングラーフィグパターン — 書き直しの現実解

### ビッグバン書き直しはなぜ失敗するか

- 書き直し中も旧システムは変化し続け、**追いかける仕様が動く的**になる
- 数年間ビジネス価値が出ない期間が生じ、途中で予算が尽きる
- 旧システムの暗黙仕様（例外処理・帳尻合わせ）を網羅的に把握している人はいない

### ストラングラーフィグ（絞め殺しの木）

Martin Fowler が紹介した、**旧システムを稼働させたまま、機能単位で少しずつ新実装に置き換える**戦略。

```
リクエスト → [ルーティング層/ファサード]
                ├─→ 新システム（置き換え済みの機能: 例 検索、通知）
                └─→ 旧システム（まだの機能: それ以外すべて）
                        ↑ スプリントごとに新側の担当範囲を広げていく
```

- 入口に**ルーティング層**（プロキシ・API ゲートウェイ・BFF）を置き、機能ごとに新旧を振り分ける
- **常に本番で動く**ため、スプリントごとに価値と学習が得られる（アジャイルの原則そのまま）
- 移行の各段階で**新旧の出力を突き合わせる**（シャドーイング: 新側にも同じリクエストを流し、結果を比較だけして捨てる）と安全に切り替えられる
- データベースの分離が最難関になりやすい。まずコードだけ新しくしDBは共有 → 後からデータを分離、の2段階が現実的

## 5. リファクタリングの投資判断

レガシー改善は「全部きれいにする」ことではありません。**変更頻度×痛みの大きい場所**に投資を集中します。

- **ホットスポット分析**: バージョン管理の履歴から「変更頻度が高く」かつ「複雑度が高い」ファイルを特定する（`git log` の変更回数集計だけでも十分見える）。上位1割のファイルが痛みの大半を生んでいることが多い
- 変更されないコードは、汚くても**触らない**（動いているものに戦いを挑まない）
- 改善の時間確保のパターン: ①ストーリーの見積りに織り込む（デフォルト・交渉不要）②スプリント容量の10〜20%を改善枠として合意 ③大物はスパイク/専用ストーリーとしてPOと交渉（⑤章の技術的負債の見える化とセット）

## 6. レガシー環境での漸進的なプラクティス導入

いきなり全プラクティスは無理でも、順序を工夫すれば数ヶ月で景色が変わります。

| 順番 | 施策 | 理由 |
|---|---|---|
| 1 | ビルドとデプロイの自動化・再現可能化 | すべての土台。「リリース手順書＋手作業」の排除が最優先 |
| 2 | CI に静的解析・既存テストを常設 | 現状レベルの劣化を止める（ラチェット: 今より悪化したら失敗にする） |
| 3 | **新規・変更コードにだけ**テスト必須のルール | 全体カバレッジ目標より現実的で、触る場所から自然に安全になる |
| 4 | 仕様化テスト＋縫い目でホットスポットを攻略 | 痛い場所に集中投資 |
| 5 | フィーチャーフラグ・段階的リリース | 変更の爆発半径を制御し、リリース恐怖症を解消 |
| 6 | （必要なら）ストラングラーフィグで構造置換 | 1〜5で延命できないアーキテクチャ限界が明確になってから |

**アンチパターン**: 「カバレッジ80%目標」を全コードに課す（無意味なテスト量産で応える文化が生まれる）。「リファクタリングスプリント」を設けて通常スプリントでは負債を無視する（改善が日常から切り離され、結局後回しになる）。
