# ② スクラム

[← 総合インデックスに戻る](README.md)

## 1. スクラムとは

スクラムは、**複雑な問題に対してチームが価値を漸進的に届けるための軽量フレームワーク**で、アジャイル手法の中で世界的に最も普及しています。公式ルールは「[スクラムガイド](https://scrumguides.org/)」（Ken Schwaber / Jeff Sutherland著、無料公開・日本語版あり）にまとまっており、全部で十数ページしかありません。

スクラムの構造は「**3つのロール（責任）× 5つのイベント × 3つの作成物**」で覚えるのが定番です。

```
┌──────────────────────── スプリント（1〜4週間の固定期間）───────────────────────┐
│                                                                              │
│  スプリント          毎日:                スプリント        スプリント          │
│  プランニング   →    デイリースクラム  →   レビュー     →    レトロスペクティブ  │
│  (何を・どう作るか)   (15分の同期)         (成果を確認)      (やり方を改善)      │
│                                                                              │
└──────────────────────────────── これを繰り返す ───────────────────────────────┘
```

## 2. 3つのロール（責任）

### プロダクトオーナー（PO）— 「何を作るか」に責任を持つ

- プロダクトの**価値を最大化**することに責任を持つ
- **プロダクトバックログの管理**（項目の作成・明確化・並び替え）の最終責任者
- ステークホルダーの要望を聞くが、優先順位の**決定権はPOひとり**に集約する（合議にしない）
- 1プロダクトにつき1人。委員会ではなく個人

### 開発者（Developers）— 「どう作るか」に責任を持つ

- スプリントごとに**利用可能なインクリメント（動く成果物）**を作る
- 見積り・タスク分解・技術的な進め方は**開発者が自律的に決める**（自己管理）
- 設計・実装・テストなどプロダクトを作る作業を行う人は肩書きに関係なく全員「開発者」
- 目安は10人以下のチーム

### スクラムマスター（SM）— 「スクラムが機能すること」に責任を持つ

- スクラムの理論とプラクティスを**チームと組織に定着**させる
- 進捗管理者・指示者**ではない**。障害物（ブロッカー）の除去、会議のファシリテート、組織への働きかけを行う**サーバントリーダー**
- POの支援（バックログ管理の技法を教える等）も行う

> **注意**: SMとPOの兼任は利益相反（価値最大化 vs プロセス健全性）になりやすく非推奨。POと上司の兼任も「上司の指示」が優先順位を歪めるため注意。

## 3. 5つのイベント

すべてのイベントは**タイムボックス**（上限時間）が決まっています。以下は2週間スプリントの場合の目安です。

| イベント | タイムボックス | 目的 |
|---|---|---|
| スプリント | 1〜4週間（固定） | 他のイベントの入れ物。期間は毎回変えない |
| スプリントプランニング | 最大4時間 | 今回のゴールと作る項目、進め方を決める |
| デイリースクラム | 毎日15分 | ゴールへの進捗を検査し、その日の計画を調整 |
| スプリントレビュー | 最大2時間 | 成果物をステークホルダーと検査し、次に向けた対話をする |
| スプリントレトロスペクティブ | 最大1.5時間 | チームのやり方を検査し、改善を決める |

### スプリントプランニング

3つの問いに答えます。

1. **Why**: このスプリントはなぜ価値があるか → **スプリントゴール**を1文で決める
2. **What**: 何ができるか → バックログ上位からベロシティ（実績値）を目安に選択する
3. **How**: どうやるか → 選んだ項目をタスクに分解する

### デイリースクラム

- 開発者のための**同期と再計画の場**。上司への報告会ではない
- 形式は自由だが定番は「昨日やったこと / 今日やること / 困っていること（ブロッカー）」
- 問題の**解決**はその場でやらず、必要な人だけ残って15分の外で行う

### スプリントレビュー

- **動くソフトウェアをデモ**し、ステークホルダーからフィードバックを得る
- 単なる進捗報告会ではなく、**バックログを見直す対話の場**
- ここで出た意見は次のスプリントの計画材料になる

### スプリントレトロスペクティブ（振り返り）

- 成果物ではなく**やり方（プロセス・人間関係・ツール）**を振り返る
- 定番フォーマット: **KPT**（Keep / Problem / Try）、**Fun-Done-Learn**、**Start-Stop-Continue** など
- 「Tryを1〜2個に絞り、次スプリントで必ず実行する」ことが形骸化を防ぐコツ

## 4. 3つの作成物と「確約（コミットメント）」

| 作成物 | 内容 | 対応する確約 |
|---|---|---|
| プロダクトバックログ | プロダクトで実現したいことの優先順位付き一覧 | **プロダクトゴール**（プロダクトの長期目標） |
| スプリントバックログ | 今スプリントで行う項目＋実行計画 | **スプリントゴール**（今スプリントの目的1文） |
| インクリメント | スプリントで完成した利用可能な成果物 | **完成の定義（DoD）** |

### 完成の定義（Definition of Done, DoD）

「完了しました」の基準をチームで合意したチェックリスト。例:

- [ ] コードレビュー済み
- [ ] 自動テストが追加され、全テストがパス
- [ ] ステージング環境で動作確認済み
- [ ] ドキュメント（必要分）更新済み

DoDが曖昧だと「実装は終わったがテストしていない」ものが「完了90%」として積み上がり、進捗が信用できなくなります。

## 5. 1スプリントの流れ（2週間の例）

| 日 | 午前 | 午後 |
|---|---|---|
| 月(1日目) | スプリントプランニング | 開発開始 |
| 火〜金 | デイリースクラム(15分)→開発 | 開発（随時バックログリファインメント） |
| 翌月〜木 | デイリースクラム→開発 | 開発 |
| 金(10日目) | スプリントレビュー | レトロスペクティブ → 終了 |

**バックログリファインメント**（項目の詳細化・分割・見積り直し）は公式イベントではありませんが、次スプリントの準備として週1回程度、定例で行うチームが多いです。

## 6. スクラムの理論的土台

- **経験主義（経験的プロセス制御）**: 複雑な仕事は事前に全て計画できない。**透明性 → 検査 → 適応**のループで制御する
  - 透明性: 状況を見えるようにする（バックログ、ボード、動くデモ）
  - 検査: 定期的に確かめる（各イベントがその機会）
  - 適応: ズレていたら直す（計画・プロセスの修正）
- **リーン思考**: ムダを省き、本質に集中する
- **スクラムの5つの価値基準**: 確約・集中・公開・尊敬・勇気

## 7. よくある質問

**Q. スプリント中に急な依頼が来たら？**
原則はスプリントゴールを守り、次スプリントで対応。ただし障害対応など本当に緊急なものは、POと相談して項目を入れ替える。頻発するなら「割り込み用バッファ」を計画に組み込むか、カンバン（③章）の併用を検討する。

**Q. スプリントの長さは？**
迷ったら**2週間**が最も一般的。不確実性が高い初期は1週間で学習ループを速くする選択もある。

**Q. スクラムマスターは専任が必要？**
理想は専任。現実には開発者との兼任も多いが、その場合SMの仕事（障害除去・組織への働きかけ）が後回しになりがちな点を自覚しておく。

**Q. スクラムをやればアジャイルになる？**
ならない。イベントをこなすだけの「形だけスクラム」は⑦章のアンチパターン参照。技術プラクティス（⑤章）が伴わないと、スプリントを重ねるほど品質が劣化する。
