# Flutter 月額サブスクリプション 総合ガイド（はじめに）

このドキュメント群は、**Flutterアプリで月額サブスクリプション（自動更新課金）を開発・運用する**ために必要な知識を、事業部・開発部隊・PM の誰もが理解できるようにまとめたものです。

- 対象プラットフォーム: **iOS（App Store）/ Android（Google Play）**
- 課金方式: **App内課金（IAP）による自動更新サブスクリプション（月額中心）**
- Flutterパッケージ: **`in_app_purchase`（Flutter公式）**
- バックエンド: **自前実装**（レシート検証・権利管理を自社で行う）

> このプロジェクトでは **RevenueCat（サードパーティの課金SaaS）は使用しません。** 検証・権利管理はすべて自前で実装します。

## 2. サブスクリプションの全体像（1枚サマリ）

```
                ┌───────────────────────────────────────────────┐
                │                  ユーザー                       │
                └───────────────────────────────────────────────┘
                      │ アプリ内で「購入」ボタンをタップ
                      ▼
        ┌─────────────────────────────┐
        │  Flutterアプリ（in_app_purchase）│  ① 商品情報の取得・購入リクエスト
        └─────────────────────────────┘
                      │ 決済はストアが代行（開発者はカード情報を扱わない）
                      ▼
        ┌─────────────────────────────┐
        │  App Store / Google Play       │  ② 決済・認証・自動更新の管理
        └─────────────────────────────┘
            │ レシート/トークン           │ 状態変化を通知（Webhook）
            ▼                             ▼
        ┌──────────────┐        ┌──────────────────────────┐
        │ 自前バックエンド │◀──────│ ストアからのサーバー通知       │
        │（検証・権利管理）│        │ iOS: App Store Server Notif. │
        └──────────────┘        │ Android: Google Play (Pub/Sub)│
            │ DBが「正」              └──────────────────────────┘
            ▼
        ユーザーに機能・コンテンツへのアクセス権を付与
```

**3つの登場人物**

1. **アプリ（Flutter）** — 商品表示・購入操作・購入完了の確定。
2. **ストア（Apple / Google）** — 決済・自動更新・解約・返金を代行。手数料を徴収。
3. **自前バックエンド** — レシートを検証し、誰がいつまで使えるか（権利＝entitlement）をDBで管理する「正（Single Source of Truth）」。

## 3. 用語集（共通言語）

事業・開発で会話がかみ合うように、頻出用語をまとめます。

### 課金の基本

| 用語 | 読み / 英語 | 意味 |
|---|---|---|
| IAP | アイエーピー / In-App Purchase | アプリ内課金。ストアの決済システムを通じてデジタル商品を販売する仕組み |
| 自動更新サブスクリプション | Auto-renewable subscription | 期間終了時に自動で課金・更新される定期購入。月額・年額が代表例 |
| プロダクトID | Product ID | ストアに登録する商品の識別子（例: `com.example.app.monthly`） |
| ベースプラン / オファー | Base Plan / Offer | Android（Billing Library 5以降）の構造。商品の基本料金（ベースプラン）に割引や無料期間（オファー）を付ける |
| サブスクリプショングループ | Subscription Group | iOSで「同一サービスの複数プラン」をまとめる単位。グループ内は同時に1つだけ有効 |
| エンタイトルメント | Entitlement / 権利 | 「そのユーザーがどの機能を使えるか」を表す概念。課金状態と機能アクセスを橋渡しする |

### 料金・特典

| 用語 | 意味 |
|---|---|
| フリートライアル | 無料お試し期間。終了後に通常料金で課金開始 |
| イントロダクトリーオファー | 新規ユーザー向けの初回割引（前払い割引・継続割引・無料期間など） |
| プロモーションオファー | 既存・解約ユーザー向けの特別割引 |
| 手数料（コミッション） | ストアが売上から徴収する割合（Apple 15〜30% / Google 10〜20%＋α） |
| Small Business Program | Appleの中小事業者向け優遇。年間売上100万ドル以下で手数料15% |

### ライフサイクル・運用

| 用語 | 意味 |
|---|---|
| レシート検証 | 購入情報が本物かをストアの公式APIで確認すること。不正利用防止に必須 |
| Webhook / サーバー通知 | 更新・解約・課金失敗などの状態変化をストアがバックエンドに通知する仕組み |
| Grace Period（猶予期間） | 課金失敗後も一定期間アクセスを維持する期間。解約防止に重要 |
| Billing Retry（iOS）/ Account Hold（Android） | 猶予期間後もアクセス停止しつつストアが課金を再試行する期間 |
| チャーン（解約率） | 一定期間に解約したユーザーの割合。サブスク事業の最重要指標のひとつ |
| MRR / ARR | 月次／年次の経常収益（Monthly/Annual Recurring Revenue） |
| LTV | 顧客生涯価値（Life Time Value）。1顧客が解約までに生む累計売上 |
| restorePurchases | 「購入を復元」。機種変更・再インストール時に権利を取り戻す操作（iOS審査要件） |

## 4. このプロジェクトの基本方針（要点先取り）

詳細は各ドキュメントに譲りますが、意思決定の前提を先に共有します。

| 項目 | 方針 | 理由 |
|---|---|---|
| 課金方式 | IAP（自動更新サブスク・月額中心） | ストアポリシー上、デジタル機能の継続提供はIAPがほぼ必須 |
| パッケージ | `in_app_purchase`（公式） | 外部SaaS依存なし・ランニングコスト0・自前バックエンドと相性が良い |
| 権利管理 | 自前バックエンド＋DBを「正」とする | 複数デバイス同期・不正防止・独自ロジックに対応するため |
| 状態同期 | ストアのサーバー通知（Webhook）＋日次バッチ | リアルタイム反映と取りこぼし防止の両立 |
| RevenueCat | 使用しない | 自前で検証・権利管理を行う方針のため |
