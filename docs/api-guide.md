# APIの種類・特徴・歴史 完全ガイド

---

## APIとは何か — レストランのたとえ

- **あなた**（お客さん） = アプリやWebサービスを使う側
- **厨房** = データを持っているサーバー
- **ウェイター** = **API**

あなたは厨房に直接入れない。でもウェイター（API）に「ハンバーグ定食をください」と言えば、厨房に伝えて料理を持ってきてくれる。**APIはアプリ同士の「取次役」。**

天気予報アプリは気象庁のサーバーにAPIで「今日の東京の天気は？」と聞いて、答えを受け取って画面に表示している。

## 時代の流れ — 年表

```
1990年代
  ├─ 1991 ─ World Wide Webの誕生
  ├─ 1998 ─ XMLが標準化
  └─ 1999 ─ SOAPが登場（企業間の堅いAPI）

2000年代
  ├─ 2000 ─ RESTの概念登場（Roy Fieldingの博士論文）
  ├─ 2004 ─ Facebookが誕生（API連携の需要が爆発）
  ├─ 2006 ─ AWS・Twitter APIが公開（APIビジネスモデルの確立）
  └─ 2008 ─ iPhone普及（モバイル＋APIが当たり前に）

2010年代
  ├─ 2011 ─ WebSocket標準化（リアルタイム通信が普及）
  ├─ 2012 ─ REST APIが業界標準に定着
  ├─ 2015 ─ GraphQLをFacebookが公開
  ├─ 2016 ─ gRPCをGoogleが公開（マイクロサービス時代）
  └─ 2018 ─ API経済が本格化（APIを製品として販売）

2020年代
  ├─ 2020 ─ コロナでデジタル化が加速
  ├─ 2021 ─ tRPC登場（TypeScript型安全RPC）
  ├─ 2022 ─ ChatGPT登場、AI APIの時代が始まる
  ├─ 2024 ─ MCP（Model Context Protocol）登場
  └─ 現在  ─ REST・GraphQL・gRPC・WebSocket・SSEを用途で使い分ける成熟期
```

## 過去の主流

### SOAP — 役所の公文書のやり取り

**登場：** 1999年（Microsoft・IBM・W3Cが策定）

役所に書類を提出するとき、決まった様式・決まった順番・判子が必要……あの感じ。

```xml
<!-- SOAPのリクエスト例（こんなに書く必要がある）-->
<soap:Envelope>
  <soap:Body>
    <GetWeather>
      <City>Tokyo</City>
    </GetWeather>
  </soap:Body>
</soap:Envelope>
```

| 項目 | 内容 |
|---|---|
| データ形式 | XML（タグで囲む厳格な形式） |
| 長所 | セキュリティ高・エラー処理厳密・トランザクション対応 |
| 短所 | 書くのが面倒・データが重い・学習コスト高 |
| 向いている場面 | 銀行・医療・政府系の厳格なシステム |

**なぜ廃れたか：** XMLのペイロードが肥大化・仕様が複雑すぎ・Web/モバイル時代にRESTの軽量さが圧倒。現在はレガシー基幹システムの保守でのみ残存。

---

### XML-RPC・CORBA

- **XML-RPC（1998年）：** SOAPの前身。XMLでメソッドを呼び出す。SOAPに取って代わられた。
- **CORBA（1991年）：** オブジェクト指向の分散通信。実装間の互換性問題が多発し衰退。概念はgRPCに受け継がれた。

---

## 現在の主流

### REST — 普通の会話

**登場：** 2000年（Roy Fieldingの博士論文で提唱）

友達に「今日の東京の天気教えて」と自然に聞く感じ。決まりきった書式は不要。

```
// リクエスト（シンプル）
GET https://api.example.com/weather/tokyo

// レスポンス
{ "city": "Tokyo", "temp": 22, "weather": "晴れ" }
```

**HTTPメソッドの意味：**

| メソッド | 操作 | 例 |
|---|---|---|
| GET | 取得する | ユーザー情報を見る |
| POST | 新しく作る | 投稿を作成する |
| PUT | 全体を更新する | プロフィールを書き換える |
| PATCH | 一部を更新する | 名前だけ変える |
| DELETE | 削除する | 投稿を消す |

| 項目 | 内容 |
|---|---|
| データ形式 | JSON（読みやすい） |
| 長所 | シンプル・学習コスト低・エコシステム最大・CDNキャッシュと相性◎ |
| 短所 | Over/Under-fetching（不要なデータが来る or 足りない）・リアルタイムが苦手 |
| 向いている場面 | Webサービス全般・スマホアプリ・公開API |
| 採用例 | GitHub・Twitter・AWS・ほぼ全てのWebサービス |

---

### GraphQL — 欲しいものだけのビュッフェ

**登場：** 2015年（Facebookが公開）

ランチセットだと「サラダ・メイン・スープ・デザート」が全部来るが、メインだけ欲しい。GraphQLはビュッフェ形式で欲しいものだけ取れる。

```graphql
# 「名前」と「フォロワー数」だけが欲しい場合
query {
  user(id: "123") {
    name
    followersCount
  }
}
```

単一エンドポイント（通常 `POST /graphql`）で、クライアントがフィールドを指定する。

| 項目 | 内容 |
|---|---|
| データ形式 | JSON |
| 長所 | データの過不足なし・1回で複数リソース取得・型安全・コード自動生成 |
| 短所 | キャッシュが難しい・N+1問題が出やすい・小規模には過剰 |
| 向いている場面 | 複雑なデータ構造・SNS・BFFパターン |
| 採用例 | GitHub・Shopify・Netflix・Facebook |

---

### gRPC — 工場間の高速コンベア

**登場：** 2016年（Googleが公開）

工場Aと工場Bが生産ラインで直結され、人の手を介さず部品が高速で自動的に流れるイメージ。人間が読む必要はなく、速さと効率が優先。

```protobuf
// Protocol Buffers（.protoファイル）でスキーマを定義
service WeatherService {
  rpc GetWeather (WeatherRequest) returns (WeatherResponse);
  rpc StreamWeather (WeatherRequest) returns (stream WeatherResponse);
}
```

`.proto` ファイルから複数言語のクライアント/サーバーコードが自動生成される。

**通信パターン：**

| パターン | 説明 |
|---|---|
| Unary | 1リクエスト → 1レスポンス（通常の通信） |
| Server Streaming | 1リクエスト → 複数レスポンス（データを少しずつ返す） |
| Client Streaming | 複数リクエスト → 1レスポンス |
| Bidirectional | 複数リクエスト ↔ 複数レスポンス |

| 項目 | 内容 |
|---|---|
| データ形式 | Protocol Buffers（バイナリ）。JSONより3〜10倍小さく高速 |
| 通信 | HTTP/2 |
| 長所 | 超高速・型安全・多言語コード自動生成・ストリーミング標準対応 |
| 短所 | ブラウザから直接使いにくい・バイナリは人間が読めない |
| 向いている場面 | マイクロサービス間通信・内部API |
| 採用例 | Google・Square・Lyft・Dropbox |

---

### WebSocket — 電話（つなぎっぱなし）

**登場：** 2011年（RFC 6455として標準化）

普通のAPI通信は「手紙のやり取り」（送る→待つ→返事が来る）。WebSocketは「電話」。一度つながったら、どちらからでも、いつでも話しかけられる。

```
クライアント ──接続要求──▶ サーバー
クライアント ◀──承認────── サーバー
クライアント ◀──▶ サーバー（以降、双方向にリアルタイム通信）
```

| 項目 | 内容 |
|---|---|
| データ形式 | テキスト・バイナリ両対応 |
| 長所 | サーバーからもプッシュできる・低遅延・HTTPオーバーヘッドなし |
| 短所 | 接続維持でサーバー負荷が高い |
| 向いている場面 | チャット・オンラインゲーム・株価のリアルタイム表示 |
| 採用例 | Discord・Slack・Figma・Binance |

---

### SSE（Server-Sent Events）— ラジオ放送（一方向）

**登場：** HTML5仕様として策定

ラジオは放送局（サーバー）が一方的に電波を流し、リスナー（クライアント）は受け取るだけ。クライアントからは話しかけられない。

```
// サーバーが送るデータの形式
data: {"token": "こんにちは"}\n\n
data: {"token": "、今日の"}\n\n
data: {"token": "天気は晴れです"}\n\n
data: [DONE]\n\n
```

> **ChatGPTやClaudeが文字を少しずつ表示するのはSSEの仕組み。** LLMがトークンを生成するたびにSSEで送信している。

| 項目 | 内容 |
|---|---|
| データ形式 | テキスト |
| 長所 | 実装が簡単・通常のHTTPで動く・自動再接続が標準内蔵 |
| 短所 | サーバー→クライアントの一方向のみ |
| 向いている場面 | AIストリーミング・通知・ニュース速報 |
| 採用例 | OpenAI API・Anthropic API・各LLM API |

---

## 未来・今後のトレンド

### tRPC — TypeScript型安全RPC

**登場：** 2021年

TypeScriptを使う前提で、スキーマ定義ファイル不要。サーバー関数の型がそのままクライアントに伝わる。

```typescript
// サーバーの定義がそのままクライアントの型になる
const user = await trpc.user.getById.query({ id: "1" });
//                                          ^ 型が自動推論される
```

- **向いている場面：** Next.js・Remixなどのフルスタックアプリ
- **弱点：** TypeScript以外の言語には対応しない

---

### AsyncAPI — イベント駆動API仕様

**登場：** 2017年

RESTのOpenAPI（Swagger）に相当する、**非同期・イベント駆動APIの仕様記述標準**。Kafka・MQTT・WebSocketなどのメッセージブローカーを定義対象とする。

- IoT・金融系リアルタイムシステム・マイクロサービスで採用拡大中

---

### WebTransport — HTTP/3ベースの次世代通信

**登場：** 2022年（Chrome 97〜）

WebSocketの後継として設計。HTTP/3（QUIC）上で動作し、UDP的な低遅延通信とTCP的な信頼性を両立。

- **用途：** オンラインゲーム・ライブ映像配信・VR/AR
- **現状：** Chrome・Edgeは対応済み。Firefox・Safariは実装中。本番事例はまだ少ない。

---

### AI/LLM API — 台頭する新しい軸

ChatGPT以降、AIをAPIとして呼び出す設計が急拡大。

| 機能 | 概要 |
|---|---|
| **ストリーミング** | SSEでトークンを逐次返却。ChatGPT・Claude・Geminiが採用 |
| **Function Calling** | モデルが外部関数を呼び出すようJSON形式で宣言。エージェント実装に使用 |
| **Structured Output** | JSON Schemaを指定してレスポンスを構造化 |
| **Embeddings API** | テキストをベクトル化。類似検索・RAGに使用 |
| **MCP（Model Context Protocol）** | Anthropicが2024年に提唱。LLMがツール・リソースにアクセスするための標準プロトコル |

---

### Edge API — ユーザーの近くで処理する

Cloudflare Workers・Vercel Edge Functionsなど、CDNのエッジノード上でAPIロジックを実行。

- コールドスタートがほぼゼロ
- グローバル分散で低レイテンシ
- **用途：** A/Bテスト・認証ゲートウェイ・AI推論（軽量モデル）

## どれを選ぶべきか

```
リアルタイム通信が必要？
  ├─ YES ─ 双方向（サーバー↔クライアント）？
  │           ├─ YES → WebSocket（チャット・ゲーム）
  │           └─ NO  → SSE（通知・AIストリーミング）
  │
  └─ NO ─ サービス間の内部通信（高速さ重視）？
              ├─ YES → gRPC（マイクロサービス）
              └─ NO ─ データ構造が複雑？
                          ├─ YES → GraphQL（SNS・複雑なアプリ）
                          └─ NO ─ 銀行・医療・官公庁？
                                      ├─ YES → SOAP
                                      └─ NO  → REST（Webサービス全般）
```

> **迷ったらRESTから始める。** 世界中のほぼ全てのWebサービスで使われており、ドキュメントも豊富。特別な要件が出てきたときに他の方式を検討するのが現実的。

## APIを選んだ後の次のアクション

### 1. REST APIを選んだ後にやること

#### フレームワーク・ライブラリ選択

言語ごとの主要な選択肢を以下に示す。プロジェクトの規模・チームの習熟度・エコシステムの充実度を基準に選ぶこと。

| 言語 | 軽量・高速重視 | フルスタック・機能豊富 | 備考 |
|------|--------------|----------------------|------|
| Node.js | Express, Fastify | NestJS | Expressは学習コスト低、NestJSはDI・デコレータで大規模向き |
| Python | FastAPI, Flask | Django REST Framework | FastAPIはOpenAPI自動生成・型安全が強み |
| Go | Gin, Echo | (フレームワーク自体が軽量) | Ginが最もシェア高い。Chi も人気 |
| Java | Spring Boot (WebFlux) | Spring Boot (MVC) | MVC が安定、WebFlux はリアクティブ |
| Ruby | Sinatra | Ruby on Rails (API mode) | Rails は `--api` フラグで軽量化可能 |
| PHP | Slim | Laravel | Laravel Sanctum/Passport で認証も容易 |
| Rust | Actix-web, Axum | — | 最高レベルのスループットが必要な場合 |

**推奨の選び方:**
- チームが初めてなら **Express (Node.js)** または **FastAPI (Python)** ── ドキュメントが豊富で、サンプルが多い
- 型安全とスキーマ駆動を重視するなら **FastAPI** (Pydantic + OpenAPI 自動生成)
- 大規模エンタープライズなら **Spring Boot** (成熟したエコシステム、バッチ・セキュリティ統合)

#### チーム構成・スキル要件

**最小構成（小規模 API、2〜3名）:**
- バックエンドエンジニア × 1〜2（API 設計・実装・テスト）
- インフラ／DevOps × 0.5（兼任可）

**標準構成（中規模、5〜8名）:**
- バックエンドエンジニア × 3〜4
- API デザイン担当（または Tech Lead）× 1 ── URI 設計・バージョニング戦略を統一
- QA エンジニア × 1
- インフラ／SRE × 1

**必須スキル:**
- HTTP/1.1・HTTP/2 の基礎（メソッド、ステータスコード、ヘッダー）
- RESTful 設計原則（リソース指向、冪等性、ステートレス）
- OpenAPI (Swagger) によるスキーマ定義
- 認証・認可（JWT、OAuth 2.0、API キー）
- 基本的なデータベース操作（SQL または NoSQL）

**あると望ましいスキル:**
- キャッシュ設計（HTTP キャッシュヘッダー、Redis）
- レート制限・スロットリングの実装経験
- コントラクトテスト（Pact など）

#### 開発工数の目安

以下は **CRUD エンドポイント 1リソース（例: `/users`）あたり** の目安。チームの習熟度・要件定義の完成度で大きく変わる。

| フェーズ | 工数（人日） | 内容 |
|---------|------------|------|
| API 設計・OpenAPI 定義 | 0.5〜1 | パス・メソッド・レスポンス設計 |
| 実装（ハンドラ + バリデーション） | 1〜2 | ビジネスロジック除く |
| 認証・認可の組み込み | 2〜4（初回のみ） | JWT 検証、ロール制御 |
| テスト（単体 + 統合） | 1〜2 | ハッピーパス + エラー系 |
| ドキュメント整備 | 0.5 | Swagger UI 確認・補足記述 |

**プロジェクト全体の目安:**
- MVP（10〜15エンドポイント）: **4〜8週間**（2名）
- 中規模 API（50エンドポイント前後）: **3〜5ヶ月**（4〜5名）
- 大規模（100エンドポイント超・認証・管理画面連携）: **6ヶ月〜**

> バージョン管理（`/v1/`, `/v2/`）の戦略を最初に決めておかないと、後から修正コストが跳ね上がる。

#### 性能特性とインフラ設計

**スループット目安（参考値、シングルインスタンス）:**

| フレームワーク | RPS（Hello World 相当） | メモリ使用量 |
|--------------|----------------------|------------|
| Gin (Go) | 100,000〜200,000 | 〜10MB |
| Fastify (Node.js) | 70,000〜100,000 | 〜50MB |
| FastAPI (Python, async) | 20,000〜40,000 | 〜80MB |
| Spring Boot (WebFlux) | 30,000〜60,000 | 〜300MB |
| Express (Node.js) | 30,000〜60,000 | 〜60MB |

※実際のビジネスロジック・DB クエリが入ると数分の一に下がる。ベンチマークは参考程度に。

**インフラ設計のポイント:**
- **水平スケール前提**: ステートレス設計（セッションをサーバー側に持たない）にして、ロードバランサー背後に複数インスタンスを配置
- **HTTP キャッシュ活用**: `GET` レスポンスに `Cache-Control`, `ETag`, `Last-Modified` を付与し、CDN やリバースプロキシ（Nginx/Varnish）でキャッシュ
- **コネクションプール**: DB 接続はプール管理（PgBouncer for PostgreSQL など）
- **タイムアウト設計**: クライアント側・サーバー側・DB クエリそれぞれにタイムアウトを設定（例: API 全体 30s、DB クエリ 5s）
- **レート制限**: Redis + Sliding Window アルゴリズムで IP またはトークン単位で制限（例: 1,000 req/min）

**推奨構成（AWS 例）:**
```
クライアント → CloudFront (CDN) → ALB → ECS/EKS (API コンテナ) → RDS / ElastiCache
```

#### コスト

**開発コスト（概算）:**
- 小規模（エンジニア 2名 × 3ヶ月）: 人件費 300〜600万円
- 中規模（4名 × 4ヶ月）: 800〜1,600万円

**インフラコスト（AWS、月額概算）:**

| 規模 | 構成例 | 月額目安 |
|------|-------|---------|
| 小規模（〜10万 req/日） | ECS Fargate × 2, RDS t3.small | 3〜5万円 |
| 中規模（〜100万 req/日） | ECS × 4〜8, RDS t3.medium, ElastiCache | 10〜20万円 |
| 大規模（〜1,000万 req/日） | EKS + Auto Scaling, RDS Multi-AZ, CloudFront | 50万円〜 |

**コスト最適化の観点:**
- REST はレスポンスのキャッシュが効きやすいため、CDN 活用でオリジンへのリクエスト数を削減できる
- 静的・変化の少ないリソースは `Cache-Control: max-age=3600` 程度を設定するだけでコストを大幅に下げられる

#### 注意点・落とし穴

1. **URI 設計の一貫性がなくなる** — 動詞を混在させない（`/getUser` ではなく `/users/{id}`）。最初に命名規約ドキュメントを作りレビューチェックリストに含める
2. **バージョニング戦略を後回しにする** — URI バージョニング（`/v1/`）か Header バージョニングかを開発開始前に決定する
3. **エラーレスポンスの形式が統一されていない** — 全エンドポイントで共通のエラーオブジェクト構造を定義する（RFC 7807 参照）
4. **N+1 問題（DB クエリ）** — ORM の `include`/`eager loading` を活用
5. **過剰なデータ返却（Over-fetching）** — `?fields=id,name,email` のスパースフィールドセットや用途別エンドポイントを検討
6. **認証トークンの失効処理を設計しない** — JWT のブラックリスト処理が必要な場合は Redis で管理する仕組みを最初から組む

#### 最初にやること（チェックリスト）

```
[ ] OpenAPI 3.x でリソース・エンドポイント一覧を定義する（コーディング前）
[ ] URI 命名規約・バージョニング戦略をチームで合意してドキュメント化する
[ ] エラーレスポンスの共通フォーマットを決める
[ ] 認証方式（JWT / OAuth 2.0 / API キー）を選定し、認証基盤の実装または導入を先行する
[ ] フレームワークを選定し、プロジェクトのひな形（boilerplate）を作成する
[ ] Linter / Formatter / CI パイプラインを初日に整備する（後回しにしない）
[ ] ローカル開発環境を Docker Compose で構築し、チーム全員が同一環境で動かせる状態にする
[ ] Swagger UI または Redoc でドキュメントが自動生成・閲覧できることを確認する
[ ] ヘルスチェックエンドポイント（`GET /health`）を最初に実装する
[ ] レート制限・タイムアウトの基本設定を初期実装に含める
```

| Java | graphql-java, DGS Framework (Netflix) | DGS は Spring Boot 統合が容易 |
| Ruby | graphql-ruby | Rails との統合が容易。バッチローダー内蔵 |
| Rust | async-graphql | 高性能・型安全 |
| PHP | Lighthouse (Laravel) | Laravel と密結合。スキーマファーストで直感的 |

**スキーマ定義アプローチの選択:**

| アプローチ | 向いているケース | 代表ライブラリ |
|-----------|---------------|--------------|

#### チーム構成・スキル要件

**最小構成（小規模、2〜3名）:**
- GraphQL 経験者 × 1（Tech Lead 兼務）── スキーマ設計をリード
- バックエンドエンジニア × 1〜2

**標準構成（中規模、5〜8名）:**
- GraphQL アーキテクト / Tech Lead × 1 ── スキーマ設計・命名規約・パフォーマンス
- バックエンドエンジニア × 3〜4
- フロントエンドエンジニア × 1〜2（クエリ設計・Fragment 管理）
- インフラ／SRE × 1

**必須スキル:**
- GraphQL 仕様の理解（Query, Mutation, Subscription, Fragment, Directive）
- スキーマ設計（型システム、Relay スタイル vs シンプルスタイルの選択）
- DataLoader パターン（N+1 問題の解決）
- Apollo Client または urql などのクライアントライブラリ基礎知識

#### 開発工数の目安

| フェーズ | 工数（人日） | 内容 |
|---------|------------|------|
| スキーマ設計（初期） | 3〜7 | 型定義・クエリ設計・命名規約策定 |
| リゾルバ実装（1リソース） | 1〜2 | Query / Mutation ハンドラ |
| DataLoader 実装 | 1〜2（初回）/ 0.5（2回目以降） | バッチ・キャッシュ処理 |
| 認証・認可（フィールドレベル） | 3〜5（初回のみ） | ディレクティブ or ミドルウェア実装 |
| テスト | 1〜2 per リゾルバ | クエリテスト・エラー系 |

**プロジェクト全体の目安:**
- MVP（10〜15 クエリ・ミューテーション）: **6〜10週間**（2〜3名）── REST より初期コストが高い
- 中規模（30〜50 Operation）: **4〜6ヶ月**（4〜5名）
- Federation 構成（複数サービス統合）: **6ヶ月〜 + 専任アーキテクト**

#### 性能特性とインフラ設計

**GraphQL 特有のパフォーマンス問題と対策:**

| 問題 | 内容 | 対策 |
|------|------|------|
| N+1 クエリ | 各リゾルバが個別に DB アクセス | **DataLoader** でバッチ処理（必須） |
| 深いネストクエリ | 無制限なネストで大量データ取得 | クエリ深度制限（例: max depth 7） |
| 高コストクエリ | 1リクエストで大量データ取得 | クエリコスト分析（costLimit） |
| イントロスペクション悪用 | スキーマ全体の取得・解析 | 本番環境ではイントロスペクション無効化 |

**インフラ設計のポイント:**
- **CDN キャッシュは難しい**: `POST /graphql` が主なため HTTP キャッシュが効かない → Redis + DataLoader のキャッシュレイヤーで対応
- **Subscription**: WebSocket 水平スケール時は Redis Pub/Sub でブローカーが必要
- **Apollo Federation** でマイクロサービス統合する場合はゲートウェイが必要

#### コスト

**インフラコスト（AWS、月額概算）:**

| 規模 | 構成例 | 月額目安 |
|------|-------|---------|
| 小規模（〜5万 req/日） | ECS × 2, RDS t3.small, ElastiCache | 4〜7万円 |
| 中規模（〜50万 req/日） | ECS × 4〜6, RDS t3.medium, ElastiCache | 15〜30万円 |

**追加コスト:** Apollo Studio（GraphOS）: Free プランあり。Enterprise は $500〜/月

#### 注意点・落とし穴

1. **スキーマ設計を後から変更するコストが高い** — `@deprecated` を活用し、スキーマレジストリ（Apollo Studio, Hive）で変更を自動チェック
2. **DataLoader を後付けにする** — N+1 問題は最初のリゾルバ実装時から DataLoader を使う設計にする
3. **認可設計が複雑** — `graphql-shield`（Node.js）など専用ライブラリを活用する
4. **HTTP 200 でエラーが返る** — GraphQL はエラーが発生しても HTTP 200 を返すことがある
5. **ファイルアップロードは標準外** — `graphql-upload` ライブラリを使うか、別途 REST エンドポイントを設けるか先に決める
6. **本番環境でイントロスペクションを有効のまま放置** — 必ず無効化する

#### 最初にやること（チェックリスト）

```
[ ] スキーマ設計方針（スキーマファースト vs コードファースト）をチームで決定する
[ ] 命名規約を定める（型名: PascalCase, フィールド: camelCase, Enum: UPPER_SNAKE_CASE）
[ ] GraphQL Playground または GraphiQL を開発環境に設定する
[ ] DataLoader の基底クラス / ユーティリティを最初に実装する
[ ] 認証方式を決定し、コンテキストへのユーザー情報注入を実装する
[ ] クエリ深度制限・クエリコスト制限を設定する（例: max depth 7）
[ ] スキーマレジストリ（Apollo Studio Free または Hive）を導入する
[ ] 本番環境用の設定項目をリストアップする（イントロスペクション無効化など）
```

### 3. gRPCを選んだ後にやること

#### フレームワーク・ライブラリ選択

| 言語 | ライブラリ | 備考 |
|------|-----------|------|
| Node.js | `@grpc/grpc-js` | 純粋なJavaScript実装、ネイティブ拡張不要 |
| Python | `grpcio` + `grpcio-tools` | コード生成ツールセットを含む |
| Go | `google.golang.org/grpc` | 公式実装、パフォーマンスに優れる |
| Java | `grpc-java` | Android対応も含む公式実装 |
| .protoファイル管理 | `buf CLI` | Lint・破壊的変更検知・コード生成を統合管理 |

.protoファイルの管理には **buf CLI** の導入を強く推奨する。`buf lint` でスキーマ品質を保ち、`buf breaking` で破壊的変更を自動検知できる。

#### チーム構成・スキル要件

- **.proto定義の設計者が必要**: APIの契約となる `.proto` ファイルを正確に設計できるメンバーをアサインする
- **Protocol Buffersの理解**: フィールド番号の不変性、スカラー型とメッセージ型の違い、`oneof`・`repeated` の使い方を理解しているメンバーが少なくとも1名必要
- **HTTP/2の基礎知識**: ストリーミングRPCを扱う場合はフロー制御の概念も把握しておく

#### 開発工数の目安

| フェーズ | 工数目安 | 内容 |
|---------|---------|------|
| .proto設計 | 1〜3日 | サービス定義・メッセージ型設計・エラーモデル策定 |
| コード生成環境の整備 | 0.5〜1日 | buf CLIセットアップ・CIへの組み込み |
| サービス実装 | 1〜2日/エンドポイント | Unaryは短め、ストリーミングは長め |
| テスト・デバッグ環境 | 0.5〜1日 | grpcurlの導入・テスト用クライアントの整備 |

#### 性能特性とインフラ設計

- **HTTP/2の多重化**: 1つのTCP接続で複数のRPCを並行処理できる
- **バイナリプロトコル**: Protocol BuffersはJSONと比較してシリアライズ速度が速く、ペイロードサイズも小さい（目安として30〜50%削減）
- **L7ロードバランサーが必須**: HTTP/2を使用するためL4ロードバランサーでは負荷分散が機能しない。EnvoyまたはNginx（1.13.10以降）を使用

```
ブラウザ → gRPC-Web (HTTP/1.1 or HTTP/2) → Envoy（トランスコード）→ gRPC (HTTP/2) → バックエンドサービス
```

#### コスト

- インフラコストはREST APIと同等
- Envoy/Istioをサイドカーとして使う場合はメモリ使用量が増加（Envoy 1インスタンスあたり数十MB〜100MB程度）

#### 注意点・落とし穴

1. **ブラウザから直接呼び出せない** — gRPC-Web + Envoyの構成か、BFFでREST/GraphQLに変換する設計が必要
2. **デバッグが難しい** — バイナリプロトコルのため `grpcurl` を必ず導入する
3. **破壊的変更のルール** — フィールド番号の変更・削除は後方互換性を壊す。`buf breaking` をCIに組み込む
4. **gRPCのステータスコードをHTTPと混同しない** — `NOT_FOUND`・`INVALID_ARGUMENT` など独自のステータスコード体系を理解する

#### 最初にやること（チェックリスト）

```
[ ] buf CLIをインストールし、buf.yaml・buf.gen.yamlをリポジトリに追加する
[ ] .protoファイルを proto/ ディレクトリに配置し、サービス定義の初稿を作成する
[ ] buf lint を実行してスキーマのスタイルを確認する
[ ] grpcurl を全メンバーの開発環境に導入する
[ ] ロードバランサー（Envoy/Nginx/ALB）のL7対応を確認する
[ ] ブラウザクライアントの有無を確認し、必要であればgRPC-Web対応の設計を追加する
[ ] buf breaking をCIに組み込み、破壊的変更を自動検知できる状態にする
[ ] エラーステータスコードの方針をチームで合意し、共通のエラーメッセージ型を.protoに定義する
```

### 4. WebSocketを選んだ後にやること

#### フレームワーク・ライブラリ選択

| 言語・用途 | ライブラリ | 備考 |
|-----------|-----------|------|
| Node.js（低レベル） | `ws` | 軽量・高性能。プロトコルのみ実装し、ルーム管理などは自前 |
| Node.js（高レベル） | `Socket.io` | 再接続・ルーム・名前空間などの機能が組み込み済み |
| Python | `websockets` | asyncio対応の純粋なWebSocket実装 |
| Go | `gorilla/websocket` | Go製WebSocketのデファクトスタンダード |
| スケーラブルなWSサーバー | `Centrifugo` | チャンネル購読・ユーザー認証・スケールアウトをサポートする専用サーバー |

#### チーム構成・スキル要件

- イベント駆動・非同期処理の実装経験（コールバック・Promise・async/await）
- ステートフルな通信の設計経験 — REST のようなステートレス設計とは異なる
- 切断・再接続・タイムアウト・バックプレッシャーなど障害設計の経験
- Redis Pub/Subの設定・運用ができるインフラ担当

#### 開発工数の目安

| フェーズ | 工数目安 | 内容 |
|---------|---------|------|
| 基本接続・メッセージ送受信 | 1〜2日 | ハンドシェイク、メッセージ型定義 |
| 接続管理（切断・再接続） | 2〜3日 | ハートビート、指数バックオフ付き再接続 |
| ルーム・チャンネル機能 | 2〜4日 | 参加・退室・メッセージブロードキャスト |
| 認証・認可 | 1〜2日 | 接続時のトークン検証 |
| スケールアウト対応 | 2〜4日 | Redis Pub/Subアダプターの導入・テスト |

単純な通知機能であれば合計1〜2週間で実装できるが、ルーム管理・スケールアウトまで含めると3〜5週間を見込む。

#### 性能特性とインフラ設計

**スティッキーセッション vs Redis Pub/Sub**

| アプローチ | メリット | デメリット |
|-----------|---------|-----------|
| スティッキーセッション | 実装が簡単 | 特定サーバーへの負荷集中、サーバー障害時に接続が切れる |
| Redis Pub/Sub | 真の水平スケールが可能 | Redisの可用性が重要になる、レイテンシが若干増加 |

#### コスト

- WebSocketは接続を維持し続けるため、同時接続1万件を超えるあたりからREST APIとのコスト差が顕著になる
- AWS ALBはWebSocket接続時間に基づいた課金が発生する（LCU単価に注意）

#### 注意点・落とし穴

1. **切断・再接続の実装が必須** — クライアント側で指数バックオフ付きの再接続ロジックを実装しないと、一斉再接続によるサーバーへの負荷集中が発生する
2. **認証のタイミング** — 接続後認証の場合は認証完了までメッセージの配信を保留する設計にする
3. **メモリリーク** — `close` イベントと `error` イベントの両方でクライアントオブジェクトのクリーンアップ処理を実装する
4. **ハートビートの実装** — AWS ALBのデフォルトアイドルタイムアウトは60秒。ping/pongフレームを定期的に送信する

#### 最初にやること（チェックリスト）

```
[ ] 接続数の想定（ピーク時の同時接続数）を算出し、スケール戦略（スティッキーセッション or Redis Pub/Sub）を決定する
[ ] ライブラリを選定する（シンプルな通知なら ws、ルーム機能が必要なら Socket.io または Centrifugo）
[ ] クライアント側の再接続ロジック（指数バックオフ + ジッター）を最初に実装する
[ ] 認証方式を決定する（ハンドシェイク時認証 or 接続後最初のメッセージで認証）
[ ] サーバー側の接続管理（close・errorイベントでのクリーンアップ）を実装し、メモリリークがないことを確認する
[ ] ハートビート（ping/pong）を実装し、ロードバランサーのアイドルタイムアウト設定と合わせる
[ ] K8sにデプロイする場合は terminationGracePeriodSeconds を設定する
```

### 5. SSE（Server-Sent Events）を選んだ後にやること

#### フレームワーク・ライブラリ選択

| 言語・用途 | ライブラリ | 備考 |
|-----------|-----------|------|
| ブラウザ | `EventSource` (Web API) | ネイティブサポート、追加ライブラリ不要 |
| Node.js / Fastify | `fastify-sse-v2` | Fastify用プラグイン |
| Node.js / Express | `express` + `res.write()` | 生のHTTPレスポンスをストリームとして使用 |
| Python / FastAPI | `EventSourceResponse` (`sse-starlette`) | async generatorでSSEを送信 |
| Ruby on Rails | `ActionController::Live` | Railsの組み込み機能 |

```javascript
// Express でのSSE実装例
app.get('/events', (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');
  res.setHeader('Connection', 'keep-alive');

  const send = (data) => res.write(`data: ${JSON.stringify(data)}\n\n`);

  const timer = setInterval(() => send({ time: Date.now() }), 1000);
  req.on('close', () => clearInterval(timer));
});
```

#### チーム構成・スキル要件

- HTTP長時間接続（Chunked Transfer Encoding）の基本理解
- EventSourceのブラウザAPIの知識
- サーバー側の非同期I/Oの実装経験

#### 開発工数の目安

- シンプルな通知機能（サーバー→クライアントの一方向）: 1〜3日
- 認証・セキュリティ対応含む: 3〜5日

#### 性能特性とインフラ設計

- WebSocketより実装がシンプルで、HTTP/HTTPS標準プロトコルを使用するためプロキシ通過が容易
- **プロキシ・ロードバランサーのタイムアウト設定**: SSEは長時間接続のため、Nginx や AWS ALB のアイドルタイムアウトを延長する（Nginx: `proxy_read_timeout 86400s;`）
- スケールアウトはWebSocketと同様にRedis Pub/Subが必要

#### 注意点・落とし穴

1. **ブラウザのSSE接続数制限（HTTP/1.1は同一ドメインで6本まで）** — HTTP/2を使用すれば制限が大幅に緩和される
2. **クライアントからの送信は不可** — SSEは単方向通信。双方向が必要ならWebSocketを選ぶ
3. **自動再接続はブラウザが行うが、再接続間隔の制御はサーバー側から可能**（`retry:` フィールドで指定）
4. **IE/旧Edgeは非対応** — polyfill（`event-source-polyfill`）か、対応ブラウザのみのサービスに限定する

#### 最初にやること（チェックリスト）

```
[ ] 対象ブラウザのSSEサポートを確認する
[ ] サーバー側で適切なヘッダー（Content-Type: text/event-stream, Cache-Control: no-cache）を設定する
[ ] プロキシ・ロードバランサーのアイドルタイムアウトを長めに設定する
[ ] HTTP/2対応を確認し、ブラウザの接続数制限に対処する
[ ] クライアントの再接続時に使用する Last-Event-ID によるイベント再送の仕組みを実装する
```

### 6. SOAP / XML-RPCを選んだ後にやること

| Node.js | `node-soap` | WSDLからのクライアント生成対応 |

```python
# zeep を使ったSOAPクライアントの例
```

#### チーム構成・スキル要件

- WS-Security（認証・署名・暗号化）の実装経験があると望ましい

#### 開発工数の目安

- WSDL解析・スタブ生成: 1〜2日
- クライアント実装: 2〜4日（WSDL品質による）
- WS-Security対応: 追加2〜3日

#### 注意点・落とし穴

1. **新規開発での採用は避けるべき** — SOAPは主に既存の企業システム・官公庁APIとの連携用途。新規開発ではREST/gRPCを選ぶ
2. **WSDLの品質にばらつきがある** — 自動生成コードが正常に動作しないWSDLも多く、手動修正が必要になることがある
3. **デバッグが困難** — XMLエンベロープが複雑でエラーメッセージが分かりにくい。SOAPUIツールを必ず使う
4. **パフォーマンスが低い** — XML処理のオーバーヘッドが大きく、同等のRESTより2〜10倍遅いケースがある

#### 最初にやること（チェックリスト）

```
[ ] SOAPUIまたはPostmanでWSDLを読み込み、エンドポイントの動作を確認する
[ ] 対応ライブラリ（zeep / node-soap / Axis2）でWSDLからコードを自動生成する
[ ] 生成されたコードを実際のWSDLに合わせて調整する
[ ] WS-Security要件（認証方式）を相手先に確認する
[ ] エラーレスポンスのXML形式を把握し、パースロジックを実装する
```

### 7. tRPCを選んだ後にやること

#### フレームワーク・ライブラリ選択

| パッケージ | 用途 |
|-----------|------|
| `@trpc/server` | サーバー側ルーター・プロシージャ定義 |
| `@trpc/client` | クライアント側の型安全な呼び出し |
| `@trpc/next` | Next.js統合（APIルートとして使用） |
| `@trpc/react-query` | TanStack QueryとのReact統合 |
| `zod` | バリデーション・スキーマ定義（事実上必須） |

```typescript
// サーバー側（router定義）
import { initTRPC } from '@trpc/server';
import { z } from 'zod';

const t = initTRPC.create();
export const appRouter = t.router({
  getUser: t.procedure
    .input(z.object({ id: z.string() }))
    .query(async ({ input }) => {
      return await db.user.findUnique({ where: { id: input.id } });
    }),
  createUser: t.procedure
    .input(z.object({ name: z.string(), email: z.string().email() }))
    .mutation(async ({ input }) => {
      return await db.user.create({ data: input });
    }),
});

// クライアント側（TypeScript型補完が効く）
const user = await trpc.getUser.query({ id: '123' });
//                    ↑ 型補完・型チェックが効く
```

#### チーム構成・スキル要件

- **TypeScript必須**: フロントエンド・バックエンドともにTypeScript使用が前提
- Zodによるスキーマ定義・バリデーションの理解
- React（またはNext.js）の基礎知識（フロントエンド側）
- フルスタックTS開発経験があるメンバーがいると立ち上がりが早い

#### 開発工数の目安

- TypeScript + Next.js + Prismaのモノレポ構成前提で、RESTより30〜50%高速に実装できることが多い
- ルーター初期設定: 0.5日
- 認証ミドルウェア: 1〜2日
- 1プロシージャ（CRUD）の実装: 0.5〜1日

#### 性能特性とインフラ設計

- Next.jsのAPIルートとして動作するため、Vercelへのデプロイが最もシンプル
- サーバーレス（Lambda）でも動作するが、ストリーミングプロシージャは制限がある
- サブスクリプション（WebSocket）も対応しているが、サーバーレス環境では使えない

#### コスト

- インフラはNext.jsと同等（Vercel Freeプランで開始できる）
- 追加コストはほぼ発生しない

#### 注意点・落とし穴

1. **TypeScript以外のクライアントでは使えない** — モバイルネイティブ（Swift/Kotlin）や他言語のクライアントは別途REST/GraphQL APIが必要
2. **チーム全員がTypeScriptを使う前提** — フロントエンドがJS/他言語の場合は採用メリットがない
3. **大規模な公開APIには不向き** — 外部開発者に公開するAPIにはOpenAPI仕様があるREST/GraphQLが適切
4. **Next.js以外での設定は手間がかかる** — Express/Fastify/Hono等での設定は追加の工数が必要

#### 最初にやること（チェックリスト）

```
[ ] TypeScript + Zodが全員の環境で動作することを確認する
[ ] create-t3-app または公式テンプレートでプロジェクトを立ち上げる
[ ] 認証（NextAuth.js / Auth.js）の設定を最初に行う
[ ] コンテキストにユーザー情報を注入する仕組みを実装する
[ ] エラーハンドリング（TRPCError）の共通パターンを決める
[ ] クライアント用のtRPCラッパーコンポーネントをセットアップする
```

## 全API横断まとめ

### API選択から実装までのロードマップ

| API種別 | 1週目 | 1ヶ月目 | 3ヶ月目 |
|---------|-------|---------|---------|
| REST | OpenAPI定義・認証基盤・Boilerplate | CRUD実装・テスト・CI | 負荷テスト・監視・v2設計 |
| GraphQL | スキーマ設計・DataLoader | リゾルバ実装・認可 | Federation・パフォーマンスチューニング |
| gRPC | .proto設計・buf CLI設定 | サービス実装・grpcurl | ストリーミング・EnvoyまたはIstio導入 |
| WebSocket | 基本接続・再接続実装 | ルーム機能・認証 | Redis Pub/Sub・負荷テスト |
| SSE | エンドポイント実装・タイムアウト設定 | イベント再送・認証 | スケールアウト対応 |
| tRPC | create-t3-app・認証設定 | プロシージャ実装 | パフォーマンス最適化 |
| SOAP | SOAPUIで動作確認・ライブラリ選定 | スタブ生成・クライアント実装 | テスト整備・エラーハンドリング |

### スタック別おすすめ組み合わせ

| スタック | 推奨API | 理由 |
|---------|---------|------|
| Next.js（フルスタック） | tRPC または REST（OpenAPI） | tRPCは型安全・高速開発、公開APIが必要ならREST |
| Rails | REST（Grape / Rails APIモード） | Railsのエコシステムと相性が良い |
| Django / FastAPI | REST または GraphQL（Strawberry） | FastAPIはOpenAPI自動生成が強力 |
| Go（Gin / Echo） | REST または gRPC | GoのgRPCパフォーマンスは最高水準 |
| Spring Boot（Java） | REST または gRPC | DGS FrameworkでGraphQLも可 |
| リアルタイム（チャット・通知） | WebSocket または SSE | 単方向通知ならSSE、双方向ならWebSocket |
| マイクロサービス間通信 | gRPC | バイナリ効率・型安全・ストリーミング対応 |
| 外部公開API | REST + OpenAPI | 最も広くサポートされる、SDKも自動生成可能 |

### よくある失敗パターンと対策

1. **APIの種別を最初に選ばずに実装を始める** — 途中での変更コストは非常に高い。設計フェーズで「誰が呼ぶか（ブラウザ/モバイル/サービス間）」「双方向か」「型安全が必要か」を確定してから開始する
2. **RESTでスキーマ定義（OpenAPI）を後回しにする** — 実装後にOpenAPIを書こうとすると整合性の確保が困難。コード生成ツール（FastAPI/NestJS等）でコードからOpenAPIを自動生成する構成にする
3. **GraphQLでDataLoaderを後付けにする** — N+1問題は本番データが増えるまで気づかないことが多い。開発初期から組み込む
4. **WebSocketで再接続ロジックを省略する** — ネットワーク障害は必ず発生する。thundering herd対策（指数バックオフ + ジッター）を最初から実装する
5. **認証・認可を後から追加する** — どのAPI方式でも、認証の仕組みは最初のスプリントで構築するべき。後から追加するとセキュリティホールが生まれやすい
