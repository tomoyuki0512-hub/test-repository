# SSO（シングルサインオン）完全ガイド 〜ゼロからわかる仕組みと導入〜

「ひとつのIDとパスワードで、いろんなサービスにログインできる」――これが **SSO（シングルサインオン）** です。Google アカウントひとつで YouTube・Gmail・Google ドライブにそのまま入れるあの体験が、まさに SSO です。

このガイドは **専門知識ゼロの人でも理解できる** ことを目標に、たとえ話から始めて、しくみ・主要な技術（SAML / OAuth / OIDC）・導入方法・セキュリティ・トラブル対応まで、こと細かに解説します。

> 想定読者：これから SSO を学ぶ人、社内に SSO を導入したい人、エンジニア、情シス担当者、なんとなく言葉だけ知っている人。

## 1. SSOとは何か（たとえ話でつかむ）

**SSO（Single Sign-On / シングルサインオン）** とは、

> **一度ログインすれば、複数のサービスやアプリに、その都度ログインし直さずに使える仕組み**

のことです。

### たとえ話：遊園地の「1日フリーパス」

遊園地に入るとき、入口で身分証を見せて **1日フリーパス（リストバンド）** をもらうイメージです。

- アトラクションごとに毎回チケットを買う必要はありません。
- どのアトラクションでも、リストバンドを見せれば乗れます。
- リストバンドは「入口（＝信頼できる係員）」が発行したものなので、各アトラクションの係員はそれを信じて通してくれます。

この「入口の係員」が **IdP（アイデンティティプロバイダー）**、各「アトラクション」が利用したい **アプリ（SP）**、「リストバンド」が **認証トークン** に当たります。

### よくある誤解の整理

| 誤解 | 正しい理解 |
| --- | --- |
| 「全部のパスワードを同じにすること」 | ❌ 違います。パスワードを使い回すのは危険。SSOは“ログイン状態を共有”する仕組み |
| 「パスワード管理ツールのこと」 | ❌ それは “パスワードマネージャー”。SSOとは別物 |
| 「Googleでログイン＝SSO」 | ⭕ おおむね正しい（ソーシャルログインはSSOの一種） |

## 2. SSOがあると何が嬉しいのか

### 利用者（ユーザー）にとって

- **覚えるパスワードが1つで済む** → 「えっと、このサービスのパスワード何だっけ…」が激減。
- **ログインが速い・ラク** → 業務アプリが10個あっても、朝1回ログインすれば全部使える。
- **パスワード使い回しが減る** → 結果的に安全になる。

### 企業・管理者にとって

- **退職者のアクセスを一括停止できる** → IdP側でアカウントを止めれば、連携している全アプリから締め出せる（**重要なセキュリティ効果**）。
- **パスワードリセット対応が減る** → ヘルプデスクの負担減（問い合わせの多くは「パスワード忘れ」）。
- **アクセス管理を一元化できる** → 「誰がどのアプリを使えるか」を1か所で管理。
- **監査・ログが集約できる** → 「いつ・誰が・どこにログインしたか」を追える。
- **ポリシーを統一適用できる** → MFA必須、特定IPのみ許可、などをまとめて強制。

---

## 3. SSOのデメリット・注意点

良いことばかりではありません。導入前に必ず知っておくべき点です。

- **「鍵が1つ」になるリスク（単一障害点）**
  IdPのアカウントが乗っ取られると、連携している **全サービス** に侵入されます。だからこそ IdP には **MFAが必須**。
- **IdPが落ちると全部止まる**
  IdPがダウンすると、連携アプリにログインできなくなる可能性があります（可用性の確保が重要）。
- **初期導入の手間・コスト**
  設定（特に SAML）はやや複雑。各アプリごとに連携設定が必要。
- **すべてのアプリがSSOに対応しているわけではない**
  古いアプリや独自実装のアプリは対応していないことがあります。

> 結論：SSOは「便利さ」と引き換えに「IdPの守りを固める責任」が増える。**MFAと可用性対策はセット**で考えるのが鉄則です。

## 4. 登場人物と基本用語

SSOを理解するには、まず「登場人物」を覚えるのが近道です。

| 用語 | 読み / 正式名称 | ざっくり言うと | たとえ |
| --- | --- | --- | --- |
| **IdP** | Identity Provider（アイデンティティプロバイダー） | 本人確認をして「この人は本物です」と保証する側 | 遊園地の入口・パスポート発行局 |
| **SP** | Service Provider（サービスプロバイダー） | ユーザーが使いたいアプリ・サービス側 | 各アトラクション・入国先の国 |
| **RP** | Relying Party（リライングパーティ） | OIDCでの「SP」の呼び方。IdPを“頼る”側 | SPと同じ役割 |
| **プリンシパル** | Principal | 認証される本人（＝ユーザー） | あなた |
| **アサーション / トークン** | Assertion / Token | 「本人確認OK」を証明する電子的な証明書 | リストバンド・ビザ |
| **クレーム** | Claim | トークンに含まれる属性情報（名前・メール・所属など） | パスポートの記載事項 |
| **フェデレーション** | Federation | 別々の組織・ドメイン間で認証を信頼し合う仕組み | 国同士のビザ相互承認 |

> **覚え方**：本人を保証する「**IdP**」が、利用したいアプリ「**SP / RP**」に向けて、「この人は本物だよ」という**トークン**を渡す。これだけ押さえればOK。

## 5. SSOの基本的な流れ（ざっくり）

技術が違っても、SSOの大筋は同じです。

```
1. ユーザーがアプリ（SP）にアクセスする
        ↓
2. SPが「まだログインしてないね。IdPで本人確認してきて」とIdPへ送る
        ↓
3. ユーザーがIdPでログイン（ID/パスワード、MFAなど）
   ※すでにIdPにログイン済みなら、この画面すら出ずに素通り（これがSSO体験）
        ↓
4. IdPが「この人は本物」という“トークン（証明書）”を発行
        ↓
5. ユーザーはそのトークンを持ってSPに戻る
        ↓
6. SPがトークンを検証 → OKならログイン成功！
```

ポイントは **手順3**。一度 IdP にログイン済みなら、2つ目以降のアプリでは **ログイン画面すら出ずに通過** します。これが「シングルサインオン（一度のサインオン）」の名前の由来です。

### 図解（テキスト）

```
[ユーザー] --(1)アクセス--> [SP アプリ]
     ^                          |
     |                     (2)IdPへ誘導
     |                          v
     |                      [IdP 認証局]
     |                          |
 (5)トークン持参 <--(4)トークン発行--+
     |
     v
 (6)検証OK → ログイン完了
```

## 6. SSOを実現する主要な技術

SSOは「ひとつの技術」ではなく、いくつかの **プロトコル（約束ごと）** で実現されます。代表的なのは次の3つ。

| 技術 | 主な用途 | 形式 | ひとことで |
| --- | --- | --- | --- |
| **SAML 2.0** | 企業向けSSO（社内アプリ・SaaS） | XML | 老舗。エンタープライズの定番 |
| **OAuth 2.0** | **認可**（権限の委譲） | JSON/トークン | 「このアプリにあなたのデータへのアクセスを許可」 |
| **OpenID Connect (OIDC)** | **認証**（ログイン） | JSON(JWT) | OAuth2の上に作られた現代の主流 |

> **超重要な区別**
> - **認証（Authentication）＝「あなたは誰？」を確かめる** → SAML, OIDC
> - **認可（Authorization）＝「あなたに何を許可する？」を決める** → OAuth 2.0
>
> OAuth 2.0 は本来 “認可” の仕組みであって、“ログイン（認証）” のための仕組みではありません。これを認証目的に正しく拡張したのが **OpenID Connect** です。ここを混同すると理解が一気にこんがらがるので要注意。

## 7. SAML を詳しく

**SAML（Security Assertion Markup Language、サムル）** は、2005年に標準化（2.0）された、企業向けSSOの定番プロトコルです。**XML** 形式でやり取りします。

### 特徴

- 企業の社内システムや SaaS（Salesforce, Workday など）で広く採用。
- 認証情報（本人確認結果）を **アサーション（Assertion）** という XML 文書でやり取り。
- アサーションには **デジタル署名** が付き、改ざんを検知できる。

### SAML の登場人物

- **IdP**：本人確認をして SAML アサーションを発行（例：Okta, Azure AD/Entra ID, OneLogin）。
- **SP**：アサーションを受け取って検証するアプリ（例：Salesforce）。

### SAML の2つの開始パターン

1. **SP-Initiated（SP起点）**
   ユーザーがまずアプリ（SP）にアクセス → SPがIdPへリダイレクト → 認証 → 戻る。**一番よくあるパターン**。
2. **IdP-Initiated（IdP起点）**
   ユーザーがまず IdP のポータル（アプリ一覧画面）にログイン → 使いたいアプリのアイコンをクリック → そのアプリに入る。

### SAML の流れ（SP-Initiated）

```
1. ユーザーがSP(アプリ)にアクセス
2. SPが「SAML認証リクエスト(AuthnRequest)」を作り、IdPへリダイレクト
3. ユーザーがIdPでログイン
4. IdPが「SAMLアサーション(署名付きXML)」を発行
5. ブラウザがそのアサーションをSPへPOST送信
6. SPが署名を検証 → OKならログイン成立
```

### SAML でよく出てくる用語

- **Assertion（アサーション）**：本人確認結果の本体。
- **AuthnRequest**：SPがIdPに送る「認証して」という依頼。
- **ACS（Assertion Consumer Service）URL**：IdPがアサーションを送り返す先のSP側URL。
- **Entity ID**：IdPやSPを一意に識別するID（連携設定で必須）。
- **メタデータ（Metadata）**：連携に必要な情報（証明書・URL・Entity ID等）をまとめたXMLファイル。IdPとSPでこれを交換して設定する。
- **証明書（X.509）**：署名検証に使う公開鍵。

### SAMLのメリット・デメリット

- ✅ エンタープライズで実績豊富。属性情報を細かく渡せる。署名で安全。
- ❌ XMLで重く設定が複雑。**モバイルアプリやSPA（シングルページアプリ）には不向き**。

## 8. OAuth 2.0 を詳しく

**OAuth 2.0（オーオース）** は、「**認可（権限の委譲）**」のためのプロトコルです。**SSO（ログイン）専用ではない** 点に注意。

### 何を解決する仕組み？

> 「アプリAに、あなたのGoogleの連絡先を読み取らせてOK？」

のように、**パスワードを渡さずに**、特定の権限だけを別アプリに許可する仕組みです。

たとえば「この写真印刷アプリに、Googleフォトの写真を読む権限だけ与える（でもメールは見せない）」といった、**必要最小限の権限だけ渡す** ことができます。

### OAuth 2.0 の登場人物

| 役割 | 意味 | 例 |
| --- | --- | --- |
| **Resource Owner** | データの持ち主（ユーザー本人） | あなた |
| **Client** | データを使いたいアプリ | 写真印刷アプリ |
| **Authorization Server** | 許可を出して**トークンを発行**するサーバー | Googleの認可サーバー |
| **Resource Server** | 実際のデータを持つサーバー（API） | Googleフォト API |

### 主要な用語

- **アクセストークン（Access Token）**：APIにアクセスするための“通行証”。期限が短い。
- **リフレッシュトークン（Refresh Token）**：アクセストークンの期限が切れたとき、再発行に使う“予備の鍵”。
- **スコープ（Scope）**：許可する権限の範囲（例：`read:photos`）。
- **認可コード（Authorization Code）**：トークンと引き換える一時的なコード。

### 代表的な流れ：認可コードフロー（Authorization Code Flow）

最も安全で標準的な流れです。

```
1. アプリ(Client)がユーザーを認可サーバーへ誘導
2. ユーザーがログインし「この権限を許可しますか?」に同意
3. 認可サーバーが「認可コード」をアプリへ返す
4. アプリが裏側(サーバー間)で 認可コード → アクセストークン に交換
5. アプリがアクセストークンを使ってAPIからデータ取得
```

> 手順4を「ブラウザの表ではなく、サーバー同士の裏側」で行うのが安全のポイント。トークンが第三者に漏れにくくなります。

### PKCE（ピクシー）

モバイルアプリやSPAなど「秘密鍵を安全に隠せないアプリ」向けに、認可コードの横取りを防ぐ拡張が **PKCE（Proof Key for Code Exchange）** です。**現在はすべての認可コードフローでPKCEの利用が推奨** されています。

### なぜ OAuth 単体を「ログイン」に使ってはいけないのか

OAuth のアクセストークンは「**何ができるか（認可）**」を表すもので、「**誰なのか（認証）**」を保証する設計ではありません。これを無理にログインに使うと、なりすましなどの脆弱性につながります。**ログイン目的なら次の OIDC を使うのが正解** です。

## 9. OpenID Connect（OIDC）を詳しく

**OpenID Connect（オープンアイディーコネクト、OIDC）** は、**OAuth 2.0 の上に「認証（ログイン）」の機能を追加** した、現代のSSOの主流プロトコルです。2014年に標準化されました。

> ひとことで言うと：**OIDC ＝ OAuth 2.0 ＋ ログイン（本人確認）機能**

「Googleでログイン」「Appleでサインイン」などの **ソーシャルログイン** の多くは、この OIDC で動いています。

### OIDC の鍵：ID トークン

OAuth に対して OIDC が追加した最大のものが **IDトークン（ID Token）** です。

- 形式は **JWT（ジョット / JSON Web Token）**。
- 中身は「誰がログインしたか」を示す情報（ユーザーID・名前・メールなど＝**クレーム**）。
- デジタル署名付きで、改ざんを検知できる。

### JWT（JSON Web Token）の構造

JWTは `.（ドット）` で3つに区切られた文字列です。

```
ヘッダー.ペイロード.署名
xxxxx.yyyyy.zzzzz
```

| 部分 | 中身 |
| --- | --- |
| **ヘッダー** | 署名アルゴリズムなどのメタ情報 |
| **ペイロード** | クレーム（sub=ユーザーID, name, email, exp=有効期限 など） |
| **署名** | 改ざん検知用の電子署名 |

代表的なクレーム：

- `iss`（issuer）：発行者（IdP）
- `sub`（subject）：ユーザーの一意なID
- `aud`（audience）：このトークンの宛先（どのアプリ向けか）
- `exp`（expiration）：有効期限
- `email`, `name`：ユーザー属性

### OIDC の流れ

基本は OAuth の認可コードフローと同じですが、`scope` に **`openid`** を含めるのがポイント。これがあると、アクセストークンに加えて **IDトークン** が発行されます。

```
1. アプリ(RP)がユーザーをIdPへ誘導（scope=openid を含める）
2. ユーザーがIdPでログイン
3. IdPが「認可コード」を返す
4. アプリが 認可コード → IDトークン + アクセストークン に交換
5. アプリがIDトークン(JWT)を検証し「この人は本物」と確認 → ログイン完了
```

### OIDC が選ばれる理由

- ✅ **モバイル・SPA・APIと相性が良い**（JSONベースで軽い）。
- ✅ OAuth資産をそのまま活かせる。
- ✅ ソーシャルログインの事実上の標準。
- ✅ SAMLより設定がシンプルなことが多い。

## 10. SAML vs OAuth vs OIDC 比較

| 項目 | SAML 2.0 | OAuth 2.0 | OpenID Connect |
| --- | --- | --- | --- |
| 主目的 | 認証（SSO） | **認可**（権限委譲） | 認証（SSO） |
| データ形式 | XML | JSON / トークン | JSON（JWT） |
| 登場時期 | 2005年 | 2012年 | 2014年 |
| 得意分野 | 企業内・SaaS | API権限の委譲 | モバイル・SPA・ソーシャルログイン |
| トークン | SAMLアサーション | アクセストークン | IDトークン＋アクセストークン |
| モバイル相性 | △ 弱い | ○ | ◎ 強い |
| 設定の手軽さ | △ やや複雑 | ○ | ○ |
| 代表例 | Salesforce, Workday | API連携全般 | Googleでログイン等 |

### どれを選ぶ？ ざっくり指針

- **社内の業務SaaSをまとめてSSOしたい** → まず **SAML**（多くのSaaSが対応）。新しめなら OIDC も。
- **自社のWeb/モバイルアプリにログイン機能を作る** → **OIDC**。
- **「他社サービスのデータを読む権限だけ欲しい」** → **OAuth 2.0**。
- **新規にゼロから作るなら** → 迷ったら **OIDC** が現代的でおすすめ。

## 11. その他の方式（Kerberos / LDAP / レガシー）

Web系のSAML/OIDC以外にも、SSOに関わる仕組みがあります。

- **Kerberos（ケルベロス）**
  社内ネットワーク（特にWindows/Active Directory環境）で使われる認証方式。「チケット」を発行してサーバー間で本人確認する。Windowsドメインに参加したPCで、ログイン後にファイルサーバーへ追加認証なしでアクセスできるのはこれ。
- **LDAP（エルダップ）**
  ユーザー情報（社員名簿のようなディレクトリ）を保管・検索する仕組み。Active Directory の中核。SSOというより「ユーザー情報の置き場＆認証元」。
- **IWA（統合Windows認証）**：社内PCのWindowsログイン情報をそのままWebアプリ認証に使う。
- **レガシーな手法**：パスワードを代理入力する「**パスワードボールティング/フォームベース**」型SSO。古いアプリ向けの苦肉の策で、SAML/OIDCが使えない場合の代替。

## 12. 多要素認証（MFA）との関係

SSOを語るうえで **MFA（多要素認証 / Multi-Factor Authentication）** は切り離せません。

- **MFAとは**：パスワード（知識）に加えて、スマホアプリのコード（所持）や指紋（生体）など、**複数の要素** で本人確認すること。
- **なぜSSOに必須か**：SSOは「IdPの鍵ひとつで全アプリに入れる」ため、その鍵が破られると被害が全アプリに及ぶ。だから **IdPのログインにはMFAを必ず付ける** のが基本。

### 認証の「3要素」

| 要素 | 説明 | 例 |
| --- | --- | --- |
| 知識情報（Something you know） | 本人だけが知っている | パスワード、PIN |
| 所持情報（Something you have） | 本人だけが持っている | スマホ、ICカード、認証アプリ |
| 生体情報（Something you are） | 本人の身体的特徴 | 指紋、顔、虹彩 |

### パスキー / FIDO2 / WebAuthn

最近は、フィッシングに極めて強い **パスキー（Passkey）** や **FIDO2 / WebAuthn** が普及中。パスワードそのものを使わない「パスワードレス認証」で、SSOのIdPログインと組み合わせると非常に強固になります。

## 13. SSOと一緒に使われる仕組み（プロビジョニング・SCIM）

SSOは「ログイン」を担いますが、**「アカウントを作る・消す」** は別の仕組みが担当します。それが **プロビジョニング** です。

- **プロビジョニング（Provisioning）**：新入社員が来たら各アプリにアカウントを自動作成、退職したら自動削除（**デプロビジョニング**）する仕組み。
- **SCIM（スキム / System for Cross-domain Identity Management）**：このアカウント情報の同期を **標準化** したプロトコル。IdPで社員を1人追加すれば、連携SaaS群に自動でアカウントが作られる。

> **SSO ＋ SCIM** をセットで導入すると、「入社時のアカウント発行」も「退職時の即時停止」も自動化でき、情シスの運用が劇的にラクになります。

## 15. 導入のステップ

社内にSSOを導入する場合の、おおまかな流れです。

1. **目的とゴールを決める**
   何を解決したいか（パスワード負担減？退職時の即停止？監査強化？）を明確に。
2. **IdPを選ぶ**
   既に Microsoft 365 を使っているなら Entra ID が自然、など既存環境に合わせる。
3. **対象アプリを洗い出す**
   SSO連携したいアプリをリスト化し、各アプリが **SAML / OIDC のどちらに対応するか** を確認。
4. **連携設定（フェデレーション）**
   - SAMLなら：IdPとSPで **メタデータ（Entity ID, ACS URL, 証明書）** を交換して設定。
   - OIDCなら：**Client ID / Client Secret / リダイレクトURI** を設定。
5. **属性マッピング**
   IdPのユーザー属性（メール、所属など）を、各アプリのどの項目に渡すか対応付け。
6. **MFAポリシーを設定**
   IdPログインにMFAを必須化。リスクベース認証なども検討。
7. **（任意）SCIMでプロビジョニング自動化**
8. **テスト**
   一部ユーザーで試験運用 → ログイン・ログアウト・エラー時の挙動を確認。
9. **本展開＆周知**
   利用者へ使い方を案内し、段階的に全社展開。
10. **運用・監視**
    ログ監視、定期的なアクセス権の棚卸し。

## 16. セキュリティ上の注意点

- **IdPのアカウントを最優先で守る**：MFA必須・強力なパスワード・管理者アカウントの厳重管理。
- **可用性を確保する**：IdPがダウンすると全部止まる。冗長化やフェイルオーバーを検討。
- **トークンの有効期限を適切に**：長すぎるトークンは盗まれたときのリスク大。短め＋リフレッシュトークンで運用。
- **HTTPS（TLS）必須**：トークンやアサーションは必ず暗号化通信で。
- **署名・証明書の検証を必ず行う**：受け取ったトークンの署名を検証せず信用するのは厳禁。証明書の有効期限切れにも注意（**SAML証明書の期限切れは“あるある”障害**）。
- **リダイレクトURIを厳格に**：OAuth/OIDCで許可するリダイレクト先を限定し、オープンリダイレクトを防ぐ。
- **最小権限の原則**：スコープ・権限は必要最小限に。
- **ログアウト（シングルログアウト/SLO）の設計**：1か所のログアウトで全アプリからログアウトさせるか、挙動を明確に。
- **退職者の即時失効**：デプロビジョニングと連動させ、アクセスを速やかに止める。

## 17. よくあるトラブルと対処

| 症状 | よくある原因 | 対処 |
| --- | --- | --- |
| ログインループ（無限リダイレクト） | 時刻ずれ、Cookie設定、ACS URL誤り | サーバー時刻(NTP)同期、URL設定見直し |
| 「署名検証に失敗」エラー | 証明書の不一致・期限切れ | IdP/SPの証明書を再交換・更新 |
| ログインはできるが属性が空 | 属性マッピング漏れ | クレーム/属性の対応付けを設定 |
| 一部ユーザーだけ入れない | アプリへの割り当て漏れ | IdPでそのユーザー/グループにアプリを割り当て |
| 突然全員入れなくなった | **証明書の有効期限切れ** | 証明書を更新し再連携（期限を事前に監視） |
| `redirect_uri_mismatch`（OIDC） | リダイレクトURIの登録ミス | アプリ設定のリダイレクトURIを正確に登録 |
| トークン期限切れ多発 | 有効期限が短すぎ/リフレッシュ未実装 | リフレッシュトークン運用を実装 |

> **時刻ずれ** と **証明書の期限切れ** が、SAML/OIDCの二大トラブル原因です。NTPによる時刻同期と、証明書の期限監視は必ず仕込んでおきましょう。

## 18. 用語集

- **SSO**：一度のログインで複数サービスを使える仕組み。
- **IdP**：本人確認をしてトークンを発行する側。
- **SP / RP**：ユーザーが使いたいアプリ側。
- **フェデレーション**：組織間で認証を信頼し合うこと。
- **認証(Authentication)**：「あなたは誰か」を確かめること。
- **認可(Authorization)**：「あなたに何を許可するか」を決めること。
- **SAML**：XMLベースの企業向けSSOプロトコル。
- **アサーション**：SAMLでの本人確認結果（署名付きXML）。
- **OAuth 2.0**：権限委譲（認可）のプロトコル。
- **OIDC**：OAuth2に認証を足したログイン用プロトコル。
- **JWT**：署名付きの軽量トークン形式（OIDCのIDトークンなど）。
- **クレーム**：トークンに入る属性情報（名前・メール等）。
- **スコープ**：許可する権限の範囲。
- **アクセストークン**：APIにアクセスするための通行証。
- **リフレッシュトークン**：アクセストークン再発行用の鍵。
- **PKCE**：認可コード横取りを防ぐOAuth拡張。
- **MFA**：複数要素で本人確認する多要素認証。
- **パスキー / FIDO2 / WebAuthn**：フィッシングに強いパスワードレス認証。
- **SCIM**：アカウント情報を自動同期する標準プロトコル。
- **プロビジョニング**：アカウントの自動作成・削除。
- **Kerberos**：チケット方式の社内ネットワーク認証。
- **LDAP**：ユーザー情報を保管・検索するディレクトリ。
- **SLO（シングルログアウト）**：1か所のログアウトで全アプリからログアウト。

## 19. まとめ

- **SSO** は「**一度のログインで複数サービスを使える**」便利な仕組み。
- 中心にいるのは **IdP（本人確認役）** と **SP/RP（使いたいアプリ）**。IdPが発行する **トークン** を信頼してログインが通る。
- 実現技術は主に3つ：
  - **SAML** … 企業向け・XML・老舗。
  - **OAuth 2.0** … “認可”（権限委譲）。ログイン専用ではない。
  - **OIDC** … OAuth2＋ログイン。現代の主流（ソーシャルログイン）。
- **認証（誰か）と認可（何を許すか）の違い** を押さえるのが理解のカギ。
- 便利な反面、**IdPが単一障害点** になるため **MFA・可用性・証明書管理** がセットで重要。
- **SCIM** と組み合わせると、入退社のアカウント管理まで自動化できる。

SSOは「ラクになる」だけでなく、**正しく運用すればセキュリティも強くなる** 仕組みです。鍵をひとつにまとめる代わりに、その鍵を全力で守る――この発想を持って導入・運用していきましょう。
