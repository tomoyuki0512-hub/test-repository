# ⑤ セキュリティとMFA・パスキー 〜攻撃シナリオから学ぶ守り方〜

> **この章でわかること**
> - SSO環境が狙われる典型的な攻撃シナリオと、それぞれの対策
> - MFA（多要素認証）がなぜ「SSOとセット」なのか
> - パスキー / FIDO2 / WebAuthn がフィッシングに強い理由
> - その他の関連方式（Kerberos / LDAP / レガシーSSO）

---

## 1. 大前提：SSOは「鍵を1つにまとめる」仕組み

SSOの構造上の宿命は、**IdPが「全アプリの鍵束」になる**ことです。守りをIdPに集中できるのは利点ですが、裏を返せば **IdPが破られたら全部やられる** ということ。だからSSOのセキュリティは「IdPをどう守るか」に尽きます。

以下、実際に起こる攻撃シナリオごとに「何が起きるか → どう防ぐか」を見ていきます。

---

## 2. 攻撃シナリオと対策

### シナリオA：IdPアカウントの乗っ取り（最悪のケース）

> 攻撃者がフィッシングメールで社員のIdPパスワードを入手。IdPにログインされ、**連携する全SaaS（メール・人事・経理・顧客データ）に一気に侵入される。**

| 対策 | 内容 |
| --- | --- |
| **MFAの必須化** | パスワードが漏れても2つ目の要素で止める。**SSO導入とMFAは必ずセット**（後述） |
| パスキー導入 | フィッシングでそもそも盗めない認証へ（後述） |
| 条件付きアクセス | 「社外IPからは追加認証」「未知のデバイスはブロック」などリスクベースで制御 |
| 管理者アカウントの分離 | IdP管理者権限は日常業務アカウントと分け、最厳重に守る |

### シナリオB：トークン・アサーションの窃取

> マルウェアや中間者攻撃で、発行済みのトークンやSAMLアサーションを盗み、本人になりすましてアプリへ。

| 対策 | 内容 |
| --- | --- |
| **HTTPS（TLS）必須** | トークンが流れる経路は必ず暗号化。平文HTTPは論外 |
| **トークンの有効期限を短く** | SAMLアサーションは数分、アクセストークンは数分〜1時間が目安。盗まれても使える時間を最小化 |
| リフレッシュトークンの失効管理 | 異常検知時に即失効できる仕組みを用意 |
| 送信者制約 | 可能なら「そのデバイスからしか使えないトークン」（DPoP等）を検討 |

### シナリオC：署名検証の手抜きを突く

> SP側の実装が署名検証を省略・不備にしていると、攻撃者が**偽のアサーション/トークンを自作**してログインできてしまう。

| 対策 | 内容 |
| --- | --- |
| **署名・証明書を必ず検証** | 受け取ったトークンの署名、`iss`（発行者）、`aud`（宛先）、`exp`（期限）をすべて検証する |
| 実績あるライブラリを使う | 自前実装は事故のもと。各言語の標準的なSAML/OIDC/JWTライブラリを使う |
| 証明書の期限監視 | 期限切れは「全員ログイン不能」障害に直結（→ [⑥導入と運用](06-deployment-ops.md)） |

### シナリオD：リダイレクトURIの悪用（OAuth/OIDC）

> 認可コードの戻し先 `redirect_uri` の検証が緩いと、攻撃者のサイトへ認可コードを横流しさせられる（オープンリダイレクト）。

| 対策 | 内容 |
| --- | --- |
| **リダイレクトURIの完全一致登録** | ワイルドカードや前方一致を避け、正確なURLだけ許可 |
| **PKCEの利用** | コードを横取りされてもトークンに交換できない（→ [③OAuth 2.0](03-oauth2.md)） |
| `state` パラメータの検証 | CSRF（リクエスト偽造）を防ぐ |

### シナリオE：退職者・削除漏れアカウントの残存

> 退職した元社員のアカウントが一部のSaaSに残っていて、そこから情報が持ち出される。

| 対策 | 内容 |
| --- | --- |
| **IdP起点の一括無効化** | SSO化されていれば、IdPで止めた瞬間に全連携アプリへのログインが止まる |
| SCIMによる自動デプロビジョニング | アカウント削除まで自動化（→ [⑥導入と運用](06-deployment-ops.md)） |
| 定期的な棚卸し | SSO外のアプリ・共有アカウントが残っていないか点検 |

### シナリオF：IdP自体の障害（攻撃ではないが同じくらい痛い）

> IdPがダウンし、**全アプリに誰もログインできない**「全社ログイン停止」状態に。

| 対策 | 内容 |
| --- | --- |
| 冗長構成・SLA確認 | IdPサービスの稼働率保証と障害履歴を確認して選定 |
| 緊急アクセス手順の整備 | 「ブレークグラス（緊急用）アカウント」を用意し、金庫保管などで厳重管理 |

## 3. MFA（多要素認証）〜SSOの必須の相棒〜

**MFA（Multi-Factor Authentication）** とは、パスワードに加えて別の「要素」でも本人確認することです。

### 認証の3要素

| 要素 | 説明 | 例 |
| --- | --- | --- |
| 知識情報（Something you **know**） | 本人だけが知っている | パスワード、PIN |
| 所持情報（Something you **have**） | 本人だけが持っている | スマホ（認証アプリ）、セキュリティキー |
| 生体情報（Something you **are**） | 本人の身体的特徴 | 指紋、顔 |

「パスワード＋SMSコード」のように**異なる要素を2つ以上**組み合わせるのがMFAです（パスワード＋秘密の質問は同じ“知識”同士なのでMFAとしては弱い）。

### MFA方式の強さ比較

| 方式 | フィッシング耐性 | 手軽さ | 備考 |
| --- | --- | --- | --- |
| SMSコード | △ | ◎ | SIMスワップ詐欺のリスク。無いよりは遥かにマシ |
| 認証アプリ（TOTP） | △ | ○ | 偽サイトにコードを入力させられると突破される |
| プッシュ通知承認 | ○ | ◎ | 「疲労攻撃」（連打して誤タップ狙い）に注意。番号マッチング推奨 |
| **パスキー / セキュリティキー（FIDO2）** | **◎** | ○ | **偽サイトでは動作しない＝フィッシング原理的に無効** |

## 4. パスキー / FIDO2 / WebAuthn

**パスキー（Passkey）** は、パスワードそのものを使わない「パスワードレス認証」の最新標準です（技術的には **FIDO2 / WebAuthn** という規格の上に成り立っています）。

### 仕組みをざっくり

1. 登録時、デバイス（スマホ等）の中で**鍵ペア（公開鍵＋秘密鍵）**を作る
2. サービスには**公開鍵だけ**を渡す。秘密鍵はデバイスから出ない
3. ログイン時は、サービスから来た「お題」に秘密鍵で署名して返す（指紋や顔でデバイスのロックを解除して実行）

### なぜフィッシングに「原理的に」強いのか

- パスキーは**登録したサイトのドメインと紐付いて**保存される
- 偽サイト（`examp1e.com` など）では、本物用のパスキーが**そもそも呼び出されない**
- 「うっかり偽サイトに入力してしまう」という人間のミスが、仕組み上起こり得ない

> **実務での位置づけ**：IdPのログインをパスキー化するのが現在のベストプラクティスの方向です。Entra ID・Okta・Googleはいずれもパスキー対応済み。まずは管理者アカウントから導入するのがおすすめです。

## 5. 補足：その他の認証方式

Web系のSAML/OIDC以外にも、SSOの周辺でよく出てくる仕組みがあります。

| 方式 | ひとことで | 主な用途 |
| --- | --- | --- |
| **Kerberos（ケルベロス）** | 「チケット」を発行して社内サーバー間の本人確認をする方式 | Windows/Active Directory環境。社内PCログイン後にファイルサーバーへ追加認証なしで入れるのはこれ |
| **LDAP（エルダップ）** | ユーザー情報（社員名簿的なディレクトリ）の保管・検索 | Active Directoryの中核。SSOというより「ユーザー情報の置き場・認証元」 |
| **IWA（統合Windows認証）** | WindowsのログインをそのままWebアプリ認証に流用 | 社内イントラのWebアプリ |
| **パスワードボールティング** | IdPがパスワードを代理入力するレガシー型SSO | SAML/OIDC非対応の古いアプリへの苦肉の策 |
